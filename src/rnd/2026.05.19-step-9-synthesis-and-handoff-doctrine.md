## 1. Motivation

### 1.1 The gap

The `/plan-authoring-cascaded` workflow as ratified through Run 3 (2026-05-19) ends at the **cascade-complete signal** — all sections cap-locked, post-cascade fold bundle filed, manager declares cascade closed. Done-defined stops there.

This is **load-bearing incomplete**. A cascade closure that produces only "ratified findings + cosmetic fold bundle" is not enough to hand off to an implementer cold. The cascade artifacts are spread across 4 (or N) section topic files + a pipeline summary + a parent topic file; the parent design doc is stale (still says "DRAFT — awaiting Q-decisions walkthrough"); the ratified ACs live inline in author-revision posts inside topic files; cross-section dependencies are implicit; sequencing for implementation is unstated.

An implementer picking up cold from "cascade-complete signal posted" faces a synthesis-archeology task before they can start writing code. The cascade workflow doctrine doesn't currently require anyone to do that synthesis.

### 1.2 Diagnosis: BOTH omission AND unfinished — omission is load-bearing

**Omission** (primary, load-bearing): the workflow doctrine has no explicit phase between cascade-complete and implementer-handoff. The post-cascade fold bundle exists but it is a low-stakes cosmetic-polish step (≤5 items typically; documented-not-revised stragglers). The HIGHER-stakes "translate the cascade outputs into a contract an implementer can execute cold" step is missing.

**Unfinished** (secondary, derivative): even at Run 3, the synthesis work was always implicitly required — the cascade artifacts ARE the design, just structurally fragmented. We just didn't codify "consolidate these into a single canonical contract" as a workflow phase. So in practice the Manager either ad-hocs it (Run 3: Tiberius did this solo post-cascade today, 1,225 LOC of synthesis + execution plan + design-doc amendments) or skips it (and the implementer faces archeology + likely pre-flight escalations).

The framing that wins: **omission is load-bearing**. Without codifying the phase, the work either doesn't happen (implementer struggles) or happens inconsistently (Manager ad-hoc; quality depends on Manager's bandwidth and instincts). With it codified, we get repeatable hand-off-ready output.

### 1.3 Empirical evidence — Run 3, 2026-05-19

The case for codifying Step 9 rests on two specific empirical findings from the Run 3 cascade and post-cascade work:

**Finding 1: Manager solo-authored 1,225 LOC of post-cascade synthesis work that wasn't in the workflow**:
- Phase 0 cascade synthesis doc: 476 LOC (canonical contract from 4 section files + pipeline summary + post-cascade fold bundle)
- Phase 1 parent design-doc amendments: ~50 LOC across 7 edit blocks (status flip, per-cluster ratification markers, Q-C2 + Q-D1 cascade-closure notes, AC table cross-ref)
- Phase 2 implementation-handoff execution plan: 749 LOC (DAG-first per implementer's framing preference; per-node file inventory, function signatures, step ordering, AC list, test pyramid, done-defined, standing doctrine memories)

None of this work was in the cascade workflow's Steps 1-8 done-defined. It happened because Rick explicitly asked for it, and the Manager (Tiberius) saw the gap and filled it. A different Manager, or a Manager with less bandwidth, might have produced a less complete handoff and pushed more synthesis-archeology onto the implementer.

**Finding 2: Implementer's pre-flight surfaced 2 cascade-design gaps in his first 30 minutes — both were already buried as conditional Recon items in the cascade artifacts**:
- **mic_monopoly wire field doesn't exist server-side** — Section D Q-D3 designed a mic-monopoly visual indicator (`data-mic-monopoly="true"` on pinned card → CSS pulsing overlay) assuming the data would arrive via `conversation_mode_changed` payload field `mic_monopoly`. Recon-D2 was acknowledged in the cascade as "verify the field name on the wire" but never resolved. No such field exists in the server emitter today. Result: implementer escalated to Manager → Manager escalated to user → ratified Path δ (defer the indicator entirely, file as follow-on).
- **`conversation_mode_changed` type doesn't exist on the wire** — server emits `type="speakerphone_changed"` with `payload.on: bool`; plan + smoke tests reference `type="conversation_mode_changed"` with `payload.active`. `conversation_mode_changed` is NOT in the server's `valid_types` enumeration. Recon-D1 + the cascade's "verify wire routing per 2026-04-29 cleanup pattern at code-write" pointed at this but didn't resolve it. Result: implementer escalated to Manager → Manager unilateral ratification (Path III bridge — accept both type strings client-side; forward-compat) → file server-side type-rename as future TODO.

Both gaps were technically inside the cascade artifacts (acknowledged as Recon items conditional on code-write verification). The Manager-as-synthesizer carried them forward into the synthesis doc verbatim but did NOT elevate them as "these have unresolved server-side dependencies and should be escalation-worthy at synthesis time, not implementer-pre-flight time."

**Why these findings are the empirical anchor**: synthesis-by-Manager-alone has a blind spot. The Manager is in "faithful synthesis" mode after a long cascade, not "challenge the cascade assumptions" mode. A reviewer with fresh eyes would have pivoted to that mode and flagged both gaps as escalation candidates at synthesis time, saving the implementer two pre-flight round-trips and saving the user one runtime ratification.

**Conclusion**: **synthesis is not packaging. Synthesis is a quality gate.** Step 9 must include both a synthesis step AND a light-review step. Without the light-review, the synthesizer's blind spot leaks gaps into the handoff package.

## 2. Three-Artifact Spec for `/plan-authoring-cascaded`

Step 9 produces three artifacts. Each artifact has a specific shape, required content, and acceptance criteria.

### 2.1 Synthesis doc (the "why anchor")

**Naming**: `NN-cascade-synthesis.md` (where NN is the next-available number in the design-doc directory). Companion to the parent design doc.

**Purpose**: Single canonical contract derived from all section artifacts. Implementer (and future readers) read this to understand WHAT was ratified and WHY.

**Required content**:
- §1 Purpose statement + sister-doc cross-refs
- §2 Cascade telemetry (closure metrics, per-section findings count, verbatim-accept rates, escalation counts, hard-verification-gate count, doctrine-candidate count)
- §3 Per-section ratified synthesis — **one §3.X subsection per cascade section**, each containing:
  - Cluster Q-decisions ratified (verbatim verdicts; including user escalations and manager-unilateral ratifications inline-cited)
  - Final acceptance criteria (full AC table, verbatim text per AC including EXECUTOR column)
  - Execution steps (final revised text after all cascade revisions)
  - Pre-flight recon items (final list including retired/rescoped/added)
  - Files affected inventory (NEW vs EDITED)
  - Reused functions / utilities (pre-surfaced reuse map)
  - Cross-section dependencies (E-edge identifiers + contract surface)
  - Stage findings closure trail (Stage 1/2/3 per section)
- §4 Cross-section dependency map (mermaid graph + edges table)
- §5 Post-cascade fold bundle (final shape; items deferred to documentation polish, NOT new code work)
- §6 Doctrine candidates brief index (cross-ref to PIP-side deep redline)
- §7 Sequencing recommendation for implementation
- §8 Open items + hand-off to design-doc amendment + execution plan

**Acceptance**:
- All ratified ACs appear verbatim (zero paraphrase; cite cascade artifact line-refs for non-trivial extractions)
- All user escalations + manager-unilateral ratifications inline-cited with timestamps + authority chains
- All hard-verification gates (e.g., AC-B15 in Run 3) called out explicitly with cross-section enforcement semantics
- All Recon items final-state explicit (RETIRED / RESCOPED / CARRIED / RESOLVED / NEW)
- Cross-section dependency map covers all E-edges
- Implementer can read §3.X for any section and reconstruct that section's contract without back-referencing the cascade topic files (cold-context test, §4 below)

### 2.2 Parent design-doc amendments (the "reference of record")

**Naming**: in-place edits to the existing parent design doc (e.g., `10-phase6c-persona-focus-recorder-design.md` in Run 3). NO new file.

**Purpose**: The parent design doc was created at design-mode time as the canonical reference. After the cascade ratifies the design, the design doc must reflect that ratified state — otherwise it becomes stale and contradicts the cascade outcomes.

**Required edits**:
- **Status header**: flip from "DRAFT — awaiting Q-decisions walkthrough" (or equivalent) to "🟢 CASCADE-RATIFIED YYYY-MM-DD" + sister-doc cross-refs (synthesis + execution plan paths)
- **Cascade closure metadata**: brief paragraph in the header table capturing wall-clock, sections-closed, findings totals, verbatim-accept rate, user escalations, manager-unilateral ratifications, reviewer reassignments, hard-verification gates introduced, doctrine candidates filed
- **Per-cluster ratification markers**: each Cluster N section gains a "✅ FULLY RATIFIED (X/Y) YYYY-MM-DD (Run N cascade)" line + brief cascade-closure paragraph describing notable closures (verbatim-rate, user escalations, manager-unilateral ratifications)
- **Per-Q ratification markers**: each Q that had a user escalation or manager-unilateral ratification gains a dedicated "✅ RETIRED via user escalation YYYY-MM-DDTHH:MM UTC" or similar marker with cascade-closure narrative inline
- **AC tables**: replace the original DRAFT AC tables with final ratified per-section AC tables (or cross-ref the synthesis doc §3.X for the full ratified tables; keep the legacy DRAFT table appended for historical context)
- **Files affected section**: replace rough inventory with definitive per-section inventory (or cross-ref synthesis + execution plan); preserve legacy rough inventory below for historical context

**Acceptance**:
- Anyone reading the design doc post-amendment can tell immediately that the cascade closed AND find the canonical synthesis + execution plan via the cross-refs
- No DRAFT-era statements remain unmarked (everything that was overruled has a strikethrough or supersession note)
- Manager-unilateral ratifications and user escalations are inline-cited (not buried in cascade-artifact line-refs only)

### 2.3 Execution plan (the "what + how")

**Naming**: `NN+1-execution-plan.md` (numbered after the synthesis doc).

**Purpose**: Implementer's handoff doc. Anyone picking this up cold should be able to ship section-by-section without re-deriving cascade context.

**Structure** (per implementer's framing preference — Roscoe in Run 3 asked for DAG-first; the workflow doctrine should leave the structure flexible per implementer's stated preference):
- §1 DAG / sequencing graph (mermaid; per-edge rationale; concurrency options)
- §2 Global standing rules (doctrine memories that apply across all nodes; test pyramid required at every node; code style invariants; boot ordering invariants if applicable)
- §3 Per-node deliverables (in DAG order), each containing:
  - Provides / Depends-on / Coordinates-with (cross-node)
  - Synthesis cross-ref (to §3.X of the synthesis doc — the why-anchor)
  - Files to write (NEW) — path + purpose + size budget
  - Files to edit (EDITED) — path + what changes
  - Function signatures (key public surface)
  - Step-by-step sequence (must-run-in-order)
  - Pre-flight recon (verify at code-write time)
  - Acceptance criteria cross-ref to synthesis §3.X
  - Done-defined for this node (tabular pass/fail per tier)
- §4 Cross-cutting gates (hard-verification gates, full coverage gate, regression schedule, boot handshake)
- §5 Done-defined for the overall Phase
- §6 Post-cascade fold bundle items (5+ cosmetic-polish items, mostly already folded by synthesis or amendments)
- §7 Implementer coordination surface (escalation paths, manager/synthesizer contact, parallel-track owners)

**Acceptance**:
- Implementer (or proxy / cold-reader) can read the execution plan end-to-end and identify (a) where to start, (b) what file to touch first, (c) what function to write, (d) what tests to run, (e) what gates to pass — without back-referencing the cascade topic files
- All standing doctrine memories that affect the implementer's work are explicitly listed (Lupin-wide coverage mandate, never-auto-commit-push, recraft-speech-for-TTS, etc.)
- All cross-cutting gates from the synthesis are reified into the execution plan (e.g., a hard-verification grep-gate at the boundary between two nodes)
- Done-defined explicitly names: what reports come back from the implementer, what tabular shape, what cross-phase regressions

## 3. One-Artifact Spec for `/plan-review-cascaded`

The `/plan-review-cascaded` flavor has a lighter handoff need — the input plan already exists (came in from outside the cascade) and the cascade produces a ratified revision package, not a from-scratch implementation plan. But the cold-context test still applies: the original author of the input plan (or a new implementer) shouldn't have to re-read 4 section topic files to know what to revise.

### 3.1 Revision-handoff doc (single artifact)

**Naming**: `NN-cascade-revision-handoff.md` (companion to the input plan).

**Purpose**: Single canonical handoff doc consolidating the cascade's revision package for the original author or implementer.

**Required content**:
- §1 Purpose + cross-ref to input plan
- §2 Cascade telemetry (same shape as authoring-cascade synthesis §2)
- §3 Per-section revision summary, each containing:
  - Findings closed (severity-by-severity tabular: foundational / inconsistency / cosmetic)
  - Verbatim revisions accepted (with author-revision line-refs to cascade artifacts)
  - Documented-not-revised stragglers (cap-preserved)
  - User escalations + manager-unilateral ratifications inline-cited
  - Impact on input-plan ACs (which ACs amended; which remain unchanged)
- §4 Cross-section ratification summary (which inputs survived cascade; which were retired)
- §5 Post-cascade fold bundle
- §6 Doctrine candidates brief index
- §7 Hand-off statement (input plan author should revise plan with the cascade's outcomes folded in; OR implementer should implement the input plan as cascade-revised)

**Acceptance**: same cold-context test — original author or implementer can read this doc and know exactly what to revise / implement without back-referencing the cascade topic files.

**Note**: review-cascade Step 9 is single-artifact because the parent input plan already plays the "reference of record" role (no design-doc amendments needed in the review-cascade context; the parent IS the design). Execution plan is also less applicable because the revisions are amendments to an existing plan, not a from-scratch DAG.

## 4. Authorship + State

### 4.1 Authorship — Manager-extension default

Step 9 authorship is the **Manager's** responsibility by default (the same persona who ran the cascade). Rationale:
- Manager has the freshest cross-section context (facilitated all section closures, ratified all findings, witnessed all escalations)
- Manager already maintains the "manager-classification" thread across the cascade — synthesis is a natural extension of that thread
- Avoids cast expansion (no new Persona 6 to allocate, no inter-persona handoff overhead)

**Escape hatch for future v3**: workflow doctrine may add a "designated synthesizer" role if cascades grow large enough that synthesis becomes its own full-time job. Phase 6c (4 sections, 43 findings, 1,225 LOC of synthesis output) is at the edge of Manager-solo feasibility. A future cascade with N=8 sections might warrant the split. For v1, default = Manager; escape hatch documented but not invoked.

### 4.2 State — `implementation_handoff_ready` (new closure_action enum)

Step 9 ends with the cascade declared **`implementation_handoff_ready`**. This is a new closure_action enum value in `plan-authoring-cascaded-defaults.md`.

**State semantics**: the cascade has cleared synthesis + light-review + cold-context test. The 3 artifacts (or 1 in review-cascade flavor) are landed at their canonical paths. An implementer can pick up the artifacts cold and start.

**Distinct from `cascade_complete`**: `cascade_complete` is the existing closure state after all sections cap-lock + cascade-complete signal fires. `implementation_handoff_ready` is the NEW state that gates "handoff to implementer." A cascade can be `cascade_complete` without being `implementation_handoff_ready` (intermediate state during Step 9 work).

### 4.3 Cold-context test acceptance criterion (option (a), self-administered)

The Manager-as-synthesizer reads the 3 artifacts (or 1 for review-cascade) end-to-end as if cold. Passes if:
1. They can describe the implementation path section-by-section without back-referencing the cascade topic files
2. All Recon items have a clear resolution path (RETIRED / RESCOPED / CARRIED with caveat / RESOLVED / NEW with conditional-executability)
3. All cross-section dependencies are explicit (the implementer doesn't need to infer what section X needs from section Y)
4. Standing doctrine memories that affect the implementer's work are explicitly listed
5. No "TBD at code-write" or "verify on the wire" items remain WITHOUT either (a) being clearly tagged as conditional-executability with documented branches, or (b) being escalated to user / manager-unilateral resolution before Step 9 closes

**Self-administration cost**: ~15 min for the Manager to read the 3 artifacts end-to-end + answer the 5 questions.

**Option (b) external-administered**: deferred to v2 future. A peer session (one of the 4 cascade reviewers, or a future Persona 6) reads the artifacts cold. Higher confidence but higher cost. Reserve for cases where v1 self-administration proves to leak (e.g., Run 3's mic_monopoly + type-rename gaps — would these have surfaced under self-administration alone? Empirically: probably not, because the Manager was the one who let them through in the first place. v2 may need to add this gate).

## 5. Light-Review Gate (NEW per Tiberius+María convergence, 2026-05-19)

### 5.1 Rationale

Run 3's empirical evidence (Roscoe's pre-flight finding 2 cascade-design gaps in his first 30 minutes — gaps that were technically inside the cascade artifacts as conditional Recon items but were not surfaced by the Manager-as-synthesizer) demonstrates that **synthesis-by-Manager-alone has a blind spot**.

The blind spot mechanism: the Manager is in "faithful synthesis" mode after a long cascade. Their cognitive frame is "compile what was ratified" not "challenge the cascade assumptions." A fresh-eyes reviewer would pivot to that challenge-mode and flag conditional Recon items as escalation candidates BEFORE handoff, instead of letting them surface at implementer pre-flight.

Without a light-review gate, the Manager's blind spot leaks gaps into the handoff package. The cost is borne by the implementer (pre-flight escalation round-trips) and the user (mid-implementation ratification asks).

With a light-review gate, the Manager's synthesis is read by ONE additional cascade-participant who is NOT the Manager, with a focused rubric on "are there hidden assumptions in the synthesis that an implementer would surface at pre-flight?"

### 5.2 Reviewer selection

**Default**: one of the 4 cascade reviewers (Persona 3 / 4 / 5 — Usability/Reuse / Viability/Gap / Ownership/Conv 6). Preferably:
- The reviewer with **freshest context on the most-impacted section** (i.e., the section with the most findings, the longest revision trail, or the most cross-section dependencies)
- For Run 3, that would have been Arnold 🪨 (Section B Stage 2 review caught the AC-B15 hard-verification gate; Section D Stage 2 review caught the dual-emission contract refinement — both high-context closures)

**Alternative**: María-as-doctrine-consultant (the role she played in Run 3 as the 6th cascade participant). María's vantage is doctrine-rather-than-section-implementation, but doctrine-aware-fresh-eyes is also a useful check on synthesis quality.

**Escape hatch**: if no cascade participant has bandwidth, the Manager can declare self-administered cold-context-test sufficient AND file a TODO for v2 to revisit. Cascade-quality risk is real but bounded.

### 5.3 Light-review rubric (NOT a full Persona 3/4/5 review)

Reviewer reads the 3 artifacts (synthesis + design-doc amendments + execution plan) end-to-end with this specific focus:

**Pass criteria** (all must hold):
1. **External-system assumptions check**: any Recon item still unresolved that touches an external system (server wire, infra, third-party API) is either (a) resolved before Step 9 closes, or (b) explicitly escalation-worthy AT Step 9 (synthesizer surfaces it as "this needs Rick's ratification before implementer pre-flight"), or (c) explicitly tagged as conditional-executability with documented branches
2. **AC conditionality check**: any AC that's conditionally-executable on something the synthesis didn't pin down — the synthesis explicitly lists the condition + the resolution path
3. **Cross-section contract check**: any cross-section dependency mentioned in the synthesis matches the dependency mentioned in the execution plan (no drift between docs)
4. **Hard-verification gate check**: any hard-verification gate introduced in the cascade (e.g., AC-B15 in Run 3) is reified into the execution plan's cross-cutting gates section
5. **Standing doctrine memory check**: standing doctrine memories that affect implementer work (coverage mandate, commit discipline, test venue routing) are explicitly listed in the execution plan §2

**Output**: thumbs-up OR list of specific synthesis gaps to address before handoff.

**Cost**: ~10-15 min reviewer-time. Output as commons-post entry on the cascade's parent topic (or a sister `cascade-step-9-review` topic).

### 5.4 Manager response to light-review findings

If reviewer thumbs-up: cascade flips to `implementation_handoff_ready`.

If reviewer finds gaps:
- Manager addresses gaps in synthesis-doc / execution-plan / design-doc amendments
- Capped at **1 revision turn** (no Round-2 cap-extending — this is a single-pass refinement, NOT a full cascade re-open)
- Manager re-runs self-administered cold-context test post-fix
- Reviewer thumbs-up on the revision → cascade flips to `implementation_handoff_ready`
- If reviewer finds MORE gaps after Manager revision: escalate to user (rare case; suggests synthesis quality problem worth user-attention)
