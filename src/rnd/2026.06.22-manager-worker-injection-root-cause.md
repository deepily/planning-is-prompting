# Root Cause: Manager→Worker (DM-to-parked-pane) Injection Failure

**Date**: 2026-06-22 · **Author**: María 🌸 · **Bug**: `baf5ea6d` (P1, crew-wide blocker) · **Status**: ROOT-CAUSE (grounded, read-only) — fix-spec for whoever builds (Mr Radio / fresh worker), review Tiberius

**Symptom (Tiberius, firsthand)**: both his workers (Rachel, Clayton) went dark "parked with stale unsent buffered text"; his `dm_send` returned `dispatched:true` at the API but the body never reached their tmux input. Worker→manager DMs still work. Independently corroborated this tick: Mr Radio's own session parked at `idle_prompt` since 19:25 UTC and unreachable by my DMs.

---

## Root cause (grounded in code)

The inbound-DM delivery is **two-stage and hook-TRIGGERED, not push-delivered**:

1. **Buffer stage** — `cc_notification_listener.py` (per-session WebSocket subprocess spawned at SessionStart) receives `user_initiated_message`/`ai_to_ai` notifications and **only appends them to** `~/.claude/sessions/cc-buffer-<sid8>.jsonl`. It does **not** inject to tmux.
2. **Delivery stage** — `hook_common.py::deliver_pending_peer_dms(session_id)` (lines 799-851) drains that buffer and calls `inject_qualifier_via_tmux(...)` per DM. But it runs in only **two places** (per its own docstring): the **Stop hook speakerphone branch** and the **Notification hook at `idle_prompt`**.

**The gap**: delivery fires only when one of those hook events occurs. A session that has finished its turn and is **parked quiescent at an idle prompt fires no further hook events** — so a DM that arrives *after* the pane went idle has **no triggering event** and strands in the buffer indefinitely. The `idle_prompt` drain runs at the *transition into* idle, not on each new buffer arrival while already idle.

This exactly explains: `dispatched:true` (listener buffered it) + "unsent buffered text" (never drained) + worker→manager works (the manager is actively cycling, so its hooks fire) + manager→parked-worker fails (the worker is quiescent).

**Scope correction**: this is NOT manager-specific — it's *delivery to any quiescent/parked session*. Active, cycling sessions (Tiberius, María) receive fine because their hooks keep firing the drain.

---

## Fix directions (for the build spec)

1. **PREFERRED — listener push-injects on receipt.** `cc_notification_listener` already runs as an event-driven subprocess holding the `session_id`; on buffering an `ai_to_ai` DM it should **directly call `inject_qualifier_via_tmux`** (or a thin idle-aware variant) rather than wait for a future hook. Removes the dependence on a triggering hook event entirely.
2. **Fallback — idle re-drain poll.** A lightweight periodic drain of `cc-buffer-<sid8>.jsonl` for idle panes (timer or the listener's own loop) so buffered DMs to a parked pane are delivered within N seconds.
3. **Guard** — whichever path: keep the existing atomic `drain_voice_buffer` (no double-inject) and the `wrap=False` framing; verify `inject_qualifier_via_tmux` tmux `send-keys` actually targets the worker's pane (confirm the tmux target resolution for a spawned worker session).

**Acceptance**: a DM sent to a session that is already parked at an idle prompt lands in its tmux within seconds (no further user/agent activity required); verified across a spawned worker (not just a manager).

---

## Why this gates everything (chicken-and-egg)

The crew build-review-fix loop depends on managers reaching workers. With delivery-to-parked-panes broken, any worker that finishes a sub-task and parks becomes unreachable for redirection → dark. The fixes for the false-idle family (`de653086`, `09f64040`) need a crew; the crew needs this delivery path. **Bootstrap path**: a FRESH spawn receives its brief at launch (not via tmux-inject), so a fresh worker with a COMPLETE self-contained brief can build *this* fix without needing mid-flight redirection — that's the way to break the cycle (pending Rick's cross-project-spawn authorization, his gate).

## Fix design — adversarial refinements (Tiberius, 2026-06-22, validated)

Tiberius read the spec, independently confirmed the root cause from the code (exactly two `deliver_pending_peer_dms` callers — `notification.py:62`, `stop.py:1630` — and an append-only listener), and raised three points that **change the fix shape**. All accepted:

1. **CRUX — inject only when the pane is PARKED (not naive push-on-receipt).** The hook-triggered design is safe precisely because it injects only at known-prompt-safe points. A naive "inject on receipt" would `send-keys` into a pane that is **actively running a turn**, corrupting/interrupting it. So the correct fix is: **listener injects on receipt IFF the pane is parked at an idle prompt; otherwise leave it buffered for the next hook drain.** The hard, load-bearing sub-problem is **how the listener subprocess detects pane-idle-state** — this is the design crux, NOT the earlier "thin idle-aware variant" footnote.
   - **Proposed idle-detection (existing signal, no new tmux scraping):** the session already emits `kind="idle_prompt"` heartbeat-events when it parks, and busy transitions (Stop `honored`/`poke`, UserPromptSubmit) when it works. The listener tails the session's `heartbeat-events/<sid>.jsonl` (or a dedicated `state` marker the hooks write) and treats the session as parked iff its most-recent event is `idle_prompt`. Inject only then; else buffer. This reuses the liveness event stream rather than fragile capture-pane parsing.
2. **Inter-PROCESS drain safety (flock, not in-process atomicity).** The listener subprocess AND the Stop/Notification hook process can drain `cc-buffer-<sid8>.jsonl` concurrently. The no-double-inject guard MUST be an inter-process lock (`flock` on the buffer file), not just `drain_voice_buffer`'s in-process atomicity — otherwise the two processes race and a DM double-injects or is lost.
3. **Acceptance test — add the NEGATIVE case.** Positive (Tiberius's): a DM to a *parked* pane injects within seconds with NO Stop/Notification event. **Negative (added):** a DM arriving while the pane is **mid-turn** does NOT corrupt/interrupt the running turn and lands cleanly *after* it parks. Both together prove idle-awareness — the positive alone would pass even a naive (unsafe) push-inject.

**Net**: the build brief must lead with the idle-state-detection mechanism (#1), specify the `flock` guard (#2), and gate on BOTH acceptance cases (#3). Tiberius will reproduce-not-trust against these the moment a builder lands it.

### Idle-detection — closing the going-idle race (Tiberius round 2, accepted)

The single-trigger "inject on DM receipt IFF latest event == idle_prompt" has a residual race: a DM arriving **during going-idle** (pane parked but the `idle_prompt` event not yet written to the jsonl) reads "busy" → buffers → and if injection only fires ON-DM-RECEIPT, nothing re-evaluates when the event finally lands → strands (same bug class). Close it with **two listener triggers + a TOCTOU guard + a measure-first material choice**:

- **Trigger #1 (already-parked):** a new DM arrives AND latest event == `idle_prompt` → inject.
- **Trigger #2 (going-idle-gap):** the listener OBSERVES a new `idle_prompt` event in its own event-tail AND the buffer is non-empty → drain/inject. This is the listener replicating the hook-drain off its own event-tail; **without #2 the race bites.**
- **TOCTOU guard:** immediately before `send-keys`, RE-READ the latest event; if it's no longer `idle_prompt` (a turn just started), ABORT + re-buffer. Shrinks the inject-into-a-busy-pane window to the read→send-keys gap. (The in-hook drain doesn't need this — it runs synchronously at the transition; the async listener can't get that for free.)
- **MEASURE-FIRST material choice (builder's step 0):** measure the latency between pane-parks and the `idle_prompt` event landing in the jsonl.
  - **bounded-small** → event-tail (triggers #1/#2) + the re-read guard is sufficient.
  - **long/variable** → use an explicit **hook-written idle-epoch marker** instead: the hook sets an epoch on park, bumps it on UserPromptSubmit; the listener injects then re-checks the epoch unchanged (atomic-ish CAS). **Do NOT pick the marker blind — measure, then choose.**

**Updated build sequence**: (step 0) measure the park→event latency → pick event-tail vs epoch-marker; (1) implement the chosen idle-detector with triggers #1 + #2; (2) `flock` inter-process guard on `cc-buffer-<sid8>.jsonl`; (3) TOCTOU re-read before send-keys; (4) pass BOTH acceptance cases (parked→injects-in-seconds / mid-turn→no-corruption-lands-after-park).
