# Always-On Manager-Figure: The Manager-Tick Loop (design)

**Status**: v0.2 (2026-06-15, María — Workflow Steward / PIP manager-figure) — all 5 §6 decisions RULED by Rick via guided walkthrough. Authored on Rick's directive ("design the workflow first — usable tomorrow when everyone has the internal task list"). The §6 rulings are folded (incl. the D2 poke/chase-only reframe into §3.3 + §5); **ready to graduate to a `workflow/manager-autonomy.md` section** (next build step).

**Purpose**: make a manager-figure *always-on and pushing forward* — structurally, not by disposition. A recurring self-wake ("tick") drives the manager to review a **global task list** and act on every owed item, so work advances without the user nudging.

---

## 1. The problem — drive can't depend on disposition

Observed (Rick, 2026-06-15): Opus-4.8 manager-figures (María, Mr Radio, Tiberius) run **laid-back** — they finish a task and *wait*, where the Fable-era managers *drove tasks to completion*. Founding case this evening: I folded a doc, said "I'll surface it when Krishna pings," and idled ~40 min on a peer who was alive and free (→ memory `feedback_proactively_drive_idle_peers`).

The fix is **not** "try harder to be proactive" (vibes don't persist across `/clear`, and disposition varies by model). The fix is **structural**: a scheduled, agenda-driven loop that re-asks the manager to do its job on a timer, with a concrete agenda so the prompt isn't a dumb nag. Drive becomes a mechanism, not a mood.

## 2. The mechanism — `/loop` (+ `ScheduleWakeup`)

**`/loop`** is the Claude Code skill for running a prompt or slash-command on a recurring cadence:

- `/loop 7m <prompt-or-command>` — re-injects `<prompt>` every 7 minutes. The harness wakes the session, runs the prompt, and re-arms.
- `/loop <prompt>` with **no interval** — the model **self-paces**: each turn it decides when to wake next via **`ScheduleWakeup(delaySeconds, prompt, reason)`**. Self-pacing is what enables *adaptive* cadence (tighten when active, back off when quiet) instead of a blind fixed timer.
- Omitting `ScheduleWakeup` on a turn ends the loop. So the loop is self-terminating by construction — the manager stops ticking when there's nothing left to drive.

**How it becomes "remind the manager to do its job":** the looped `<prompt>` *is* the reminder. Each tick re-injects a **manager-tick prompt** that says, in effect: *"You are the manager. Here is your standing job — pull your owed work from the task store and drive it."* The recurrence is the always-on; the prompt content is the job description; the task store is the agenda.

> Native-tool note: `/loop` is the **local, per-session** self-wake. The `:8001` arbiter is the **external, fleet-level** complement (it pokes a manager when the fleet shows owed work, push-not-poll). They compose: the local tick drives *my* agenda; the arbiter is a backstop that pokes me if I've gone dark. Start with `/loop` (self-contained, no deploy gate); layer the arbiter later.

## 3. The design — trigger · agenda · tick-procedure · cadence · scope

### 3.1 Trigger
A self-paced `/loop` (no fixed interval) so cadence is **adaptive**. Default active cadence ~7–8 min; back off toward ~20–30 min when quiet (cache-window aware — see `ScheduleWakeup` guidance: stay <5 min only when actively polling external state, else 1200s+).

### 3.2 Agenda — the global task list (the linchpin)
Each tick begins with one query against the unified task store (Phase-2.1 live on `:7999`/`:8000`, verified 2026-06-15):

```
task_query( accountable_manager = me, status ∈ {queued, claimed, in_progress, blocked} )
```

This is what turns "go manage" (vague nag) into "**I've got Krishna on X — is it past its chase-time? should I check in?**" (concrete checklist). Without the store the loop is a nag; with it, the loop has a real, queryable agenda — and `next_chase_ts` per item tells the manager *exactly* when each owed item is due for a poke.

### 3.3 Tick procedure (the manager's "job" each fire)
For each owed item from §3.2:
1. **Overdue?** Is `now > next_chase_ts`? (un-due items are left alone — no premature nagging.)
2. **Liveness?** Is the owner alive + producing? (`commons_who` + recent receipts / commits.)
3. **Decide & act** (standing authority — act-then-announce; **poke/chase only per D2**):
   - progressing on time → leave it (bump nothing).
   - overdue + owner idle/alive → **poke** (`dm_send`) with a specific status ask + next work.
   - blocked on *me* → action it now.
   - blocked on *Rick* → surface async (don't block him; `notify`).
   - owner dark/unproductive → **flag for a deliberate reap/replace decision OUTSIDE the tick** (D2 ruling: the tick never auto-reaps; MANAGE-not-BUILD still applies when the manager *does* act on the flag).
   - done + receipted → verify the receipt, close the item.
4. **Cheap exit** — if nothing was actionable this tick, exit fast (delta-aware: don't re-narrate an unchanged board).
5. **Persist** — write transitions / bump `next_chase_ts` back to the store so the loop's own state survives `/clear` and is visible to the arbiter + Rick.
6. **Re-arm or stop** — `ScheduleWakeup` for the next tick at the adaptive delay; **omit it** (end the loop) when K consecutive ticks find zero owed work.

### 3.4 Cadence — adaptive, not blind
Reuse the cascade-heartbeat **KEEP-WITH-TUNINGS** rulings (2026-06-12): **delta-aware ticks** (wake-and-act only on change/threshold), **honored adaptive cadence** (M=8/4/2 style backoff), **pane/state differential before poking** (don't poke an unsaveable session). A fixed 7-min nag that fires forever is the anti-pattern these tunings exist to prevent.

### 3.5 Scope — managers-first
Runs for **manager-figure sessions** (the `manager-autonomy.md` §2.1 predicate: the repo's env-keyed standing persona). Mirrors the work-owed oracle's managers-first scope and the F4 task-store write scope. Widen to all sessions later (the double-anchored widening rider).

### 3.6 Hygiene
Always leave a memento on stop; never poke past the per-session cap; the tick is **read-the-board + act**, never make-work. Non-compliance (a manager not ticking) is a practice bug, flagged once — fail-open, not a liveness alarm.

## 4. Relationship to the existing keep-alive mechanisms

| Mechanism | Kind | Job | Relationship |
|---|---|---|---|
| **Stop-hook heartbeat** | local, event-driven (on Stop) | "don't fall asleep at the wheel" — re-prompt a session that's about to idle | The *floor* — keeps a session awake. The manager-tick is the *content* of what an awake manager then does. |
| **Manager-tick `/loop`** (this doc) | local, timed/self-paced | review my task-store agenda + drive every owed item | The manager's own always-on drive. |
| **`:8001` arbiter operator loop** | external, fleet-level | poke a manager when the fleet shows owed work; escalate genuine stalls | The backstop — pokes me if I've gone dark; reads the *same* store. |

All three read/write the **same task store** — no parallel truth. The manager-tick is the missing middle layer: the Stop-hook keeps you awake, the arbiter watches you from outside, but neither makes *you* proactively work *your* list. This loop does.

## 5. The manager-tick prompt (reusable template — the "job description")

> *Draft — refine after §6 rulings.* The text `/loop` re-injects each tick:

```
You are the manager-figure for <repo>. This is your standing manager-tick — drive your owed work, do not wait to be asked.
1. task_query(accountable_manager=me, status in {queued,claimed,in_progress,blocked}).
2. For each item: overdue vs next_chase_ts? owner alive + producing (commons_who + receipts)?
3. Act (standing authority, act-then-announce — POKE/CHASE ONLY per D2): poke an idle/overdue owner with a specific ask + next work; action items blocked on you; surface items blocked on Rick (async notify); FLAG a dark/unproductive owner for a deliberate reap/replace decision OUTSIDE the tick (do NOT auto-reap); verify+close done+receipted items.
4. Persist transitions / bump next_chase_ts to the store.
5. If nothing was actionable, exit cheap (delta-aware — don't re-narrate an unchanged board).
6. ScheduleWakeup for the next tick at an adaptive delay (tighter when active, 1200s+ when quiet); OMIT it (end the loop) after K consecutive empty ticks. Leave a memento on stop.
```

## 6. Decisions — RULED 2026-06-15 (Rick, guided walkthrough)

All five ruled live. **Key reframe (D2): this loop serves the MANAGER-figure (María / Mr Radio / Tiberius) — to stay on top of their own work + the work they've assigned to workers — NOT Rick. He is not in this loop.**

1. **Cadence** → **adaptive self-paced** (`ScheduleWakeup`; tight when driving, back off when quiet). Fixed timers nag + burn cache; inherits the cascade-heartbeat tunings.
2. **Self-management depth** → **poke / chase ONLY**. The tick advances the manager's own items + pokes/chases workers; **reassign / reap+replace is a deliberate manager action OUTSIDE the tick**, never auto-fired. (Committing/pushing to the repo remains the standing Rick-gate — unchanged background rule, not part of the tick.)
3. **Primary driver** → **`/loop` now**; the `:8001` arbiter composes in later as an external backstop (no deploy gate; usable tomorrow).
4. **Heartbeat overlap** → **keep separate layers** — the Stop-hook heartbeat is the keep-awake floor; the manager-tick is the drive-your-agenda loop. Distinct jobs; the tick ships without touching `stop.py`.
5. **Empty-agenda behavior** → **low-frequency keep-alive while the store fills** (back off ~20–30 min on an empty board, tighten as items land). Memento-and-stop is the genuine-wrap end-state, not the day-one empty-then-filling case.
