## 1. Source-of-Truth Snapshot (the new MCP contract)

From `<lupin>/src/lupin_mcp/cosa_voice_mcp.py:888-934` (the actual MCP tool signature Claude Code sees on session start):

```python
def ask_yes_no(
    question: str,
    default: str = "no",          # "yes" or "no" only — Neither is NEVER a default
    timeout_seconds: int = 60,
    priority: str = "medium",
    abstract: Optional[ str ] = None,
    job_id: Optional[ str ] = None
) -> str:
    """
    Convenience wrapper for quick binary decisions, with a third "Neither"
    escape hatch for cases where the question itself needs re-framing.
    ...
    Ensures:
        - returns one of:
          * "yes", "no", "neither"
          * "yes [comment: ...]", "no [comment: ...]", "neither [comment: ...]"
        - returns the default value (as string) on timeout or error
        - On "neither", Claude should treat the response as a signal that the
          question needs re-framing rather than as a soft yes or no — read the
          comment (if present) and ask a clearer follow-up question
    """
```

**Three semantic shifts** the docs must absorb:

| # | Old (binary) | New (ternary) |
|---|--------------|---------------|
| 1 | 4 response variants: `yes` / `no` / `yes [comment:...]` / `no [comment:...]` | **6 variants** — add `neither` / `neither [comment:...]` |
| 2 | `default` could in theory have been `"yes" \| "no"` (already true) | **Default is explicitly NOT `neither`**; Neither requires an explicit user click |
| 3 | Handler pattern: `if response.startswith("yes"): ... elif response.startswith("no"): ...` | **Ternary handler**: third arm must re-frame the question rather than treat Neither as soft-yes/soft-no |

The third shift is the load-bearing one. The first two are mechanical table edits; the third changes how Claude *reasons* about the answer. The plan emphasizes it accordingly.

## 3. The "Handling Neither" Hub Pattern (the single source-of-truth)

To keep PIP DRY and avoid duplicating the ternary parser across 15 files, the plan establishes a **single canonical pattern block** in `workflow/cosa-voice-integration.md` that every spoke cross-references. Drafted text below — this is what gets inserted as a new sub-subsection between `#### Qualified Comments` and `### converse()`:

```markdown
#### Handling Neither — the Re-Frame Escape Hatch

Neither is NOT a soft no. When the response starts with `neither`, the user is
signaling that the question itself needs re-framing — they couldn't honestly
answer yes OR no without misrepresenting their intent. Treating Neither as a
silent skip or a deferred-yes is the failure mode this escape hatch exists to
prevent.

**The ternary parser pattern**:

```python
response = ask_yes_no(
    question="Commit these 5 files to the repository?",
    default="no",
    timeout_seconds=300,
    priority="high"  # MANDATORY for blocking tools
)

if response.startswith( "yes" ):
    proceed_with_commit()
elif response.startswith( "no" ):
    skip_commit()
elif response.startswith( "neither" ):
    # Extract the qualifying comment (if present) and re-ask with a
    # narrower question that addresses the user's concern.
    comment = ""
    if "[comment:" in response:
        comment = response.split( "[comment: " )[1].rstrip( "]" )
    # Re-frame using the comment as the signal for what was ambiguous.
    # Do NOT default to "no" and continue silently.
    notify( f"Re-framing — your concern: {comment}", priority="high" )
    # Then issue a more specific ask_yes_no() or ask_multiple_choice().
```

**Anti-patterns** (PROHIBITED):

| Anti-pattern | Why it's wrong |
|--------------|----------------|
| Treat `neither` as `no` and silently skip the gate | The user explicitly said the question was ambiguous; skipping resolves nothing |
| Treat `neither` as `yes` to keep momentum | Same failure mode in the other direction; potentially destructive |
| Ignore the `[comment: ...]` qualifier on Neither | The comment is the user's signal for what to ask instead — read it |
| Re-ask the *same* question after Neither | The question already failed once; ask something narrower |

**When to expect Neither**:
- The gate question conflates two decisions (e.g., "commit AND push?" — user wants to commit but not push)
- The gate question's scope is unclear (e.g., "delete the old backups?" — which old ones?)
- The user has new information that makes the question stale (something changed since the gate was framed)

**Critical contexts (CRITICAL)**:
For destructive operations (push, merge, tag, branch cleanup, file deletion,
config rewrite), Neither MUST NOT proceed. Re-frame and re-prompt. The
`default` parameter is meaningless here because Neither requires an explicit
user click — it cannot arrive via timeout.
```

Every Tier-3 spoke then says, near its `ask_yes_no()` callsite:

> See `workflow/cosa-voice-integration.md` → "Handling Neither — the Re-Frame Escape Hatch" for the ternary parser and CRITICAL destructive-op contract.

This is the **same hub-and-spoke pattern** that Session 81 used for the TTS Brevity Mandate (canonical spec in cosa-voice-integration.md, lightweight cross-references across 25+ spokes).
