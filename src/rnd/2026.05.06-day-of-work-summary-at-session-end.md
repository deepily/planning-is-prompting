## Context

The user wants the **very last thing they see** at session-end to be a count/breakdown of the day's lines-of-code added vs. removed, broken down by code type (code / comment / docstring per language). Optionally, a comparison against what the repo is *made of* (size baseline) so the day's work has scale context.

### Why now

- The current `workflow/session-end.md` ends on a near-empty "Final Verification" — there's no closing artifact that summarizes the work done, and the session simply trails off after the commit/push/backup steps.
- The **lupin codebase-analysis tools already do exactly the math the user wants**: `BranchChangeAnalyzer` produces git-diff LoC deltas with code/comment/docstring separation per language; `DirectoryAnalyzer` does the same for a static directory tree. Both emit JSON, markdown, or console output. There's no need to build a new analyzer.
- The skill is **global** (`~/.claude/skills/codebase-analysis/SKILL.md`), so any project Claude operates in can call it — the analyzer scripts themselves live in lupin's `src/cosa/repo/`, requiring `cosa` on PYTHONPATH.
- PIP is a **meta-repo**: a new step in `workflow/session-end.md` propagates to every downstream project that installs the session-end workflow via `/plan-install-wizard`. This is the right altitude for a feature the user wants in *every* session-end.

### Intended outcome

Adding **Step 6: Day's Work Summary** between Step 5 (Backup Prompt) and the Final Verification line. After this step:
- A markdown table is rendered to the terminal with the day's LoC delta broken down by language and by code/comment/docstring.
- A `notify()` fires with a one-line spoken/displayed headline + the rich table in `abstract` — so it's the last notification the user sees, hears (in conversation mode), or finds in the notification card.
- An optional repo-baseline section compares the day's work to the repo's overall composition.
- Graceful degradation: if `LUPIN_ROOT` isn't set or cosa isn't importable, fall back to native `git diff --shortstat` (line counts only, no code/comment/docstring split) with a friendly note explaining the upgrade path.

## Recommended Approach

### Step 6 invocation contract

**Default scope**: branch-since-`main` (matches `BranchChangeAnalyzer`'s default). This is the cleanest answer for "what did I do on this branch" and naturally captures multi-day, multi-session work without arbitrary calendar boundaries.

**Default repo-baseline**: ON (run DirectoryAnalyzer on the project root for context). User can disable per-invocation via slash-command flag if it adds too much wall-clock at session-end.

**Default channel**: terminal markdown table + `notify()` with `priority="medium"`, `notification_type="task"`, headline in `message`, full table in `abstract`. This makes the summary the last visible AND audible artifact of the session.

### Fallback chain

```
1. Check $LUPIN_ROOT exists and points to a valid lupin checkout
   (specifically: $LUPIN_ROOT/src/cosa/repo/run_branch_analyzer.py exists)
2. If yes → run via:
     cd $LUPIN_ROOT/src && python -m cosa.repo.run_branch_analyzer \
       --repo-path <pip-or-target-project-root> \
       --output json
3. Parse JSON, render markdown table, fire notify().
4. If cosa unavailable → fall back to native git diff:
     git diff --shortstat $(git merge-base HEAD main)..HEAD
   Render a degraded summary (file count + line totals only),
   with a one-line note: "Upgrade: set LUPIN_ROOT to enable
   code/comment/docstring breakdown."
5. If even that fails (no main branch, etc.) → emit a "no diff
   data available" line and continue to Final Verification.
```

This pattern matches the existing PIP precedent for env-var-gated optional tools (`PLANNING_IS_PROMPTING_ROOT` in `workflow/backup-version-check.md`, `workflow/installation-wizard.md`).

### Rendering convention

Match the existing `branch-pr-and-merge.md` table style (the Explore agent confirmed this precedent — `workflow/branch-pr-and-merge.md` line ~272):

```markdown
## Day's Work Summary

**Branch**: wip-v0.1.3-...  →  main
**Files changed**: 7  |  **Net**: +189 lines

| Language     | Added | Removed | Net  | Code | Comment | Docstring |
|--------------|-------|---------|------|------|---------|-----------|
| Python       | 234   | 45      | +189 | 84.6%| 15.4%   | 0.0%      |
| Markdown     | 18    | 0       | +18  | 100% | —       | —         |

### Repo Baseline (current composition)

| Language     | Total LoC | Code | Comment | Docstring |
|--------------|-----------|------|---------|-----------|
| Python       | 12,403    | 78%  | 12%     | 10%       |
| Markdown     | 4,221     | 100% | —       | —         |

**Today's contribution**: +189 LoC against a 16,624-LoC base (≈1.1%)
```

### Spoken headline (TTS Brevity Mandate-compliant)

In conversation mode: 1 sentence, conversational, verdicts not inventory:

> "Day's wrap: plus one-eighty-nine lines net across seven files, mostly Python."

The full table goes to `abstract`. No file lists, percentages, or hash literals in the spoken line — the brevity mandate (`workflow/cosa-voice-integration.md` §Conversation Mode) forbids them.

In non-conversation mode: same `notify()` call still fires; user sees the rich abstract in the notification card.

### Flag-set on the slash command

`/plan-session-end` gets two new optional flags (default: both ON):

| Flag | Default | Effect |
|------|---------|--------|
| `--summary` / `--no-summary` | `--summary` (ON) | Toggle the whole Step 6 |
| `--baseline` / `--no-baseline` | `--baseline` (ON) | Toggle the DirectoryAnalyzer comparison section |

Power users who want to skip this on a fast wrap-up can `/plan-session-end --no-summary`. The default-on behavior matches the user's stated intent ("the very last thing the user sees").

## Failure modes considered

| Failure | Handling |
|---------|----------|
| `LUPIN_ROOT` unset | Fall back to `git diff --shortstat <merge-base>..HEAD` with upgrade-path note. |
| `LUPIN_ROOT` set but `src/cosa/repo/run_branch_analyzer.py` missing | Same fallback as unset; treat env var as advisory. |
| Branch has no `main` to diff against (orphan branch, fresh repo) | Skip Step 6; emit "no diff data available — first commit on this branch?" line. |
| No commits on the branch yet | Skip Step 6; emit "nothing to summarize — no commits since branching from main." |
| `python -m cosa.repo.run_branch_analyzer` exits non-zero | Catch, fall back to native git diff, log the cosa error to terminal (not abstract — user can see if they want to debug). |
| JSON parse error | Same fallback. |
| Repo is huge (DirectoryAnalyzer takes >10s) | DirectoryAnalyzer is the optional opt-out via `--no-baseline`. Branch Analyzer alone is fast (it only walks the diff). |
| PIP itself is mostly markdown — `language_details` may be sparse | Render the table with whatever languages are present. Markdown shows `Code: 100%, Comment: —, Docstring: —` and that's fine. |
| Conversation mode — long table read aloud is TTS-hostile | Already addressed: `message` carries 1-sentence headline only; full table in `abstract`. Per the TTS Brevity Mandate. |

## Design forks (flagged for user redirection at plan review)

These are the three places where my default may not match the user's intent. I'm picking defaults below; user can redirect at ExitPlanMode time.

| Fork | My default | Alternatives |
|------|------------|--------------|
| **Day-of-work scope** | branch-since-main (BranchChangeAnalyzer default) | calendar-day (`since="midnight"`) — fragile across midnight; or session-only (parse `.claude-session.md` for this session's commits) — granular but misses multi-session days. |
| **Repo-baseline default** | ON (`--baseline` on by default) | OFF — keeps session-end fast; user opts in via `--baseline` when they want the comparison. |
| **Slash-command flag default** | both `--summary` and `--baseline` ON | Either could default OFF. |

If the user says "session-only scope" or "baseline opt-in", the change is two-line edits to the slash-command flag defaults — trivial to redirect.

## Out of scope (deferred to future work)

- **Per-day rollup across multiple branches** — current design is per-branch only.
- **Persisting the day's stats to a `.session-stats.md` log** — interesting future feature for "weekly velocity" graphs, but YAGNI for the immediate ask.
- **Auto-installing the codebase-analysis skill into projects that don't have it** — the skill is already global at `~/.claude/skills/codebase-analysis/SKILL.md`, so this is a non-issue.
- **Adding the analyzer scripts to PIP itself** — would create duplication with lupin. Better: keep them in lupin, treat lupin as the canonical home, gate via `LUPIN_ROOT`.
- **Non-cosa-aware fallback richer than `git diff --shortstat`** — could synthesize per-language counts via `git diff --numstat` + extension classification, but adds complexity for a path that's the degraded-fallback case. Stick with shortstat.
- **History.md auto-injection of the day's summary** — could append the table into the session-end's history.md entry. Useful but adds coupling between Step 1 (history update) and Step 6 (summary). Defer.
