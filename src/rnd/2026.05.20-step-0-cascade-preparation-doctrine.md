## 1. Motivation

### 1.1 The gap

The `/plan-authoring-cascaded` workflow as currently codified has THREE phases:

1. **Steps 1-8** (execution): cascade runs — authors draft, reviewers iterate, manager classifies, user escalates as needed. Fully codified in `plan-authoring-cascaded.md`.
2. **Step 9** (synthesis + handoff): post-cascade synthesis doc + design-doc amendments + execution plan. Just codified 2026-05-19 (María + Tiberius).
3. **The MIDDLE is documented. Both ENDS were not.**

Step 9 was codified after Run 3's empirical evidence (Manager Tiberius ad-hoc'd 1,225 LOC of post-cascade synthesis work that the workflow guidance didn't require). The mirror gap — **Step 0, the preparation phase that takes a raw design doc and produces cascade-ready inputs** — was just surfaced by Rick on 2026-05-20T01:13 UTC: "When you briefed Mr Radio on the Phase 7 slicing manifest, you had to explain the shape of the input to the cascade process. Is that codified anywhere so someone else can come in cold?"

**Honest answer: NO.** What I (Tiberius) verbally walked Mr Radio through was tribal knowledge living only in my manager-seat experience. A different manager, or a different fresh-cast cascade preparer, would have to re-derive the input-shape from scratch (or skip steps entirely).

### 1.2 Diagnosis — symmetric to Step 9: omission, load-bearing

**Same pattern as Step 9**: the cascade workflow guidance documents the EXECUTION of the cascade (Steps 1-8) but not the PREPARATION (Step 0) or the SYNTHESIS-HANDOFF (Step 9). Both ends are required for a cold-cast cascade to fire successfully; both were ad-hoc'd by the manager in Runs 1-3; both deserve codification.

**Omission, load-bearing**: without Step 0 workflow, the cascade is gated on the manager having internalized the input-shape from past runs. A new manager (or a designated preparer in a future cast) would face a re-invention problem. The workflow becomes high-friction-and-error-prone for any cast that doesn't include a veteran manager.

**Why the gap surfaced now and not earlier**: Runs 1-3 all used a manager (Tiberius) who had been involved in pre-cascade design work as a participant. The manager carried the input-shape context implicitly. Today's onboarding of Mr Radio (a returning cast member rotating to a NEW role — Phase 7 slicing manifest author) was the first time the input-shape needed to be transferred to a fresh-context-cold author. It surfaced.

## 2. Step 0 Sub-Step Spec

Step 0 fires when a manager (or designated preparer) takes a raw design document and prepares it for cascade entry. Six sub-steps, sequential.

### 2.1 Step 0.1 — Input intake + slice-ability assessment

**Trigger**: a raw design document lands on the manager's desk. Examples: parent design doc from a prior planning session, a roadmap entry that names a phase to scope, a design-mode artifact from `/p-is-p-01-planning`.

**Manager activity**:
- Read the raw design doc end-to-end
- Identify scope boundaries: what's in, what's out, what's deferred
- Assess slice-ability: does this naturally decompose into N sub-features, OR is it one cohesive surface?
- Cross-reference against any pre-existing slicing manifest in the project's R&D directory (e.g., `07-phase6-slicing-manifest.md` as a precedent shape)

**Decision**: sliced (N >= 2 sub-slices) OR unsliced (single design + single cascade run).

**Output of 0.1**: a written decision (one paragraph) declaring sliced vs unsliced, with rationale citing the raw design doc's natural sub-features (or absence thereof). Posted to the project's R&D directory as a brief decision note OR folded into the next sub-step's slicing manifest.

**Escalation**: if the slice-ability is ambiguous, the manager surfaces to user via `ask_multiple_choice` with pros/cons + recommendation per `feedback_always_include_pros_cons_recommendation`.

### 2.2 Step 0.2 — Slicing manifest (if sliced)

**Trigger**: 0.1 declared sliced.

**Manager activity**: author a slicing manifest at the project's R&D directory, numbered as the next sister doc to the raw design doc. Pattern: existing `07-phase6-slicing-manifest.md`.

**Required content per slice**:
- **Scope** — what's in this slice (be specific: function names, file paths, AC sketches)
- **Source citations** — where the requirement comes from (raw design doc section / roadmap entry / pre-existing R&D doc)
- **Dependencies** — prior-phase artifacts this slice consumes; libraries / APIs / SDK pinned versions
- **Independence** — claim how this slice doesn't conflict with the others
- **Out of scope per slice** — what is explicitly NOT in this slice (server-side, multi-tab, etc.)

**Manifest-level content** (alongside per-slice entries):
- **Recommended order** — which slice goes first and why (empirical rationale per ordering)
- **Permanently out of scope across all slices** — items the raw design doc already excludes
- **Per-slice file naming convention** — design doc + review findings + code-execution plan naming pattern
- **Acceptance per slice** — what does "this slice is done" look like

**Output of 0.2**: `NN-phaseX-slicing-manifest.md` (or equivalent naming) at the canonical R&D path.

**Skip 0.2 if unsliced**: proceed directly to 0.3 with the slice-count effectively N=1.

### 2.3 Step 0.3 — Per-slice design docs (or single design doc if unsliced)

**Trigger**: 0.2 complete (or 0.1 declared unsliced).

**Author activity**: extend the raw design doc into one or more cascade-ready design docs.

**Cascade-ready means**:
- Scope statement per slice (verbatim from slicing manifest)
- Q-decisions enumerated + PROPOSED stance per Q (not yet ratified — that's 0.5)
- Pre-flight Recon items enumerated (verify-at-code-write items per `feedback_audit_plans_at_execute_time`)
- Reuse map (functions / utilities / patterns to reuse vs invent)
- Files affected — rough inventory (will tighten during cascade)
- Acceptance criteria — draft (will refine during cascade)
- Out-of-scope confirmation (carryover from slicing manifest, sliced for this slice)

**Output of 0.3**: per-slice design doc(s) at the project's R&D directory. Pattern: existing `10-phase6c-persona-focus-recorder-design.md`.

**Authorship**: by default the SAME person who authored the slicing manifest (often the manager). If multiple authors collaborate (e.g., one per slice), they coordinate via DM to keep cross-slice consistency.

### 2.4 Step 0.4 — Q-decision matrix per slice

**Trigger**: 0.3 design doc(s) drafted.

**Author activity**: extract or refine the Q-decisions from the design doc into a matrix. Per slice, per Q:
- Q-number (e.g., Q-A1, Q-A2, ...) with cluster letter matching slice (A/B/C/D)
- Question statement (1-3 sentences)
- PROPOSED answer (the author's stance)
- Alternatives walked (1-2 sentences each on options NOT chosen + tradeoff)
- Recommendation per `feedback_always_include_pros_cons_recommendation`
- Conditional-executability flag if the answer depends on a Recon item resolution (per `feedback_audit_plans_at_execute_time`)

**Output of 0.4**: Q-decision matrix per slice — typically table-shaped, embedded in the per-slice design doc OR as a sister doc.

### 2.5 Step 0.5 — Pre-cascade user ratification

**Trigger**: 0.4 Q-decision matrix complete.

**Manager activity**: surface the Q-decision matrix to user via `ask_multiple_choice` or `converse`. Per cluster:
- Walk user through Q-decisions in cluster-batches (per `feedback_pip_plan_review_is_sequential` discipline)
- User ratifies PROPOSED stance OR flips to alternative OR opens a sub-discussion
- Ratifications captured in the Q-decision matrix as "✅ RATIFIED YYYY-MM-DD" markers

**Output of 0.5**: ratified Q-decision matrix per slice. Cascade Step 1 can fire against the ratified inputs.

**Skip-or-streamline option**: for low-stakes slices where all PROPOSED stances are obviously correct, the manager may bundle the ratification ask into a single `ask_yes_no` per slice ("All Q-decisions per slice X PROPOSED stances correct — ratify?"). Speeds the gate when content is uncontroversial. Use sparingly; complex slices warrant per-Q walkthrough.

### 2.6 Step 0.6 — Cascade-readiness gate (state flip)

**Trigger**: 0.5 user ratifications complete.

**Manager activity**:
- Run the **Step 0 light-review gate** (see §5 below) — single reviewer confirms cascade-ready
- Run the **Pre-cascade Recon checklist** (see §6 below) — author confirms standing-workflow guidance + persona-onboarding artifacts in place
- Declare cascade state: `cascade_input_ready`
- DM the cast (or notify user if cast is being spun up at cascade time) with the entry-point doc paths + brief

**Output of 0.6**: cascade state is `cascade_input_ready`. Cascade Step 1 (Stage 0 author draft per `/plan-authoring-cascaded`) can fire.

## 3. Artifact Specs

### 3.1 Slicing manifest

**Naming**: `NN-phaseX-slicing-manifest.md` where NN is the next-available number in the design-doc directory; phaseX is the parent phase identifier.

**Length**: typically 100-300 LOC; concise per-slice tables + slicing-rationale narrative.

**Required sections** (per `07-phase6-slicing-manifest.md` precedent):
- §1 Cadence per slice (one-liner about per-slice plan-review pass)
- §2 Slice status (live) — table updating as slices close (e.g., 6a ✅ CLOSED 2026-05-06)
- §3 Why slice (recap) — 1-2 paragraph justification anchored in surface-area vs single-cycle scope
- §4 Slice boundaries — per-slice subsection with scope + dependencies + independence claim
- §5 Recommended order — sequencing rationale + dependency-respect notes
- §6 Permanently out of scope (across all slices)
- §7 Per-slice file naming convention
- §8 Acceptance for each slice
- §9 Where this manifest lives in the cadence

**Acceptance**:
- All slices independently reviewable
- Cross-slice dependencies explicit
- Recommended order respects dependencies (don't sequence slice B before slice A if B consumes A's contract surface)
- Per-slice file naming convention matches the precedent shape from the same R&D directory

### 3.2 Q-decision matrix

**Format**: table embedded in the per-slice design doc OR as a sister doc.

**Required columns**:
- Q-number (e.g., Q-A1)
- Question (1-3 sentences)
- PROPOSED stance
- Alternatives (1-2 lines each)
- Recommendation
- Conditional-executability flag (if PROPOSED depends on a Recon item)

**Acceptance**:
- Every Q has a PROPOSED stance (no "TBD" — that means the question isn't ripe)
- Every Q has at least one named alternative (no "no other option considered")
- Recommendations cited rationale (not just "this is best")
- Conditional-executability flags exhaustive (no hidden conditionals)

### 3.3 Author-onboarding standing-workflow guidance list (per §6 below)

**Format**: brief checklist embedded in the slicing manifest §1 cadence OR as a sister "Step 0 onboarding pointer" doc.

**Required content**: list of standing feedback memories that apply to authoring + cascade work for THIS project. See §6 for the canonical pre-cascade Recon checklist.

## 4. Authorship + State

### 4.1 Authorship — Manager-default

By default, the **Manager** (the persona who will run the cascade) authors Step 0. Rationale:
- Manager will be the synthesizer at Step 9 too; carrying the design context end-to-end reduces handoff cost
- Manager has the cross-section visibility needed to identify cross-slice dependencies in 0.2
- Manager already maintains the Q-decision authority for the cascade itself

**Escape hatch**: workflow guidance may allow a "designated preparer" role if a manager is over-committed or if the project benefits from separating design-preparation from cascade-execution (e.g., a parent-design-doc author IS the natural Step 0 preparer; the cascade manager picks up at Step 1). For v1, default = Manager.

### 4.2 New state — `cascade_input_ready`

A new closure_action enum value in `plan-authoring-cascaded-defaults.md`.

**State semantics**: Step 0 has completed all 6 sub-steps; the slicing manifest exists (if applicable); per-slice design docs + Q-decision matrices exist; user has ratified Q-decisions; light-review gate passed; pre-cascade Recon checklist passed. The cascade Step 1 can fire.

**Distinct from `cascade_complete`** (existing state): `cascade_complete` is the state AFTER Steps 1-8 finish. `cascade_input_ready` is the state BEFORE Step 1 fires. The two are at opposite ends of the cascade lifecycle.

**Distinct from `implementation_handoff_ready`** (Step 9 state): `implementation_handoff_ready` is the state AFTER Step 9 closes. The full lifecycle is:
- `raw_design_received` → (Step 0) → `cascade_input_ready` → (Steps 1-8) → `cascade_complete` → (Step 9) → `implementation_handoff_ready` → (implementation) → `shipped`

### 4.3 Cold-context test analog at 0.5

The Step 9 cold-context test asks: "can the implementer pick up the 3 artifacts cold?" The Step 0 analog asks: "**can the cascade Stage-0 author pick up the per-slice design doc + Q-decision matrix cold?**"

Manager (or preparer) reads the per-slice design doc(s) + Q-decision matrix as if they were a fresh-cast Stage-0 author. Passes if:
1. They can identify the scope of the slice without back-referencing the raw design doc
2. They can identify what PROPOSED Q-decisions to carry forward into Stage-0 verbatim
3. They can identify which Recon items need verification at code-write time
4. They can identify cross-slice dependencies without inferring them
5. Standing workflow guidance memories that affect the slice's implementation are listed (§6 below)

**Self-administration cost**: ~10-15 min for the manager to read the per-slice docs end-to-end + answer the 5 questions.

## 5. Light-Review Gate at Step 0 Close (symmetric to Step 9)

### 5.1 Rationale

Per the Step 9 workflow guidance (codified 2026-05-19), synthesis-by-Manager-alone has a blind spot. The same blind-spot mechanism applies at Step 0: the Manager-as-preparer is in "construct the inputs" mode, not "challenge the input assumptions" mode. A fresh-eyes reviewer would pivot to challenge-mode and flag input-shape gaps before cascade Stage 0 fires.

**Empirical evidence forthcoming**: Step 0 hasn't been live-tested yet, but the symmetry with Step 9's blind-spot is strong. Run 4 (Phase 7's first cascade) will validate.

### 5.2 Reviewer selection

**Default**: one of the cast members assigned to the upcoming cascade — preferably someone OTHER than the manager. Ideal: the Persona 4 reviewer (Viability/Gap) whose rubric most closely matches "is this design cascade-ready?"

**Alternative**: María-as-workflow guidance-consultant (analogous to Step 9's reviewer alternative).

**Escape hatch**: if no cast member has bandwidth, the Manager declares self-administered cold-context test sufficient AND files a TODO for v2 to revisit.

### 5.3 Light-review rubric (NOT a full Persona 3/4/5 review)

Reviewer reads the Step 0 outputs (slicing manifest + per-slice design docs + Q-decision matrices) with this specific focus:

**Pass criteria** (all must hold):
1. **Slice independence check**: each slice's "independence" claim holds under scrutiny; cross-slice dependencies are explicit in both directions (provider AND consumer documented)
2. **Q-decision completeness check**: every Q has a PROPOSED stance + alternatives + recommendation; no hidden conditionals
3. **Recon item resolvability check**: every Recon item names where the verification happens (code-write, integration test, manual recon)
4. **Source citation check**: every requirement traces back to either the raw design doc or a pre-existing R&D doc (no "magic" requirements with no provenance)
5. **Sequencing soundness check**: recommended order respects all stated cross-slice dependencies (no cycles; no out-of-order dependencies)
6. **Author-onboarding completeness check**: standing workflow guidance memories applicable to the slice are listed (cross-ref §6)

**Output**: thumbs-up OR list of specific Step 0 gaps to address before cascade Stage 0 fires.

**Cost**: ~15-20 min reviewer-time.

### 5.4 Manager response to light-review findings

If reviewer thumbs-up: cascade state flips to `cascade_input_ready`.

If reviewer finds gaps: Manager addresses in slicing manifest / design doc / Q-decision matrix updates. Capped at **1 revision turn** (single-pass refinement; no Round-2). If reviewer finds MORE gaps after Manager revision: escalate to user.

## 6. Pre-Cascade Recon Checklist

This is the load-bearing piece for cold-cast onboarding. Codifies the standing-workflow guidance knowledge that I had to verbally walk Mr Radio through.

### 6.1 Required content (template)

The author of Step 0 outputs MUST produce (and the light-reviewer at §5 MUST verify) a brief author-onboarding checklist:

**Standing feedback memories that apply** (PROJECT-WIDE — these apply across all slices):

| Memory | What it gates | When to apply |
|---|---|---|
| `feedback_pip_plan_review_is_sequential` | Plan-review passes are sequential, never parallel | Cascade Stage transitions; never bundle |
| `feedback_100pct_coverage_multiplexer` (or project-specific equivalent) | 100% line + branch + function coverage | Every test gate |
| `feedback_always_include_pros_cons_recommendation` | Multi-option asks carry per-option pros + cons + recommendation + flip-condition | Every user-escalation |
| `feedback_tts_body_headline_and_takeaway_only` | Spoken `notify` body is headline + recommendation; pros/cons + flip-conditions in `abstract` | Every notify, every ask_multiple_choice |
| `feedback_recraft_speech_dont_pipe_terminal` | In speakerphone mode, spoken `message` is re-shaped for TTS | Every notify when speakerphone on |
| `feedback_doc_links_always_in_abstract` | Doc links go in `abstract` line 1, never in spoken body | When surfacing doc links |
| `feedback_persona_icon_adherence` | Sign DMs with YOUR persona icon, NOT another persona's | Every DM |
| `feedback_never_auto_commit_push` | Wait for explicit "commit" / "push" per change | Every commit decision |
| `feedback_test_server_monopolize_mode` | `:8000` is monopolize-mode; all submissions via `POST /api/test-suite/submit` with non-overlapping `scheduled_at` | Every `:8000`-bound test |
| `feedback_baseline_capture_disable_tfe` | Visual regression baseline-capture submissions include `auto_fix_on_failure: False` | Every visual baseline capture |

**Persona conventions** (PROJECT-SPECIFIC):
- List of available personas with current role assignments (e.g., Manager / Author / Reviewers / Implementer)
- Signing convention for DMs (your persona icon, NOT another persona's)
- Cast members currently active vs deferred

**Standing project-specific rules**:
- Test venue routing (e.g., :7999 dev vs :8000 monopolize-mode for the Lupin project)
- Commit posture (e.g., commit-only no-push pending explicit user authorization)
- Doc-viewer link routing rules (e.g., per-project scope routing for cross-repo links)

### 6.2 Format

Embedded in the slicing manifest's §1 cadence section OR as a sister "Step 0 onboarding pointer" doc.

**Recommended**: embed in slicing manifest §1 if the checklist is short (< 30 lines); spin out as a sister doc if it's long (> 50 lines).

### 6.3 Acceptance

The author-onboarding checklist passes the light-review gate (§5) if:
- All applicable standing memories are listed (no missing memory that would cause friction during cascade)
- Persona conventions match the current project state (no stale persona references)
- Project-specific rules are accurate for THIS cascade's context (e.g., test venue routing matches what the cast will actually use)

## 7. Open Questions

### 7.1 Slicing-manifest authorship — Manager-default vs raw-design-doc-author

In some cases, the person who authored the raw design doc is BEST positioned to author the slicing manifest (they have the deepest scope context). In others, the Manager-of-the-upcoming-cascade is best positioned (they understand the cascade execution constraints).

**My lean**: Manager-default for v1 (mirrors Step 9 default). Escape hatch documented.

### 7.2 Step 0.5 ratification — per-Q walkthrough vs bundled batch

Today's `ask_multiple_choice` cadence supports both per-Q walkthrough (high-stakes) and bundled batch (low-stakes). Should the workflow guidance prescribe one default, or leave it to Manager judgment?

**My lean**: Manager judgment. Complex slices warrant per-Q; uncontroversial slices can bundle.

### 7.3 Pre-cascade Recon checklist — project-specific vs cross-project

The standing memories list in §6.1 is heavy on Lupin-specific items (`feedback_100pct_coverage_multiplexer`, `feedback_test_server_monopolize_mode`). For cross-project cascades (e.g., a future PIP-only cascade), the checklist would be different.

**My lean**: workflow guidance specifies the TEMPLATE (what kinds of items belong on the checklist); per-project instantiations fill in the specifics. Step 0 workflow in PIP describes the template; the slicing manifest in each project's R&D directory hosts the project-specific checklist.

### 7.4 Light-review reviewer-selection — algorithmic vs Manager-pick

Same question as Step 9 §6.4: should workflow guidance specify an algorithm for picking the light-reviewer or leave it to Manager judgment?

**My lean**: Manager judgment for v1 (matches Step 9).
