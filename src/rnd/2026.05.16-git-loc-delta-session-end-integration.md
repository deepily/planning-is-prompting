## 1. Context & Framing

### 1.1) What María shipped

`cosa.repo.git_loc_delta` — a CLI module (entry `python -m cosa.repo.run_git_loc_delta`) that walks `git log --numstat` and buckets every file change by `(date, file_type)` with per-day totals. Three output formats (console / JSON / CSV). The CSV is **tidy-long** with a stable 6-column schema (`date,file_type,added,deleted,files_touched,commits`) and lands at a **stable per-branch filename** that overwrites cleanly each run:

```
{project_root}/io/git-loc-delta/{repo}-{branch-slug}-loc-delta.csv
```

María's README explicitly proposes integration into `/plan-session-end` (Use Case A — End-of-Day Daily Ritual).

### 1.2) Sister tool, not replacement

María's README positions `git_loc_delta` and `branch_analyzer` as **sister tools doing different jobs**, not v2 vs v1:

| | `branch_analyzer` | `git_loc_delta` |
|---|---|---|
| Question | "What did this branch change vs main?" | "What did I do each day on this branch?" |
| Driver | `git diff base...head` (3-dot) | `git log --numstat` (per-commit walk) |
| Granularity | Single branch-total bucket | Per-(date, file_type) buckets + per-day totals |
| Code/comment/docstring split | **Yes** | No (deferred `--classify-lines`) |
| Persistent artifact | No | **Yes** (stable per-branch CSV) |

The hybrid path below preserves both axes while making the per-day CSV the new default.

### 1.3) Approved path

User selected **Hybrid** via `ask_multiple_choice` (2026-05-16): swap `git_loc_delta` into §6.2 as the new default (per-day table + CSV side-effect); keep §6.5 baseline (`run_directory_analyzer`); demote `run_branch_analyzer` to an opt-in `--rich` flag for the PR-narrative use case.

## 9. Discovered During Execution — Upstream Path-Resolution Bug

**Symptom**: When `run_git_loc_delta --repo-path <other-repo> --branch --output csv` is invoked from `$LUPIN_ROOT/src` (the canonical cwd per María's README), the resulting CSV is **written to `$LUPIN_ROOT/io/git-loc-delta/{external-repo-name}-{branch}-loc-delta.csv`**, NOT to `<other-repo>/io/git-loc-delta/...`.

**Verified**: 2026-05-16 smoke test from this session.
- Command: `python -m cosa.repo.run_git_loc_delta --repo-path /...planning-is-prompting --branch --output csv`
- Expected CSV: `/...planning-is-prompting/io/git-loc-delta/planning-is-prompting-wip-v0.1.3-2026.02.16-continued-development-loc-delta.csv`
- Actual CSV: `/...lupin/io/git-loc-delta/planning-is-prompting-wip-v0.1.3-2026.02.16-continued-development-loc-delta.csv`

**Root cause** (suspected, not confirmed): `cosa.repo.run_git_loc_delta` resolves the default `--save-output` path via `cu.get_project_root()`, which reads the `LUPIN_ROOT` env var — i.e. always Lupin's root, regardless of `--repo-path`.

**Impact**: For the cross-repo session-end use case ("every session on every repo"), the persistent CSV must live with the repo it documents. The default behavior dumps all CSVs into Lupin's `io/` dir — wrong filesystem location, gitignored under Lupin's `.gitignore`, and cross-contaminating Lupin's working tree with artifacts about other projects.

**Workflow-side workaround** (applied in §6.2 of `session-end.md` during this session):

```bash
PROJECT_ROOT="$(git -C . rev-parse --show-toplevel)"
REPO_NAME="$(basename "$PROJECT_ROOT")"
BRANCH_SLUG="$(git -C "$PROJECT_ROOT" symbolic-ref --short HEAD)"
CSV_PATH="$PROJECT_ROOT/io/git-loc-delta/${REPO_NAME}-${BRANCH_SLUG}-loc-delta.csv"

python -m cosa.repo.run_git_loc_delta \
    --repo-path "$PROJECT_ROOT" \
    --branch \
    --output csv \
    --save-output "$CSV_PATH"  # ← explicit override
```

**Upstream fix recommendation** (for María's queue): when `--repo-path` is set to a path that is NOT `cu.get_project_root()` AND `--save-output` is NOT explicitly provided, resolve the default save path relative to `--repo-path` instead. Pseudocode:

```python
if args.save_output is None and args.output == "csv":
    base = Path( args.repo_path ).resolve()
    args.save_output = base / "io" / "git-loc-delta" / f"{base.name}-{branch_slug}-loc-delta.csv"
```
