# BUG: User-Broadcast Body Dropped on Per-Session Injection (Empty `USER BROADCAST` system-reminder)

**Filed:** 2026-06-03 ~03:17Z by María 🌸 (PIP session `af1465b9`, Workflow Steward)
**Reported by:** Rick ("This is a new bug!") — after sending the same broadcast twice, both arriving empty to María.
**Severity:** HIGH — silently drops user→fleet directives. A user can issue a fleet-wide order, see it "sent," and have **every recipient receive a blank**. Rick hit this while issuing an AFK overnight directive (a "go to 100% coverage, don't stop" order that the fleet never saw).
**Status:** OPEN — diagnosed, not yet fixed. Owner: cosa-voice / broadcast-listener (Lupin). Flagged to Tiberius 👑 + (likely) Rachel.

---

## 1. Symptom

Two `USER BROADCAST` `<system-reminder>` blocks were injected into María's session turns (broadcast_ids `83cb0566-1b04-4061-a699-0d54090a9420`, then `b0b31299-119a-49d3-9cf4-c922b3f3bb91`). **Both had an empty body** — the directive text was absent. María could only report "received but empty" twice.

## 2. Root-cause locus (verified — primary evidence)

Reading the `broadcasts` commons topic (`commons_read(topic="broadcasts")`) shows the **content is fully intact and correctly stored**, fanned out to all five target sessions:

| target_session_id | persona (inferred) |
|---|---|
| `af1465b9-…` | María 🌸 (me) |
| `1333e106-…` | Tiberius 👑 (manager) |
| `05fffb12-…` | worker |
| `f1fa7314-…` | worker |
| `cc7d836e-…` | worker |

Each row's `body` carries the **full** directive text; `metadata` carries `broadcast_id`, `target_session_id`, `sender_user_id`. So:

- **SEND path: OK** — the broadcast was accepted.
- **STORAGE / fan-out path: OK** — content is on the topic, one row per target session, body intact.
- **PER-SESSION INJECTION / RENDER path: BROKEN** — the step that turns a stored broadcast row into the `USER BROADCAST` system-reminder injected into a recipient's turn **drops the body**, delivering an empty reminder.

**Conclusion:** the bug is isolated to the broadcast→system-reminder injection/render layer (the listener-side hook that composes the `<system-reminder>` text), NOT the send or storage layers. Fix should target that composer: it is reading/emitting an empty body while the source row's `body` is non-empty.

## 3. Blast radius

- Every recipient of an affected broadcast gets a blank → **fleet-wide directive loss**, silent.
- Especially dangerous on **AFK hand-off**: the user issues an overnight order and goes to bed believing it landed. (Exactly this case — see §4.)
- Mitigation that DID work: the content is durably on the `broadcasts` topic, so any recipient who *polls* `commons_read(topic="broadcasts")` recovers it. Idle/heads-down sessions that don't poll never see it.

## 4. The live incident this surfaced in

Rick's intended broadcast (recovered verbatim from the topic):

> @maria @Tiberius I'm going to bed, given that you disqualified so many potential test sites as having already been implemented, I'm going to give you a new directive: I want you both to continue implementing tests until we have 100% coverage across all tiers, not just 1 or 2, understood?
> @Rachel @Cheech @Krishna: keep up the good work and don't stop until it's done!

Because the injection dropped the body, María (and presumably Tiberius + the three workers) received blanks. María recovered the text only because Rick re-pasted it directly into chat. **Remediation in-flight:** María re-fed the verbatim directive to Tiberius via DM (qid `3225d5e0`) so the manager can propagate to the fleet.

**Cross-session corroboration [verified-Tiberius, 2026-06-03 ~03:20Z]:** Tiberius (manager, session `1333e106`) independently confirmed via DM that **his** USER BROADCAST reminders also rendered **BLANK — twice** — and that SEND + STORAGE were fine, the per-session render dropped the body. This upgrades the failure from "presumed fleet-wide" to **verified across two independent sessions** (María + Tiberius), confirming the drop is in the shared injection/render layer, not a single-session fluke. Manager has re-briefed the fleet (Rachel/prediction_engine, Cheech/io_models, sam/test_suite, Krishna/reviewer) to the recovered all-tiers directive, so the directive loss is remediated even though the bug itself remains OPEN.

## 5. Suggested fix path (for the owner)

1. In the broadcast listener / system-reminder composer, trace where `body` is read when building the `USER BROADCAST` reminder — confirm it isn't keying off a wrong field (e.g. reading a non-existent `message` vs the stored `body`) or truncating on a delimiter.
2. Add a regression test: post a broadcast with a known body → assert the injected system-reminder body equals the source.
3. Consider a delivery-integrity guard: if a composed broadcast reminder body is empty while the source row body is non-empty, log/alarm rather than inject a silent blank (deliver-or-flag, mirroring the lever-A outbox philosophy from the Sam-TTS-storm fix).
