## 1. The hazard this prevents

Two arbiters must never run at once. Today the **in-process** arbiter is LIVE: `arbiter_bootstrap.submit_arbiter_if_absent(...)` is wired into the FastAPI lifespan at `src/lupin_app/main.py` (was `src/fastapi_app/main.py:847-848` pre-rename; line numbers re-pin at build time) and submits one `ArbiterConsumerJob` to CJ Flow at every server start (verified firing on :7999 — log line `[ARBITER] standing arbiter submitted to CJ Flow (B1 standing cadence)`).

When the **standalone :8001** arbiter (deploy-doc Loop B) goes live, if the in-process one is still running, **both will tap the same managers** → duplicate advisory DMs, duplicate escalations, duplicate auto-pings. The cure would double the noise it exists to reduce.

**Invariant R0 (hard):** **never two arbiters actuate the fleet at once.** A brief **zero-arbiter** window is bounded-and-acceptable ONLY during the supervised break-before-make cutover (§4) — never a steady state. (Stated this way deliberately, per María's conformance review: a naive "never zero" reading would push an implementer toward *make-before-break* — starting :8001 actuation before disabling the in-process arbiter — which creates the exact two-arbiter overlap R0 exists to prevent. The hazard is two, not zero; a few minutes blind during a supervised cutover is safe, double-tapping is not.)

## 2. The flag

| Field | Value |
|---|---|
| **INI key** | `arbiter in-process bootstrap enabled` |
| **Section** | `[Lupin: Baseline]` in `src/conf/lupin-app.ini` (inherited by all envs) |
| **Type / default** | boolean, **default `True`** (preserves today's behavior until :8001 is proven live) |
| **Read site** | once, at startup, in the lifespan handler — NOT per-request, NOT per-poll |
| **Splainer** | matching entry REQUIRED in `src/conf/lupin-app-splainer.ini` (per the standing splainer mandate) |

**Default rationale:** `True` = current behavior (in-process arbiter runs), so a config-less deploy is unchanged. The cutover is a deliberate one-line flip to `False` performed as part of standing up :8001 — not a silent default change.

## 3. Where it gates (design intent — no code yet)

The lifespan call at `main.py:847-848` becomes conditional on the flag, read **once** at startup:

- If the flag is **True** → `submit_arbiter_if_absent(...)` runs as today (in-process arbiter active).
- If the flag is **False** → the lifespan **skips** submission entirely AND logs a clear, greppable line (e.g. `[ARBITER] in-process bootstrap DISABLED by config — standalone :8001 service owns the loop`) so the absence is observable, never silent.

**Read-once guard:** the value is resolved a single time during lifespan startup and not re-read; this avoids a mid-run flap where a hot INI reload could leave two arbiters briefly co-active. (A flag flip takes effect on the next server start — acceptable, because the :8001 cutover is a deliberate operational step, not a hot toggle.)

The existing **single-instance guard** (`arbiter_already_present`) stays — it remains the belt against a double *in-process* submission; R0 is the separate belt against in-process-vs-:8001 co-activation.

## 4. Atomic cutover procedure (operational, at :8001 go-live)

The flip and the :8001 start must be sequenced so the gap contains **zero** arbiters only momentarily and **never two**:

1. :8001 service installed + supervised but Loop B **not yet actuating** (Loop B disabled or in start-period warm-up).
2. Set `arbiter in-process bootstrap enabled = False`.
3. Bounce :7999 (and :8000) → in-process arbiter does NOT re-submit (verify via the DISABLED log line + `/api/queue/pool-status` shows no arbiter job).
4. Enable :8001 Loop B actuation.
- Ordering guarantees: between steps 3 and 4 there is a brief **zero-arbiter** window (safe — a few minutes blind is far better than double-tapping); there is **never** a two-arbiter window.
- Rollback: re-flip to `True` + bounce restores the in-process arbiter; disable :8001 Loop B. Symmetric.

## 5. Test matrix (100% line + branch + function on the flag wiring)

The flag introduces one branch at the gate; both arms + the read-once contract must be covered (per the Lupin-wide 100% gate):

| # | Case | Asserts |
|---|---|---|
| T1 | flag **True** (default) | lifespan calls `submit_arbiter_if_absent` exactly once; arbiter job enqueued |
| T2 | flag **False** | lifespan does NOT call submit; no arbiter job enqueued; DISABLED log line emitted |
| T3 | flag absent (key missing) | resolves to default `True` → behaves as T1 (default-path branch covered) |
| T4 | flag malformed (non-bool string) | resolves via the config manager's bool coercion; document + test the coercion outcome (no crash; degrade to a defined value) |
| T5 | read-once contract | the flag is read a single time at startup (assert the resolver is invoked once, not per-request) |

T1–T3 cover both branch arms + the default branch; T4 covers the coercion edge; T5 locks the read-once invariant. No `pragma: no cover` anticipated.
