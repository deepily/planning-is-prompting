## 1. Problem

When a manager reaps a worker, that worker's **open task-store rows do not move**. The rows keep `owner_persona = <reaped worker>`, so the work becomes an **orphan**: invisible to the (alive) manager's own owed-work view, never picked up, silently stranded until a human notices.

Tonight's triage found **3 such orphans**, all under Tiberius's accountability:

| Row | Status | Stale owner | Work |
|---|---|---|---|
| `a98bfaca` | in_progress | krishna | Alembic migration-drift follow-on |
| `f27a4054` | in_progress | clayton | Arbiter false-staleness fix (held at design-checkpoint) |
| `45d269df` | blocked | rachel | Lane-1 commons-DM full-removal |

**Root gap**: the store API has **no reassign-owner verb**. The only re-homing path today is a human (or manager) reading a DM and acting manually. That is not self-healing — an orphan can outlive the reap indefinitely, exactly the "wake up to orphans" failure Rick wants eliminated.

This is adjacent to (but distinct from) the false-idle bug `c6751cf8`: that bug hides a *live owner's* work from the oracle; *this* gap strands a *dead owner's* work with no owner at all.

---

## 2. Design goals

1. **No orphan survives a reap** — when a worker dies, its open work must immediately have a live, accountable owner.
2. **Accountability rolls UP** — orphaned work reverts to its `accountable_manager` (consistent with the existing parent/child accountability rule), never to an arbitrary live persona.
3. **Audited** — every ownership change emits a store audit event (who/when/why), so the trail survives `/clear` and is fleet-visible.
4. **Fail-loud, never fail-silent** — a reclaim that cannot find a live manager must surface (notify), not drop the row.
5. **Minimal new surface** — prefer one primitive that several mechanisms reuse.

---

## 3. Options

### Option A — `task_reassign` verb (the primitive)
Add a store verb that mutates `owner_persona` (and optionally `accountable_manager`) on an existing row, appending an audited `reassigned` event.

- **Pros**: small, well-scoped; gives managers (and the steward) an explicit tool; the building block both B and C need.
- **Cons**: still **manual** on its own — solves "I have no verb," not "orphans heal themselves." Necessary but not sufficient.

### Option B — Auto-reclaim at the reap seam
Hook the reap path (manager-autonomy harvest, and/or the stop-hook when a session terminates): on reap, query the store for the reaped persona's open rows and auto-revert each `owner_persona → accountable_manager`, with an audit event + a manager notify.

- **Pros**: heals at the exact moment the orphan is created; deterministic; cheap (only fires on reap).
- **Cons**: only covers reaps that go **through** the instrumented seam. A worker that dies ungracefully (crash, lockout, killed tmux) without hitting the reap path leaves the orphan — needs a backstop.

### Option C — Arbiter reconciliation detector (the backstop)
Add a periodic arbiter detector (sibling of the existing D2/D3/D4 detectors in `arbiter_job.py`): each poll, reconcile open-row `owner_persona` against the live-fleet roster (the same bridge∪commons∪idle∪stop liveness union the arbiter already computes). Any open row whose owner is **not live** → auto-revert to `accountable_manager` + notify, OR raise a decision if the manager is also dead.

- **Pros**: catches **every** orphan regardless of how the owner died (the safety net B can't guarantee); reuses the arbiter's existing liveness machinery and notify sink.
- **Cons**: polling latency (orphan exists until the next sweep); must debounce against a worker that's briefly quiet-but-alive (use the same freshness window the arbiter already tolerates, e.g. the staleness threshold) to avoid reclaiming a living worker's row.

---

## 4. Recommendation

**Ship A as the primitive, then B + C on top of it.**

- **A** gives every actor (managers, steward, the reap hook, the arbiter) one audited way to move ownership.
- **B** heals the common case instantly (graceful reap).
- **C** is the backstop that guarantees goal #1 even for ungraceful deaths — this is the piece that makes the system genuinely *self-healing*.

Reclaim target = `accountable_manager`. If `accountable_manager` is itself not live, **escalate** (a `decision` row in Rick's court / a notify) rather than guessing — never silently re-home to a random live persona.

Edge cases to specify in build:
- **Quiet-but-alive guard** (C): only reclaim when the owner is absent from the liveness union beyond the arbiter's staleness window — not merely "hasn't posted in 2h."
- **Generic-persona respawn** (the "extra 1 / extra 2" case tonight): if a manager respawns a worker that gets a generic persona name, the old-named rows still orphan. Reclaim-to-manager + a manager re-assign (via A) to the new worker is the clean resolution; document the pattern.
- **Worktree-held work** (e.g. clayton's `/tmp/clayton-arbiter-tuning-wt`): reclaiming the *row* does not recover the *worktree*; the reclaim notify must flag held/worktree state so the manager can decide resume-vs-restart.
- **No double-reclaim**: B and C must be idempotent (an already-reclaimed row is a no-op).
