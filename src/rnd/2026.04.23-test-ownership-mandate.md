# Plan: Test Ownership Mandate — Embed for Maximum Cross-Repo Effect

## Context

**Problem**: The user explicitly holds a strong operating principle that is absent from every layer of their Claude-Code configuration:

> "The AI architect you're working with is not a tester — they are a designer and a user. It is your job, Claude Code, to proactively test across the entire testing pyramid (unit → E2E) so the designer ends up **using** the software without ever being asked to test it. Assume there's not enough time in the world to manually test anything."

**Why it matters now**: An Explore-agent sweep of 30+ workflow documents, two CLAUDE.md files, and all slash-command wrappers confirms this sentiment is **not** expressed anywhere. Worse, two places express the *opposite* — `workflow/bug-fix-mode.md` Step 7 instructs Claude to *ask the user* which further tests to run, and `workflow/testing-remediation.md`'s `SELECTIVE` / `CRITICAL_ONLY` scopes imply the human picks scope. Adjacent infrastructure exists (tier definitions, post-edit `py_compile` mandate, change-impact analysis skill), but the **role-separation rule** and the **scarcity argument** are missing.

**Intended outcome**: Bake the mandate in at the highest-leverage point (global CLAUDE.md — loaded in every session of every repo), reinforce it in the testing-specific workflow docs, and eliminate the existing anti-sentiment language so the corpus stops contradicting itself. User chose **"Comprehensive + bug-filing too"** scope, so the mandate also covers autonomous triage of bugs Claude discovers during testing.

## Proposed Language (Canonical Block for `~/.claude/CLAUDE.md`)

Insert as a new subsection **inside the existing `## TESTING & INCREMENTAL DEVELOPMENT` section** in `~/.claude/CLAUDE.md`, placed immediately after the "Philosophy" line and before the "Quick Reference" table:

```markdown
### TEST OWNERSHIP MANDATE

**MANDATE**: The human collaborator is the **designer and user** of the software — NOT the tester.
You, Claude Code, own testing across the full pyramid (unit → integration → E2E) AND the triage
of bugs you discover. Do not hand manual QA or bug-capture back to the human.

**Operating assumption**: There is not enough time in the world for the human to manually test
anything. Every "please verify this works" or "let me know if you hit a bug" hand-off is a
failure of this mandate.

**Role separation**:

| Responsibility | Owner |
|----------------|-------|
| Decide WHAT to build | Human (designer) |
| USE the software | Human (user) |
| Write tests | Claude |
| Run tests | Claude |
| Triage failures | Claude |
| File bugs discovered during testing | Claude (into `bug-fix-queue.md` when bug-fix-mode is active) |

**PROHIBITED phrases** — never end a code change with any of these:
- "Please try it and let me know if it works."
- "Can you verify this?"
- "Let me know if you hit any bugs."
- "Which additional tests should I run?"

**Required behavior**:
- After any behavior-changing code change, proactively extend the pyramid — a unit test for
  the changed unit, an integration test for the affected collaboration surface, and an E2E
  test for the user-observable behavior when a runnable surface exists. Use the
  change-impact-analysis skill to scope; **Claude decides the scope, not the human**.
- Report test results in tabular form (pass/fail per tier) so "tests were actually run" is
  visible at a glance.
- If a test genuinely cannot be automated (subjective feel, external-service gating, UI
  polish), state this **explicitly with the specific reason** — silent deferral to the
  human is prohibited.
- Bugs discovered during testing are auto-queued to `bug-fix-queue.md` (when bug-fix-mode
  is active) or surfaced in a tagged section of the session notes — do not ask the human to
  remember to file them.

**Why**: The designer's time is the scarce resource. Manual QA by the human is the
anti-pattern this mandate exists to prevent. A code change is not "done" when it compiles;
it is "done" when tests pass across every applicable pyramid tier AND the human can use
the software without being asked to check for breakage.

**How to apply**: Treat every code-touching tool call as incomplete until tests exist, run,
and pass at every applicable tier. The POST-EDIT VERIFICATION mandate (py_compile) is the
minimum floor, not the ceiling — compile-clean is a prerequisite for testing, not a
substitute for it.
```

**Design notes on the language**:

- Uses the **MANDATE + table + Why/How-to-apply** skeleton already established by `PARALLEL SESSION SAFETY`, `POST-EDIT VERIFICATION`, and `INTERACTIVE TOOL ROUTING` sections — tone and weight match.
- `PROHIBITED phrases` list makes the rule concrete and auditable; the human can grep their own past transcripts for violations.
- Table makes the role boundary graphic and scan-friendly.
- Explicit cross-reference to `POST-EDIT VERIFICATION` and the `testing-development` skill reinforces the floor/ceiling relationship rather than creating parallel guidance.
- Keeps the user's scarcity argument ("not enough time in the world") verbatim-flavored — that phrasing is memorable and worth preserving.

## Bug-Filing Extension (Concrete Edit)

### `workflow/bug-fix-mode.md`
Add a short subsection (after the "Atomic commits per bug" section) titled **"Autonomous Bug Capture"**:

> When Claude discovers a bug during testing or exploration, Claude appends it to
> `bug-fix-queue.md` under `Queued` with a Claude-authored title, reproduction
> steps, and a short root-cause hypothesis if available. The human is not asked to
> capture the bug. When the current session wraps, queued bugs are visible in the
> session summary — triage into `In Progress` is a separate, explicit action the
> human (or a future session) takes.
