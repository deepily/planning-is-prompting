## 1. Motivation

Today, when a Claude Code session starts, the cosa-voice MCP server's SessionStart hook allocates a **random** persona via `pick_unallocated_persona()`. There is no way for the user to declare a per-repo default ("when I open Claude Code in planning-is-prompting, I always want Rachel").

The existing per-session swap path — `/plan-session-start María` — works, but:

- Requires the user to remember to type the slash command on every fresh session
- Requires the user to remember the persona arg
- Doesn't survive `/clear` cleanly across days (the swap is one-shot)
- Doesn't generalize across the other repos the user operates in (Lupin, cosa-voice, lupin-mobile, etc.)

The user wants **per-repo declarative defaults** so each repo gets a stable narrative voice without manual intervention.

---

## 2. Proposal

Add a per-project environment variable read by cosa-voice's persona allocator at SessionStart hook time.

**Naming**: `COSA_VOICE_PREFERRED_PERSONA__<PROJECT_UPPER>`

Examples (per Rick's broadcast 2026-05-19 16:5x UTC — PIP=María, Lupin=Tiberius are the canonical defaults; others illustrative):
```bash
export COSA_VOICE_PREFERRED_PERSONA__PLAN=María
export COSA_VOICE_PREFERRED_PERSONA__LUPIN=Tiberius
export COSA_VOICE_PREFERRED_PERSONA__COSA_VOICE=Rio
export COSA_VOICE_PREFERRED_PERSONA__LUPIN_MOBILE=Aurora
```

**Why this naming**:

- `COSA_VOICE_*` prefix — the allocator lives in cosa-voice, not in any consumer project. Honors the "workflow defaults travel with the workflow" rule (memory: `feedback_workflow_defaults_travel_with_workflow.md`).
- `__<PROJECT>` suffix — the cosa-voice session bridge already knows the project name (visible in `get_session_info()` → `"project": "plan"`). Suffixing the env var with the project lets one universal lookup pattern serve every repo.
- Double-underscore separator — keeps `COSA_VOICE_PREFERRED_PERSONA` browseable as a prefix in `env | grep`, and matches conventions like `XDG_*__*` patterns elsewhere.

**Project detection**: at allocator time, cosa-voice already resolves `project` from session-bridge creation (`session_info["project"]`). Convert to upper-case, normalize hyphens to underscores (`cosa-voice` → `COSA_VOICE`), then look up `COSA_VOICE_PREFERRED_PERSONA__<NORMALIZED>`.

## 3. Conflict Behavior (per user's verbal spec, 2026-05-19)

When the requested persona is **already allocated to another live session**, the allocator MUST:

1. Allocate a random persona normally (existing fallback path — no blocking, no interactivity at the hook level)
2. Fire a `notify()` to the user with `priority="high"` describing the conflict:
   > "Requested persona María is held by session abc12345. Allocated Aurora instead. To claim María, kill that session and restart this one."
3. Optionally include the holding session's project/topic in the abstract to help the user identify which session to terminate.

**Explicitly out of scope**: auto-killing the holding session, interactive prompt at allocator time, queue-and-wait for the persona to free up. The user's stance: *"It's up to the user at that point to release that allocation manually by killing the Claude code session that the desired persona is currently attached to."*

When the requested persona name is **invalid** (typo, not in pool):

1. Allocate a random persona normally
2. Fire `notify()` with the available pool list:
   > "COSA_VOICE_PREFERRED_PERSONA__PLAN=Marya is not a known persona. Allocated Rachel instead. Available: María, Rio, Aurora, Sky, …"

When the env var is **unset** for the current project: existing random-allocation behavior, no notify.

### 4.1. Lupin / cosa-voice changes

**Files** (verified locations from `find` 2026-05-19):

- `src/cosa/rest/voice_persona_helpers.py` — add `pick_preferred_persona_from_env(project: str)` function that:
  - Normalizes project name (`cosa-voice` → `COSA_VOICE`, `plan` → `PLAN`)
  - Reads `os.environ.get(f"COSA_VOICE_PREFERRED_PERSONA__{normalized}")`
  - Returns `None` if env var unset
  - Returns persona-name string if set (does NOT validate against pool here — let the caller decide)
- `src/cosa/rest/voice_persona_helpers.py` — modify `allocate_persona_for_session()` (line 312) to:
  1. Check `pick_preferred_persona_from_env(project)` first
  2. If returned, try `_find_persona_in_pool()` + `pick_unallocated_persona()` for that name
  3. If successful: allocate, return normally (no notify)
  4. If not found in pool OR already held: fall through to random `pick_unallocated_persona()` + enqueue a conflict notify (see §4.3 for delivery mechanism)
- `src/lupin_cli/claude_code/hooks/register_session.py` — no changes expected; this file calls into the helpers above, which now silently honor the env var.

**Pseudo-diff for `allocate_persona_for_session()`**:

```python
def allocate_persona_for_session( session_id, project, pool, allocations, debug=False ):
    # NEW: try env-var preference first
    preferred_name = pick_preferred_persona_from_env( project )
    if preferred_name is not None:
        match = _find_persona_in_pool( pool, preferred_name )
        if match is None:
            _queue_conflict_notify(
                session_id, kind="invalid_name",
                requested=preferred_name,
                available=[ p.name for p in pool ]
            )
        elif match.name in allocations.values():
            holding_sid = next( sid for sid, name in allocations.items() if name == match.name )
            _queue_conflict_notify(
                session_id, kind="held",
                requested=preferred_name,
                holding_session_id=holding_sid
            )
        else:
            # happy path — allocate preferred
            allocations[ session_id ] = match.name
            return match

    # existing fallback — random allocation
    return pick_unallocated_persona( pool, allocations )
```

### 4.2. Env-var lookup helper

```python
def pick_preferred_persona_from_env( project: str ) -> Optional[ str ]:
    """
    Read COSA_VOICE_PREFERRED_PERSONA__<PROJECT> from the environment.

    Requires:
        - project is a non-empty string (e.g., "plan", "lupin", "cosa-voice")

    Ensures:
        - returns persona name string if env var set
        - returns None if env var unset
        - does NOT validate the name against any pool
        - normalizes project name: lowercase → UPPER, hyphens → underscores

    Examples:
        project="plan"        → reads COSA_VOICE_PREFERRED_PERSONA__PLAN
        project="cosa-voice"  → reads COSA_VOICE_PREFERRED_PERSONA__COSA_VOICE
        project="lupin"       → reads COSA_VOICE_PREFERRED_PERSONA__LUPIN
    """
    if not project:
        return None
    normalized = project.upper().replace( "-", "_" )
    env_key    = f"COSA_VOICE_PREFERRED_PERSONA__{normalized}"
    return os.environ.get( env_key )
```

### 4.3. Conflict notify delivery

The allocator runs **before** the MCP tools are surfaced to the Claude Code session — so it can't call `notify()` directly from the helper module (no live session-id-to-bridge mapping yet at the moment of allocation, depending on ordering).

Two delivery options to evaluate during implementation:

**Option α — Inline notify after bridge creation (preferred)**:
After `register_session.py` finishes creating the bridge file AND running the allocator, if the allocator queued a conflict, immediately call `notify(session_id=<new>, ...)`. The bridge file exists by this point so the notify route can reach the session.

**Option β — Defer to first session-info poll**:
Stash the conflict in the bridge file's metadata (`conflict_warning: {...}`). The cosa-voice MCP server returns it in the next `get_session_info()` response. Claude Code's Phase A startup reads it and surfaces it. Lower coupling but couples to Claude-Code-side handling.

Recommend Option α. It keeps the warning automatic (no Claude-side change needed) and aligns with the user's "notify, then I'll fix it manually" stance.

## 6. Open Questions

1. **Notify delivery mechanism** — option α (inline after bridge creation) vs option β (deferred to first session-info poll). **RESOLVED 2026-05-19**: α-equivalent — router-level orchestration emits `voice_persona_conflict` notify on miss. See §10.
2. **Conflict notify suppression** — if the user keeps two Claude Code sessions open in the same repo intentionally (e.g., two PIP sessions where only the first gets the preferred persona), should the second's conflict notify fire every time the second session starts? Suggest: yes, every time, until the user kills one — silence here would mask the configuration.
3. **Project-prefix overlap** — if the user later wants ONE global default (not per-repo), add a fallback chain: try `__<PROJECT>` first, then bare `COSA_VOICE_PREFERRED_PERSONA`. **Out of MVP** — not implemented.
4. **Persistence across `/clear`** — when `/clear` rebuilds the session, should the env var re-fire and potentially swap the persona? **RESOLVED 2026-05-19 — PATH A LOCKED — NO**: `/clear` preserves whatever persona is already on the bridge. Env-var preference fires ONLY on fresh allocation (no existing persona). Rationale: **persona consistency across the historical narrative is the explicit user goal** — making tracking initiatives easier through git log and session history. Persona-hopping within or across `/clear` rebuilds is precisely the failure mode this feature is designed to prevent. Once you claim a persona, you keep it until the session ends.

## 9. Rollout sequencing & commit discipline

**Per Rick's broadcast 2026-05-19**: the env-var feature ships with **narrative-attribution discipline** — the persona who commits a feature in a given repo seeds the git-log narrative as "the canonical persona for that repo." Each repo's canonical persona must be the one whose name lands on the commit, so the git log + history.md tell a single coherent story across time.

**Intended sequencing**:

1. **Env-var feature lands in working tree** (both repos) — uncommitted
   - Lupin: cosa-voice allocator change + tests
   - PIP: this plan + workflow doc updates
2. **Rick exports the env vars** in his shell rc:
   - `COSA_VOICE_PREFERRED_PERSONA__PLAN=María`
   - `COSA_VOICE_PREFERRED_PERSONA__LUPIN=Tiberius`
3. **Rick restarts the sessions** with the env vars set
   - Lupin session boots → SessionStart allocator reads env → claims Tiberius cleanly
   - PIP session boots → claims María cleanly
4. **Tiberius commits** the Lupin-side env-var allocator work → git log shows Tiberius as Lupin author from this point forward
5. **María commits** the PIP-side env-var docs → git log shows María as PIP author from this point forward

**Implication for in-flight uncommitted work**: working-tree changes survive session restarts because they're in the file system, not in session state. After restart, the canonical-persona session inherits whatever is in the tree and is the one whose name lands on the commit → narrative is preserved.

**Anti-pattern to avoid**: do NOT commit "to save progress" before the restart. That defeats the narrative-attribution purpose. Work-tree persistence is the right mechanism.

### 10.2 Architectural divergence from §4.1

The §4.1 pseudocode showed env-var orchestration extending `allocate_persona_for_session()` directly in the helper module. The actual Lupin-side implementation kept that helper untouched and put the env-var orchestration at the **router level** — `register_session.py` reads the env var, threads it to the `/allocate` router endpoint via a new `preferred_persona_name` query parameter, and the router branches into the existing `allocate_requested_persona_for_session()` helper for the lookup.

**Why this is cleaner**:
- No new control flow inside `allocate_persona_for_session()` — its existing random-allocation contract is unchanged
- The env-var path and the slash-command path (`requested_persona_name`) share the same lookup helper, just with different semantics (soft preference vs. hard request)
- Mutually-exclusive query params make the two paths obviously distinct in the API surface
- Existing 42 tests for the helper module continue to pass without change — only new router-level tests were added

**Plan §4.1 is therefore design hypothesis; §10.1 is actual delivery.** Future readers: trust §10.

### 10.3 `/clear` semantics — Path A locked

Per Rick's decision: **preferred-persona-name does NOT override an existing allocation**. The env var only fires on FRESH allocation (no persona currently on the bridge for this session). Once a session has claimed a persona — whether via the env var, via slash-command swap, or randomly — `/clear` preserves that persona regardless of env-var state.

**Stability rationale**: a mid-session persona-flip caused by `/clear` re-reading the env var would break chorus-mode disambiguation (the user expects the same voice across the same session). Path A keeps the post-allocation contract immutable.

**§9 rollout sequencing still holds**: when Rick exports the env vars and restarts the sessions, the restart counts as fresh allocation (no prior bridge), so the env-var path fires and claims the preferred personas. `/clear` after that point is a no-op for persona — and that's correct.
