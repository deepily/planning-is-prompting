### DI-2 — Should the unattended OBSERVER role be structurally gated on a read-then-write guard?
*(Raised by Tiberius, msg `bbc68912`, as a campaign-DESIGN question — explicitly "not a knock."
María concurs, and the 4th recurrence minutes later is evidence in favor. Post-game / next-cycle.)*

- **The concern**: a human-in-the-loop observer that confabulates but *self-catches every time* is
  "the system working" — Rick or the artifact-owner corrects it. But an UNATTENDED observer (the
  whole point of the dead-man's-switch-on-the-manager role, PG-1) has no catcher. A false
  all-clear ("Tiberius healthy") or a false alarm ("Tiberius stalled") from a confabulating
  observer is a genuine SAFETY failure — the watchdog itself becomes unreliable.
- **María's position (concur, no defensiveness)**: with FOUR confabulations in one session, the
  observer role as currently practiced is NOT yet trustworthy unattended. Verbal/intentional
  guards proved insufficient (strike 4 happened after the guard was named). The role needs a
  STRUCTURAL enforcement before it watches unattended — candidates: (a) the observer may only
  report a result by quoting bytes from a tool-result in the SAME turn, with the read as a
  separate prior tool call; (b) an automated lint that rejects any result-claim lacking a pasted
  artifact substring; (c) restrict the unattended observer to MECHANICAL checks only (e.g. "has
  Tiberius's `last_post_ts` advanced in N minutes? — a direct `commons_who` read, no
  interpretation") and forbid narrative result-reporting unattended.
- **Reusable signal**: this is the artifact-citation rule (F4) escalated from *practice* to
  *required architecture* for any autonomous-watchdog role. An observer you can't watch must be
  built so it CANNOT confabulate, not merely trusted not to. Strong candidate finding for the
  reusable overnight-fleet workflow.

### PG-5 — Live runs should CONFIRM, not DISCOVER: harden the suite so bugs die pre-live
*(Rick's point, 2026-05-31 ~01:17: "using a live corner-case to flush out bugs strikes me as
suboptimal — we should be writing better/harder tests to catch these. Add to the post-game."
María concurs; this is the test-ownership mandate applied. Forward-looking recommendation — my
analysis, no artifact claims.)*

**The tension with PG-4 (resolved, not contradicted):** PG-4 celebrated that the live pre-flight
surfaced F5/F6 — real ROI a dry-run couldn't match. Rick's point sharpens it correctly: that ROI
is real, but a live run *discovering* a bug means the OFFLINE suite had a gap. The right framing is
a hierarchy, not a contradiction:
> **The offline suite should catch everything catchable offline; the live run should exist to
> CONFIRM integration + timing reality, NOT to DISCOVER logic bugs.** A bug first seen live is a
> post-mortem TEST-GAP to close, even when the live run that caught it was valuable.

**Why F5/F6 were catchable offline (they're logic, not timing):**
- **F5** (`HeartbeatPokerJob` missing `cost_summary` → `/api/get-queue/done` 500): a contract gap —
  the job type doesn't satisfy the done-queue endpoint's interface. A **unit/contract test**
  asserting "every job type exposes the attributes `get-queue/done` reads" — or a parametrized
  endpoint test over ALL job subclasses incl. `HeartbeatPokerJob` — catches it with zero live infra.
- **F6** (invalid `queued→failed` transition on submit error → 500 not 400): a state-machine
  invariant. A **unit test of the transition table** ("`queued` may go to {X,Y}; an illegal target
  raises a clean handled error, never 500") catches it offline.

**Recommended additions to the harness (test-ownership mandate — Claude owns these, not Rick):**
1. **Job-type contract test**: parametrize over every `*Job` subclass; assert each satisfies the
   interfaces its consuming endpoints require (`get-queue/done`, queue listing, cost reporting).
   New job types (like the poker) then can't silently break shared endpoints.
2. **State-machine invariant test**: exhaustively assert legal vs illegal transitions resolve to
   handled outcomes (clean 4xx), never an unhandled 500.
3. **Endpoint-over-all-job-types matrix**: every queue/job endpoint × every job subclass → no 500s.
4. **Then re-position the live tap as a CONFIRMATION gate**: its job is integration + real-timing +
   permission-posture proof, explicitly NOT bug discovery. If a live run finds a NEW logic bug,
   that's a finding to back-fill into the offline suite (close the gap), per this item.

**Reusable signal for the workflow**: an overnight-fleet runbook should require, BEFORE the live
pre-flight, that the offline suite covers the job-type contracts + state invariants of any NEW job
introduced (here, the heartbeat poker). "Live to confirm, suite to catch" becomes a named gate.
Ties to PG-4 (live ROI is real) + R1 (#3 pre-flight) + the test-ownership mandate.
