## Q2 — External heartbeat scheduler cadence + scope

**Cadence**: 2-3 min during active cascade; 5+ min idle. Why this shape:
- Reviewer compute time is 1.5-2.5 min per stage (per §1)
- Detection within 2-3 min means manager wakes BEFORE the next stage's worker fires
- 5+ min idle reduces poke-volume when nothing's pending

**Scope**: manager-only.
- Workers are pull-driven by manager DMs; they don't need pokes
- Pokes-to-workers create noise without value
- Manager-poke + manager's disk-read-discipline covers the whole cascade

**Implementation shape** (suggested):
- Process: cron job, scheduled GitHub Action, systemd timer, or a small daemon
- Action: post a no-op DM to manager via `commons_send_to(recipient="tiberius", body="heartbeat", expect_reply=False)`
- Idempotency: pokes are safe to ignore when nothing changed; manager disk-reads and returns to idle
- State machine: scheduler tracks cascade-active vs cascade-complete (driven by manager's `kind: cascade_complete` post)
- Failure handling: if no manager response to 3 consecutive pokes, escalate to user via `notify()`

**Avoids the §6.4 heartbeat trap**: the manager doesn't need to autonomously tick because the external scheduler does it for him.

**Net recommendation**: medium-aggressive, every 5 min, always disk-read first, ping only on confirmed stall patterns.

## Q4 — Severity-tag metadata schema changes

Base schema worked cleanly. Suggested additions for richer Run-2 telemetry:

| Field | Type | Purpose |
|-------|------|---------|
| `cross_section` | bool | Orthogonal to severity — captures the cross-section dimension (Rachel's F2 + Arnold's F1/F4 were cross-section even within mixed severities) |
| `closure_action` | enum | `"ignore" \| "documented" \| "revised" \| "escalated" \| "voted"` — captures manager's action; enables % breakdowns |
| `parent_finding` | string (finding-id) | For downstream stages sharpening upstream findings (Arnold's F4 sharpens Rachel's cross-section coupling note) |
| `rounds_used` | int | Re-litigation rounds consumed (0 if no re-litigation) |
| `votes_called` | int | Votes invoked for this finding (usually 0; > 0 means contentious) |

**Schema rationale**: enables Run-2-versus-Run-1 comparisons like:
- Convention 4 marker resolution rate (revision via Open-sub-question-N pattern)
- Re-litigation depth distribution (mean / p95 rounds)
- Cross-section finding rate — worth tracking; is the cascade systematically surfacing these?

**Stamping discipline**: manager stamps authoritative classification at `manager_classification` post; reviewers stamp their `severity_proposed`. Both useful for telemetry comparing proposed-vs-final.

## Q5 — Operational lessons I observed from the manager seat

Six additions to your Lessons 1-5 list:

**Lesson 6 — Universal step zero**:
Manager's first action on every wake event is `disk-read every active topic`. Broader than your Lesson 1 — applies to ALL topics (section topics + DM topics + briefing topics), not just suspected-post-landed cases. Encode as pre-flight checklist.

**Lesson 7 — Pre-emptive worker probes have second-order value**:
When I probed Arnold + Rio while suspecting Rachel was phantom, the probes confirmed both were alive (Rio in 75s, Arnold in 115s) AND surfaced that dormancy was SELECTIVE — refined my hypothesis space. Worth keeping in v2 workflow guidance: when one worker is suspect, preemptive-probe the rest in parallel.

**Lesson 8 — Multi-finding clusters → single combined escalation**:
My Trigger 1+2 combined escalation gave Rick ONE decision-point instead of three per-finding escalations. Saved cognitive cycles + UX-friendly (`ask_multiple_choice` with 3 options vs 3 sequential yes-no questions).

**Lesson 9 — Manager-classification posts serve as worker-facing audit trail**:
The `kind: "manager_classification"` posts to section topics give workers visibility into how their findings were classified without me having to DM each one. Rule: manager posts classification to section topic at every stage-close, not just escalations.

**Lesson 10 — Workarounds-mid-run become workflow guidance for rest of run**:
When body-display truncation forced me into "post on topic + brief DM pointer" pattern at 16:40, I applied it for ALL subsequent communications. Rule: emergent operational patterns adopted mid-run apply universally for the run's remainder; don't treat them as one-time hacks.

**Lesson 11 (meta) — Self-audit discipline**:
I failed the disk-read discipline TWICE (Rachel false-stall, then Arnold + Rachel-B detection delay). Both times the failure was forgetting that disk-read should be step ZERO of every wake, not step N when I suspect something. Worth a hard reminder pattern (maybe a "self-audit checklist" in the manager system prompt: "Did you disk-read all active topics this turn? If no, do it before responding").
