## 0. The Problem in One Sentence

Fleet to-do state lives on **8+ surfaces held consistent by discipline, not mechanism** (Tiberius, manager-seat inventory, 2026-06-11), and **no participant вЂ” not the arbiter, not a manager, not Rick вЂ” can deterministically query "what is owed, by whom, blocked on what."**

### 0.1 The 8+ surfaces today (audit, S108)

| # | Surface | Substrate | Queryable? | Ownership |
|---|---------|-----------|------------|-----------|
| 1 | `TODO.md` (per repo) | markdown | No вЂ” LLM/heuristic parse | none (shared file) |
| 2 | Harness TaskCreate/TaskList | in-memory, per session | per-session only; **evaporates at `/clear`**; invisible to peers | single session |
| 3 | `bug-fix-queue.md` v2.0 | markdown | semi (structured sections, no API) | вњ… Owner/By session tags вЂ” best model today |
| 4 | Commons DM threads (`dm-*` qids) | append-only topic files | file parse only | de-facto OWED-WORK ledger (unanswered qid = obligation) |
| 5 | `io/mementos/*.md` | markdown snapshots | no | drift from live board at every handoff |
| 6 | `src/rnd` implementation docs | markdown | no | phase tracking, sign-off trails |
| 7 | `history.md` | markdown | no | completion record (half the audit trail) |
| 8 | Implicit | held branches/worktrees, `:8000` run queue, "Rick's court" prose | no | none |

### 0.2 What breaks (manager-seat evidence, Tiberius 2026-06-11)

- **Dual-write drift** вЂ” TaskList в†” TODO.md mirrored by hand; forget once = lost state.
- **No single owed-work query** вЂ” answering "what is owed, by whom, blocked on what" requires TODO.md + DM-board archaeology + worktree inspection.
- **Ownership/status are prose-and-emoji**, not fields.
- **Completions aren't joined to receipts** вЂ” commit hashes and test-run ids hand-copied into DM prose (no-confabulation rule has no mechanical support).
- **Memento-vs-board divergence** at every handoff.
- **Rick can't see the qid ledger at all.**
- **The meta-failure (2026-06-11, this session)**: an owed design produced zero `work_owed` signal anywhere вЂ” the steward idled, the user had to intervene by voice. The store must make that state impossible to misread.

## 1. Requirements

### 1.1 From Rick (R-series)

- **R1 вЂ” Single source of truth**: one store visible to arbiter + managers + workers + Rick.
- **R2 вЂ” Role-based read/write**: all participants read; writes scoped by role (own/assigned items).
- **R3 вЂ” Auditable**: Rick can inspect who did what, when, on whose authority, with what evidence.
- **R4 вЂ” Deterministically queryable**: a structured query (not an LLM reading markdown) answers owed-work questions, identically for every caller.

### 1.2 From the manager seat (T-series, Tiberius 2026-06-11 вЂ” endorsed)

- **T1 вЂ” Real fields**: `owner_persona`, `accountable_manager`, `status`, `blocked_by`, `gate_class` вЂ” so "Rick's court" is a QUERY, not a prose section.
- **T2 вЂ” Append-only per-item event trail**, linking qids, commit hashes, test-run ids, journal timestamps.
- **T3 вЂ” Receipts first-class**: a `completed` transition REQUIRES a receipt ref (mechanizes `feedback_no_confabulated_results`).
- **T4 вЂ” DM integration**: a review-request qid auto-creates/updates an owed item; a threaded close is a state transition.
- **T5 вЂ” Views, not parallel truths**: `TODO.md` and the harness TaskList become RENDERED VIEWS of the store.
- **T6 вЂ” Decisions as items**: `class=decision` carrying the framing payload (pros/cons/recommendation), feeding `/plan-decide`.
- **T7 вЂ” Oracle reads the SAME store**: the heartbeat work-owed oracle consumes store queries instead of inferring from scattered files.

### 1.3 Invariants carried over from ratified prior designs

- **I1 вЂ” Local poke stays `:7999`-free**: the Stop-hook self-poke must never DEPEND on the store being up (2026-06-03 heartbeat v1 ratification). Store-read at Stop is additive; on store-down the oracle falls back to current file-based inputs. Fail-open.
- **I2 вЂ” Arbiter never destructively actuates** (2026-06-09 redline) вЂ” the store gives it better eyes, not new hands.
- **I3 вЂ” Blocked items carry `next_chase_ts`** (Rick-ratified 2026-06-10, Item-1 fork 3) вЂ” no "pending X" graves; STALL в‰  QUIET.
- **I4 вЂ” Fail-open + flag-once on non-compliance** (Rick-ratified 2026-06-10, Item-1 fork 4) вЂ” a session not writing to the store is a practice bug, not a liveness signal.
- **I5 вЂ” Workflow defaults travel with the workflow** вЂ” the PIP-side practice lands in `workflow/`; only the service build is Lupin-side (`feedback_workflow_defaults_travel_with_workflow`).

## 2. Design Overview

A **small task-store service** on the existing Lupin infra: relational store + thin REST surface + MCP tool wrappers + hook-driven write paths. Everything markdown becomes a derived view.

```mermaid
flowchart TB
    subgraph writers [Write Paths]
        TW[ TodoWrite / TaskCreate<br/>PostToolUse hook ] --> API
        MCP[ MCP tools<br/>task_create / task_transition / task_query ] --> API
        DM[ commons DM hooks<br/>review-request qid в†’ item<br/>threaded close в†’ transition ] --> API
    end

    API[ REST surface on :7999<br/>/api/tasks/* ] --> DB[( task store<br/>items + events<br/>receipts first-class )]

    subgraph readers [Read Paths]
        DB --> ORACLE[ work-owed oracle<br/>at Stop hook вЂ” fail-open ]
        DB --> ARB[ :8001 arbiter<br/>fleet owed-work view ]
        DB --> RICK[ Rick: UI / doc-viewer /<br/>gate_class=ricks-court query ]
        DB --> VIEWS[ rendered views:<br/>TODO.md sections В· TaskList В·<br/>bug-fix-queue.md later ]
    end
```

### 2.1 Data model

**`items`** (one row per obligation):

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `class` | enum | `task` \| `decision` \| `review_request` \| `bug` \| `gate` |
| `title`, `body` | text | `decision` items carry the framing payload (options, pros/cons, recommendation) in `body`/`payload` |
| `project` | text | repo scope (`lupin`, `planning-is-prompting`, вЂ¦) |
| `owner_persona` | text | who does the work |
| `accountable_manager` | text | who answers for it (lineage-derived default) |
| `created_by` | text | persona + session id |
| `status` | enum | `queued` \| `claimed` \| `in_progress` \| `blocked` \| `review` \| `done` \| `dropped` |
| `created_ts`, `updated_ts` | timestamp | (Tiberius review F-extra вЂ” was implied, now explicit) |
| `blocked_by` | typed refs | list of `{kind: item\|persona\|user, id}` вЂ” TYPED, not a mixed string field (Tiberius review finding 3: untyped refs leak R4's determinism at exactly the field the oracle queries). `{kind:user}` в‡’ oracle treats as NOT-owed |
| `next_chase_ts` | timestamp | REQUIRED when `blocked` (I3) |
| `gate_class` | enum | `none` \| `manager` \| `ricks_court` вЂ” Rick's court becomes `SELECT вЂ¦ WHERE gate_class='ricks_court'` |
| `priority` | enum | P0вЂ“P3 |
| `source_qid` | text | DM question id that spawned it, if any (T4) |

**`events`** (append-only, per item вЂ” the audit trail, R3/T2):

| Field | Notes |
|---|---|
| `ts`, `item_id`, `actor` | persona + session id |
| `transition` | e.g. `queuedв†’claimed`, `in_progressв†’done` |
| `receipt_refs` | JSON: `{commit, test_run, qid, doc_path, log_line}` вЂ” **non-empty REQUIRED for `в†’done`** (T3); the API rejects a bare completion |
| `authority` | `standing` \| `user_direct` \| `manager_relay` вЂ” joins the blast-radius model to the trail |

### 2.2 API surface (sketch)

- `POST /api/tasks` вЂ” create item
- `POST /api/tasks/{id}/transition` вЂ” state change; `в†’done` validates `receipt_refs`
- `GET /api/tasks?owner=вЂ¦&status=вЂ¦&gate_class=вЂ¦&accountable_manager=вЂ¦&project=вЂ¦` вЂ” the deterministic query (R4)
- `GET /api/tasks/{id}/events` вЂ” the audit trail (R3)
- MCP wrappers (`task_create`, `task_transition`, `task_query`) so any session uses it as naturally as `notify()`; arbiter + hooks use REST directly.

**Role model (R2), enforcement-light to start**: every write stamps `actor` + `authority`; the API enforces only structural rules (receipts on done, chase-ts on blocked). Social/role rules (who may claim what) stay practice-layer in v1 вЂ” consistent with the commons trust model вЂ” with the trail making violations visible rather than impossible. Hard ACLs are a later phase if practice fails.

### 2.3 Integration: existing mechanisms become clients (not stopgaps)

| In-flight build | Today's plan | Under this design |
|---|---|---|
| Manager-TODO mirror hook (`PostToolUse(TodoWrite)` в†’ `~/.claude/manager-todos/<sid>.md`) вЂ” Tiberius | private per-session file | **same hook, write path goes to the store** (file mirror optionally kept as offline fallback per I1) |
| Acked-inbound DM ledger (`.heartbeat-acked-<sid>.json`) вЂ” Clayton | private JSON ledger | **an event class**: ack = transition on the `review_request` item the qid auto-created (T4) |
| `stop.py` oracle wiring (Rachel-gated seam) | scrape TODO files + DM topics + manifests | **one store query** (`owner=me, status in (in_progress, queued), blocked_onв‰ user`), file-based fallback on store-down (I1) |

Sequencing note: **none of the three needs to halt.** The mirror hook and acked ledger are correct shapes; this design only redirects their write/read targets. Sync with Tiberius/Clayton/Rachel before concrete pours, not a stop-work order.

Other integrations: `TODO.md` "Pending/Resume Here" sections render from the store at session-end (human narrative stays hand-authored); `/plan-decide` reads `class=decision`; bug-fix-queue.md folds in as `class=bug` in a later phase (its Owner-tag model maps 1:1).

**T5 mechanism for the harness TaskList (Tiberius review finding 4)**: the harness list cannot be externally rendered INTO вЂ” so "TaskList as view" is a **session-START practice**, not a render: at session start the session queries the store (`owner=me, statusв‰ done`) and seeds its TaskCreate list from the result; the `PostToolUse(TodoWrite)` hook closes the loop on the write side. This lands explicitly in the Phase-4 workflow docs (session-start.md integration).

## 3. Open Design Forks (for Rick вЂ” `/plan-decide` queue)

Per the decision-framing contract: recommendation first, pros/cons each. Self-ratified-and-logged items are NOT here (AFK rule) вЂ” only genuine user-level forks.

### F1 вЂ” Storage substrate
1. **SQLite on the Lupin host (Recommended)** вЂ” pros: zero new infra, single-writer fits the one-user fleet, trivially backed up (one file), Postgres-compatible-enough schema to upgrade later; cons: a ceiling if the fleet ever goes multi-host.
2. PostgreSQL now (Tiberius's instinct, largely conceded to SQLite-first on review) вЂ” pros: no later migration, richer concurrent writes, and **the "new service to operate" con is partly sunk cost вЂ” Lupin already operates Postgres** (lupin-postgres is in the arbiter's health-watch list); cons: still heavier than the v1 problem.
3. JSONL append-files (commons-style) вЂ” pros: matches existing substrate; cons: reinvents queries badly вЂ” fails R4, the core ask.

**Deployment-horizon rider (Tiberius review finding 1 вЂ” decide F1 WITH this in view)**: the **GCP cutover is Saturday (2026-06-14)**. If fleet + store ever straddle local/cloud hosts, single-file SQLite becomes the migration. SQLite-first + pg-compatible schema remains a sound v1 either way; Rick should rule F1 knowing the horizon, not in the abstract.

### F2 вЂ” Where the service lives
1. **Extend `:7999` (cosa-voice/lupin-rest) with `/api/tasks/*` (Recommended)** вЂ” pros: every session already reaches it via MCP, auth/persona plumbing exists, arbiter already proxies through it; cons: adds load to a box with known saturation history (FM-7/11/15/18). **вљ  Correction (Tiberius review finding 2, load-bearing)**: the `:7999` async-handlers/starvation fix is **FILED DEBT, UNSTAFFED вЂ” not in-flight** (v0.1 mis-stated this). Choosing F2-option-1 therefore carries a rider: **staff the async fix as a Phase-1 prerequisite**, or accept that the store inherits today's saturation risk (partially mitigated by I1's fail-open вЂ” the Stop-hook path never blocks on the store).
2. New standalone service (`:8002`) вЂ” pros: isolation from `:7999` load; cons: third daemon to deploy/systemd/monitor; another deploy gate.
3. Inside `:8001` arbiter вЂ” pros: co-locates with the main reader; cons: violates separation (arbiter = observer, store = shared infrastructure; I2 hygiene).

### F3 вЂ” Canonical-truth boundary vs TODO.md
1. **Store = canonical for LIVE work; TODO.md = durable human narrative, partially rendered (Recommended)** вЂ” pros: honors the ratified 2026-06-10 "TODO.md is never oracle input" ruling, keeps git history as the long-term record, lowest migration risk; cons: two artifacts persist (but now with a defined master).
2. Store replaces TODO.md entirely вЂ” pros: one truth, period; cons: loses the git-versioned, human-readable record Rick greps today; big-bang migration risk.

### F4 вЂ” Write-discipline scope at launch
1. **Managers-first writes, everyone reads (Recommended)** вЂ” pros: mirrors the ratified managers-first oracle scope, prove-then-widen, smallest blast radius; cons: worker items arrive only via manager/DM hooks initially.
2. All sessions write from day one вЂ” pros: complete picture immediately; cons: unproven practice Г— whole fleet; noisy first weeks.

## 5. Risks & Anti-Patterns Guarded

- **Parallel-truth regression** вЂ” every markdown surface must be demoted to a view or explicitly labeled narrative; never "also update the store."
- **`:7999` saturation** (F2 fork) вЂ” store lands with/after the async-handlers fix; the Stop-hook path never blocks on it (I1).
- **Receipt theater** вЂ” receipts must be refs the reviewer can open (commit hash, test-run id, qid), not prose claims; API validates shape.
- **Discipline collapse** вЂ” the whole point of the hook-driven write path is that the practice costs a manager nothing beyond the TodoWrite they already owe.
