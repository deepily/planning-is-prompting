## 1. Motivation

Rick repeatedly says, across every repo: *"Walk me through all the pending decisions one by one, with pros, cons, and your recommendation, using the ask_multiple_choice method."* It works (9 decisions ran cleanly this session — 5 framework + 4 triage), but it is **re-improvised every time** and **re-specified by Rick every time**. The ask is to **standardize it once** so a single phrase triggers a consistent, high-quality decision walkthrough in any repo.

This is the natural complement to the **Decision-Question Framing Contract** (pros/cons + recommendation), which is already canonical — that contract governs *one* question; this skill governs *a queue* of them.

## 2. What it is

A **Guided Decision Walkthrough**: a ritual that

1. **Gathers** the repo's pending decisions (from a standard queue + an optional live harvest),
2. **Orders** them descending by priority,
3. **Presents each** via `ask_multiple_choice` under the Framing Contract (pros/cons per option + recommended-first + "(Recommended)"), one at a time, blocking,
4. **Records** the ruling, then advances to the next,
5. **Closes** with a recap and the durable decision record.

It bakes in the rules already learned: TTS-brevity (spoken = the question; pros/cons → `abstract`), no-rubber-stamp gates, `multiSelect` where options aren't exclusive, and timeout→recommended-default so an AFK Rick still gets a sane outcome.

## 3. Activation — BOTH paths (Rick's explicit requirement)

- **Natural-language trigger (primary):** implemented as an **Agent Skill** whose description is rich with trigger phrases so intent-match auto-invokes it — *"walk me through", "walk me through the pending decisions", "walk through my options one by one", "let's decide the open items", "go through my pending decisions"*. Rick verified he wants **"walk me through"** to fire it.
- **Slash command (explicit fallback):** `/plan-decide` for a deterministic invocation.

Both resolve to the same skill body. (Skills activate on description intent-match per the agentskills.io spec / the skills-management workflow — so trigger-phrase richness in the description is the load-bearing design element.)

## 4. The ritual (skill body)

```
gather → order(desc priority) → for each decision: frame → ask_multiple_choice → record → next → recap
```

- **Gather**: read the queue (§5) + optionally harvest fresh decisions from open plan-docs / TODO / "needs a decision" markers at invocation.
- **Order**: descending priority. Apply the decision-class taxonomy (framework §5) — surface genuine *user-decisions* (irreversible / prod-behavior / true-ambiguity / scope-expansion); pure mandated work is *sequencing*, not a gate.
- **Frame** (per decision, live): 2–4 options, pros AND cons each, recommended option first + "(Recommended)", recommendation rationale — all in `abstract`; the spoken question stays a one-line headline.
- **Ask**: `ask_multiple_choice`, `priority="high"`, a sane `timeout` + a `default` = the recommended option (AFK-safe). `multiSelect=true` when options aren't mutually exclusive.
- **Record**: append the ruling to the decision record (§6) and, where relevant, inline into the source doc.
- **Recap**: a brief closing summary (spoken headline + table in `abstract`).

## 5. Decision queue — source *(DD2: recommended, pending ratification)*

**Recommended: a standard `## Pending Decisions` section in the existing `TODO.md`.**
- Reuses the file Rick already maintains and reads at session-start; no new artifact; survives `/clear`.
- Holds decision **topics** (one-liners); the skill **generates** options/pros/cons/recommendation **live at walk-time** — matching how this session's 9 decisions actually worked (none were pre-framed).
- The skill may also harvest fresh decisions at invocation as a bonus.

Alternatives considered: a dedicated `decisions.md` (richer structure, but one more file to sync, over-structured if framing is live); ad-hoc-harvest-only (no persistence — deferred decisions get forgotten).

## 6. Outcome record — where rulings land *(DD3: recommended, pending ratification)*

**Recommended: a lightweight `## Decisions Log` (date + decision + ruling + one-line rationale), plus inline capture into the relevant doc when one exists** (as this session recorded D1–D5 in post-game §3.6 and the triage in §3.7).
- ADR-lite: durable, greppable, low-ceremony. Not full ADR files unless a repo wants them.

## 7. Standardization / rollout

Same hub-spoke model as every other PIP workflow:
1. **PIP owns** the canonical `workflow/decision-walkthrough.md` + the skill template + the `/plan-decide` command.
2. **Global `~/.claude/CLAUDE.md`** gets a short pointer (the trigger-phrase headline + "see PIP → workflow/decision-walkthrough.md").
3. **Wizard install** (`/plan-install-wizard`) adds it to the catalog so any repo installs the skill + command with the repo's `[PREFIX]`.
4. **Dogfood in PIP first**, then roll out.

## 9b. Generalization — multi-source / live-conversation (Rick, 2026-06-02)

The decision **source is pluggable; the TODO queue is NOT required.** Sources in priority: (1) **the live conversation** — open forks raised in the current design discussion, reconstructed from context, nothing pre-written (the source for *"walk me through the decisions we've been discussing"*); (2) the persistent `## Pending Decisions` queue (cross-session); (3) doc harvest. **New convention:** during any design conversation the assistant **proactively keeps a running "Open Decisions" tally** so the walkthrough always has a ready list — *deferred* decisions persist to the queue, *resolved* ones go to the log, and a purely in-the-moment walkthrough may persist nothing. Wired into `workflow/decision-walkthrough.md` (§Decision sources + §Running tally) and the SKILL.md description + gather step.
