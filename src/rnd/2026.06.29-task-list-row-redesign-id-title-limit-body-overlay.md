## 1. Problem

In **both** task-list clients — the **old notifications client** and the **new multiplexer client** — workers stuff entire paragraphs into the task **title** field. The row becomes an unreadable wall of text, there is no compact row identifier, and the store's `body` field (the proper home for detail) sits underused.

## 2. Framing (Rick, voice, 2026-06-29 — D5)

This is a **stopgap quality-of-life fix** to make the **old notifications client bearable while the Multiplexer MVP is built**. Keep it light, ship fast.

- The **client render** layer is explicitly **throwaway** — per-client, drift accepted, because the old client is being replaced.
- The **durable** layers (title-hygiene **language** + store-side **guard**) are **not** throwaway — they fix the data at the source and benefit the multiplexer and the store going forward.

## 3. Ruled decisions (guided walkthrough, 2026-06-29)

| **D1** | How to keep titles short | **BOTH** — workflow language (harden title-hygiene `47ba26fd`) **+** client render-truncation backstop |

## 4. Design

### 4.1 Row anatomy (left → right)

```
┌──────────┬───────────────────────────────────────────┬──────┐
│ 98dcb156 │ Harden Face B manager language: MUST fir…  │  📄  │
└──────────┴───────────────────────────────────────────┴──────┘
   ID col            truncated title (≤~60, hover=full)   detail
 (8-char hash)                                            emoji
```

1. **ID column** *(new, leftmost)* — first **8 characters of the item's `id_hash`** (e.g. `98dcb156`), monospace, compact.
2. **Title** — truncated to **~60 chars**, single line, trailing ellipsis `…`; the **full title** appears on **hover-tooltip**.
3. **Detail emoji** — a clickable affordance (proposed glyph **📄**); on click, a **small overlay/popover** renders the task **`body`**. Dismiss on click-away / Esc.

### 4.2 Layer 1 — Title-hygiene language *(PIP, María — durable)*

Harden the existing title-hygiene rule (`task-store-discipline.md`, seed task `47ba26fd`): **succinct imperative title (≤~60 chars); detail → `body`.** Add the explicit char target and the *why* (clients truncate; paragraph-titles get clipped and the ID/affordance crowd the row).

### 4.3 Layer 2 — Store-side guard in `task_create` *(Lupin — durable, D3 phase 1)*

A **non-destructive soft guard** on write: if `title` exceeds the cap (~60), **do not reject** the write (fail-open) — instead **trim the stored title to the cap** and, if `body` is empty, **move the overflow text into `body`** so nothing is lost; emit an advisory in the response.
*Sub-question for Rick/SWE (§6):* warn-only vs auto-trim+move-overflow vs reject. **Recommendation: auto-trim + move overflow → body when body empty, non-rejecting.**

### 4.4 Layer 3 — Client render *(Lupin — throwaway, per-client, D5)*

Each client **independently**:
- renders the **ID column** (8-char hash),
- **truncates** the title to ~60 + ellipsis + **hover-tooltip** showing the full title,
- renders the **detail emoji** → click → **overlay** showing `body`.

**Old notifications client lands first** (Rick's active pain point); the multiplexer mirrors the same behavior. Drift between the two is acceptable.

## 6. Handoff decisions — RULED (Rick, 2026-06-29, voice)

1. **Store-guard behavior** → **auto-trim + overflow to body.** `task_create` trims an over-long title to the cap and moves the overflow into `body` when `body` is empty; it **never rejects** the write. Non-destructive; fixes the data at the source.
2. **Detail emoji glyph** → **📄** (document).
3. **Empty-body behavior** → **dim the emoji in place.** The emoji stays in the column for layout uniformity but is **dimmed / disabled (non-clickable)** when `body` is empty — no dead click, no missing slot. *(Rick's exact ruling: "dim the emoji and leave it in place.")*
4. **Payload check** *(SWE verification, not a gate)* → confirm `id_hash` is exposed in both clients' payloads (it's in the store; verify the notifications payload carries it).
5. **Cap number** → **~60 chars**, applied identically to the store-guard max and the client truncation; tune later if real rows warrant.

> **§4.3 / §4.4 above are superseded by these final rulings where they differ** (§4.3 store-guard = auto-trim+overflow→body; §4.4 emoji = 📄, empty-body = dim-in-place).

## 7. SWE handoff — acceptance criteria

- Row renders **[8-char ID | truncated title | detail emoji]** in both clients.
- On-screen title never exceeds the cap; ellipsis present; **hover shows the full title**.
- Clicking the emoji opens an **overlay with the full `body`**; empty body handled per §6.3.
- `task_create` **soft-guards** over-long titles **non-destructively** (no rejected writes; no data loss).
- **Notifications client ships first.**
