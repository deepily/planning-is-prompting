## 1. Symptom (what Rick observed)

A `USER BROADCAST` from Rick at ~00:27Z:

> "You've got issues with your heartbeat poker. It just fired off a zillion times in rapid succession. And I hear Sam broadcasting an error TTS message. Something is broken!"

Two surface signals: (a) a rapid-fire repetition that *looked* like a runaway heartbeat poker, and (b) the **Sam** default-error persona speaking error TTS repeatedly.

## 2. Root cause

**Not a runaway poker.** There was **no live heartbeat-poker process** firing — zero `heartbeat_poker` lines in `lupin-rest-dev` since the bounce; the only poker errors anywhere in the container log were two days stale (2026-05-31). **[verified-María]**

The real cause was a **drain-first replay of a stale undelivered backlog**, triggered by the `:7999` + `:8000` server bounce Rick had just performed:

1. The bounce sent every WebSocket listener close code **`1012 (service restart)`** → all listeners reconnected. **[verified-María]** (a `1012` burst is visible in `cc-listeners.log` at bounce time).
2. On reconnect, the **just-shipped lever-A durable outbox** (live for the first time *this* bounce) drained the recipient's undelivered set.
3. The DB held **8,692 undelivered rows** (`delivered_at IS NULL`) — including 119 urgent + 28 `heartbeat_poker` "shutdown_pool" error notifications — **all months old** (2026-04-24 → 06-01, none from the incident day). **[attested-Tiberius]** (author is DB-permission-blocked).
4. Those months-old error rows were replayed through the **Sam default-error persona** (the fallback when a notification's originating context can't resolve a persona) → the "zillion + Sam error TTS" Rick heard. **[joint]** — author confirmed the Sam-persona fallback path; Tiberius supplied the backlog source.

### The landmine (the actual bug)

The lever-A durable-outbox drain had **no age cap / backlog bound**. `get_undelivered_for_recipient` / `count_undelivered_for_recipient` applied a row limit but **no `created_at` filter**, and the count was fully unbounded. So **every** reconnect/bounce would re-replay the entire historical undelivered backlog as a storm. The backlog stayed armed for the next bounce.

**Irony for the campaign narrative:** the coordination-plane **P0 fix** (durable messaging, lever A) *introduced* a coordination-plane **P0** (stale-backlog replay storm). This is exactly the class of regression the dependable-coverage framework exists to catch.

## 3. Resolution (two parts)

Owner: **Tiberius** (infra lane; Rick-authorized the DB mutation directly — satisfies the blast-radius bar for an 8,692-row bulk write on a just-shipped feature).

**Part 1 — neutralize the stale backlog (reversible).** `delivered_at` backfill (NOT delete): 8,691 pre-bounce undelivered rows set `delivered_at = now()`, state → delivered, cutoff 00:20Z. Residual undelivered = **1** (post-bounce, fresh). Audit-preserving, reversible. **[attested-Tiberius]** — author DB-blocked; recorded as attested, not witnessed.

**Part 2 — structural age cap (the real FM closure).** Commit **`8447eec`** on `wip-v0.1.8`, 7 files. **[verified-María]**
- INI key `notification undelivered max age hours = 24` + splainer entry (runtime-tunable, mirrors `notify outbox ttl seconds`). **[verified-María]**
- Query-level filter `Notification.created_at >= now() - timedelta(hours=max_age_hours)` added to **both** `get_undelivered_for_recipient` **and** `count_undelivered_for_recipient` in `notification_repository.py`. Because it is a query-level filter, it covers the **live path** — any row that goes stale *while undelivered* is skipped going forward, not just today's backlog. **[verified-María]**
- Integration test `test_age_cap_excludes_stale_undelivered` (DB-backed): inserts one fresh + one stale (48h) row, asserts the 24h-capped query returns **fresh-only** while uncapped returns both, then self-cleans. **[verified-María]** (read the committed assertions).

## 5. New failure-mode (for the framework FM catalog)

**FM-20 — Unbounded durable-outbox drain replays stale backlog as storm.**
- **Class:** coordination-plane reliability (same family as FM-7/11/15/18).
- **Trigger:** a durable/at-least-once delivery mechanism drains *all* historical undelivered items on reconnect/bounce, with no age or count bound.
- **Blast:** months-old notifications (incl. error-class) replay en masse; if any replay through a default-error persona, it manifests as a TTS storm.
- **Structural guard:** a **query-level** age cap on both the getter and the surfaced count (so the "N missed" count cannot phantom-count rows the getter would skip). Query-level (not a one-time cleanup) is required so it covers the live path.
- **Tripwire for next time:** any new durable-delivery lever must ship with an age/count bound AND a "stale-row is skipped" test *before* it goes live on a bounce.
