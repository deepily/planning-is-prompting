## 1. Problem

`manager-autonomy.md §9.1` (receipts-of-progress, codified 2026-06-22, task `ba27a555`) says a manager must prove a worker is progressing with an **artifact-delta**, not rest on an unproven "worker is working" assumption. That is a *doctrine* fix. Rick's ask (this task): can we also enforce it in **code** — in the heartbeat Stop-hook work-owed calc — so the oracle keeps a manager awake until it actually looks in on its workers?

**Today's gap (grounded in as-built code):**
- `heartbeat_work_owed.py::evaluate_work_owed(...)` already accepts an `outstanding_delegations` signal, and fires `outstanding_delegation` iff ≥1 alive un-reaped worker exists (lines 232-234, 264-266). It is **binary**: *alive child ⇒ owed*. It clears **only** when the child dies/reaps.
- `stop.py::_gather_outstanding_delegations(session_id)` is the IO half: manifest (`spawned-<id>.json`) ∩ live bridge discovery → alive children.
- So a manager with live workers DOES stay awake — but the signal says nothing about whether the manager has **verified progress** this cycle. A manager can be poked, rationalize "my worker is working," and sit. The oracle can't tell "actively verifying" from "passively waiting."

**The loophole**: liveness ≠ progress, and *being owed* ≠ *doing the owed thing*. The current signal proves the manager is *needed*; it does not compel the manager to *act* (verify).

---

## 2. Design seed (Rick's debounce proviso)

From the task body + memento: a `last_looked_in_on_workers_ts` debounce —
- **< 10 min** since last look-in → no-op (don't nag a manager who just verified).
- **≥ 10 min** while workers are out → owed work → poke the manager to verify; reset the clock on verification.

This is the minimal, robust shape: time-based, no need to diff every worker's artifacts on the hot path. It converts "I have workers" into "I have workers I haven't looked in on recently."

---

## 3. Signal model

Three quantities, per manager:

| Quantity | Source (as-built) | Notes |
|---|---|---|
| **worker-set** | `_gather_outstanding_delegations()` (manifest ∩ live bridge) | already computed |
| **worker last-progress-ts** | `max(` worker's last task-store transition ts, worker's last commit ts on its branch `)` | task-store transition ts already tailed by the arbiter (`kind="task_transition"`); git ts is a cheap `git log -1`. **Capture-pane is NOT available** in the arbiter and is too costly on the Stop hot-path — exclude it |
| **manager last-verification-ts** (`last_looked_in_on_workers_ts`) | the manager's own most-recent *verification receipt* | see §4 |

**Owed-for-verification** fires iff workers are out AND:
- (debounce) `now − last_verification_ts ≥ threshold` (Rick: 10 min), OR
- (escalation, optional v2) any worker `last_progress_ts` is **stale** beyond a stall threshold (worker itself may be stuck and the manager hasn't acted).

v1 = the debounce alone (simplest, matches Rick). v2 layers the stall-aware escalation once v1 proves out.

## 4. What counts as a "verification receipt" (clears the clock)

Reuse existing manager artifacts — do **not** invent a mandatory new verb first:
- a manager **DM to/from a worker** (already in the commons/dm log),
- a manager **task-store transition on a worker's task** (e.g. routing it to review, or commenting),
- a manager **commons post** referencing the worker.

Any of these stamps `last_looked_in_on_workers_ts = now`. This makes the loop self-clearing through normal manager behavior: a manager who is actually managing resets the clock for free; a manager who sits does not. (Optional explicit `looked_in` stamp/verb as a fallback if the implicit signals prove too noisy — defer.)

## 5. Where it lives — two layers (recommend BOTH, Stop-hook primary)

1. **Stop-hook owed-calc (self-regulation, PRIMARY)** — enrich `_gather_outstanding_delegations` so each entry carries `needs_verification: bool` from the debounce, and have `evaluate_work_owed` keep `outstanding_delegation` firing while `needs_verification` is true (instead of clearing only on death). This keeps the manager's **own** session awake to go verify. It's the manager's keep-alive — the right primary home.
2. **Arbiter (external backstop)** — in `arbiter_job.py::_classify_owed`, a manager with workers-out + stale verification classifies **ACTIVE** (pokeable), not DONE. `resolve_active_managers` + `manager_resolver` already give the arbiter the manager→worker lineage; the task-store transition tail already gives freshness. This is the external poke if self-regulation fails.

**Recommendation**: implement layer 1 first (self-poke is the direct §9.1 enforcement), layer 2 as the backstop.

## 9. The outward twin — user-gate re-ask canonization (Rick "Both", 2026-06-22)

**Problem (mirror of §1)**: today an `awaiting: user:rick` hold is treated as *permission to go quiet* — the session declares it needs Rick and then waits passively for him to come back. But Rick can't see a buried/forgotten gate; a pending direct ask that isn't re-surfaced is the exact failure `feedback_never_bury_the_decision_ask_directly` / `feedback_gate_is_a_direct_ask` exist to kill. Same loophole as the manager twin — *owing while silent* — pointed at the user instead of at workers.

**Inversion of hold semantics for user-gates**: a normal hold = "I'm legitimately blocked, don't poke me." A **user-gated** hold is the opposite — it is a **standing obligation to re-fire the direct `ask_*` every tick** until the gate clears. The hold suppresses *peer/arbiter* nagging but must NOT suppress the *re-ask to Rick*.

### 9.1 Doctrine (the rule) — Rick ruled BOTH
Canonize in `workflow/cosa-voice-integration.md` (extend the existing "Gate = a Direct Targeted Ask" section) + a pointer in `workflow/manager-autonomy.md`:
> A pending direct user-gate (a decision/question only Rick can answer) is **owed work that must be re-surfaced, not parked**. While it is open, re-fire the dedicated `ask_*` on every self-tick (default every 10 min, Rick `cd610b8a`) until Rick answers. Going silent on an open user-gate is the same violation as a manager sitting on unverified workers. The hold's `awaiting: user:rick` is the *trigger* to re-ask, never a license to stop asking.

### 9.2 Mechanism (the structural half)
- **State**: the session's hold/state file gains a structured `pending_user_gates: [ { id, question, ask_kind, ask_payload_ref, first_asked_ts, last_asked_ts, reask_interval_s, answered: bool } ]`. (Today `awaiting` is a free-text string — promote the user-gate part to structured rows.)
- **Re-ask driver (primary = the session's own tick)**: on each `/loop`/Stop-hook self-tick, for every open gate where `now − last_asked_ts ≥ reask_interval_s`, RE-FIRE the `ask_*` and stamp `last_asked_ts`. An answer (any non-timeout result) sets `answered:true` and clears it.
- **Owed-oracle unification**: `evaluate_work_owed` treats `any(open pending_user_gate)` as `work_owed = true` (a new owed reason, e.g. `outstanding_user_gate`), exactly parallel to `outstanding_delegation`. This keeps the session alive to re-ask and stops the idle-beacon from announcing "idle" while a gate is open (now safe to rely on, since `ff4caf36` makes the beacon consult the oracle).
- **Arbiter backstop (mirror of §5 layer 2)**: a session with an open, aged user-gate classifies ACTIVE; if the session itself has gone dark (not re-asking), the arbiter surfaces the gate to Rick on the session's behalf (escalation), so a *dead* session's owed gate still reaches him.
- **Escalation cadence (v2)**: hold a flat 10-min re-ask for v1; optionally tighten interval or raise priority as a gate ages N cycles unanswered.

### 9.3 Symmetry (why this belongs in ONE design)
| | Inward twin (§1–§8) | Outward twin (§9) |
|---|---|---|
| Who is owed | the manager | Rick |
| Owed action | verify a worker (artifact-delta) | re-ask a pending gate |
| Clears when | fresh verification receipt | Rick answers |
| Owed reason (oracle) | `outstanding_delegation` (+ `needs_verification`) | `outstanding_user_gate` |
| Failure it kills | manager sits on unverified workers | session goes silent on a needed decision |

Both are the same predicate — *owed ⇒ act, don't sit* — wired into the same `evaluate_work_owed` oracle. That is why this is `6929f4ac`'s natural completion, not a new board item.
