## Note 1 — Doctrine + mechanism for surfacing worker→Rick questions (the 10-min timer was a HACK)

**Rick's framing (verbatim intent, 2026-06-22 voice):** *"How do we get the fact that you and other workers — Mr Radio, Tiberius — have questions that require my attention [to him]? The 10-minute timer for pinging me was just a hack to force you to surface your questions. We really do need doctrine and mechanism for surfacing questions that you have for me that require an answer."*

**The problem.** When a worker/manager-figure (María, Mr Radio, Tiberius) reaches a decision only Rick can make, there is no *reliable, unified* path for that question to reach him — especially when he is AFK or the asking session is parked/idle. Today it depends on the asking session being active and choosing to ask, plus a crude timer to force re-surfacing.

**The hack being retired.** The ~10-min `ScheduleWakeup`/`/loop` re-ask loop (Rick directive `cd610b8a`, "re-ask any unanswered gate every ~10 min") is a *forcing function*, not a real mechanism: it makes each session independently re-poke Rick on a flat cadence. Problems: (a) noisy — N sessions each re-ask on their own timer; (b) unreliable — depends on each session staying alive + remembering to re-fire; (c) no single place for Rick to see *everything awaiting him*; (d) it conflates "keep the session alive" with "surface the question."

**What already exists (build on, don't reinvent):**
- **cosa-voice `ask_*` / `converse`** — surfaces a question to Rick *when the asking session is active and chooses to ask*. The live channel; no persistence, no cross-session aggregation.
- **`pending_user_gates` + heartbeat `outstanding_user_gate` work-owed signal** (shipped tonight, task `6929f4ac`) — an open gate counts as `work_owed`, keeps the session alive, and the Stop-hook NAMES the due gates so the session re-fires the `ask_*`.
- **Arbiter dark-session gate-resurface** (case-18, the receipts backstop deployed tonight) — if the owning session goes DARK with an aged open gate, the arbiter re-surfaces it to Rick. This is the closest thing to a real backstop already in place.
- **Task-store `gate_class = ricks_court`** — the store ALREADY has a typed "Rick's court" class. A latent substrate for a unified question queue.

**The gap the hack papered over.** There is no **single, Rick-facing "questions awaiting you" queue** across the whole fleet, and no **clean push** that reaches him when a NEW question is raised (vs. a flat re-poll). Each session surfaces independently.

**Design seed / open questions (for the design session, NOT to build now):**
1. **Canonical substrate** — make every user-blocking question a typed `gate` item (`gate_class=ricks_court`) in the unified store at the moment it's raised. One query (`task_query(gate_class=ricks_court)`) = everything awaiting Rick, fleet-wide. (Aligns with store-canonical.)
2. **Surfacing/push** — who actively notifies Rick on a NEW gate, and re-surfaces an unanswered one? Candidate: a SINGLE aggregator (the `:8001` arbiter) reads the ricks_court queue and pushes/re-surfaces — replacing N per-session timers with one queue-driven reminder. (Extends the case-18 resurface from dark-session-only to all-open-gates.)
3. **Urgency tiering** — urgent question interrupts immediately; normal batches into a digest; low sits in the queue. Avoids both flooding and starvation.
4. **Answer routing back** — Rick answers once (voice / a queue UI); the answer reaches the right asking session and clears the gate (stops the re-surface). The asking session may be parked → needs a wake/inject path.
5. **Doctrine half** — WHEN does a worker mint a `ricks_court` gate vs. ask inline vs. decide under standing authority? (Ties to blast-radius authorization + "gate = a direct ask, never buried.") The mechanism is worthless if workers don't reliably *classify* a question as Rick-blocking.
6. **Relationship to the 10-min loop** — the loop becomes a fallback keep-alive, NOT the surfacing mechanism. The queue + aggregator-push is the mechanism; the timer is demoted.
