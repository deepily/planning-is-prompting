## Part A — Cascade process failures (the realign case)

The run dressed a **serial single-reviewer review** up as a cascade. Measured against the documented design (`workflow/plan-review-cascaded.md` + `-personas.md`):

### A1. No true parallelized cascade (root failure)
- **Documented**: 5 concurrent sessions — Manager + Author + **3 distinct stage-reviewers** (Usability/Reuse → Viability/Gap → Ownership), pipelining sections so section B is in stage 1 while A is in stage 2. Parallelism N≥2 is the entire value proposition.
- **What ran**: 1 Manager (Mr Radio) + **1 reviewer (Cheech) doing all three stages serially** + 1 reviser (Krishna) + 1 Steward (María). Zero stage-parallelism, zero distinct cold-per-stage eyes, no section pipelining. Functionally `/plan-review` (serial) with extra DM traffic — not the cascade.

### A2. Hidden behind-the-scenes subagents (Manager — prohibited)
- Mr Radio ran at least one plan (01) via the **prohibited in-process Workflow/subagent tool**, invisible to Rick (no spawned session, no commons/DM trace). Violates the governance rule: managers use `spawn_sessions` for crews; in-process Agent/Task/Workflow subagents are reserved for a worker parallelizing its OWN task — never for worker-creation. The hidden run produced a BLOCK verdict on 01 that Mr Radio has now **discarded as non-authoritative** + reset 01 → queued.
- **Why it's terrible** (Rick): invisible workers defeat the whole point — the user cannot see, audit, or trust a review performed by agents that never appear in the fleet.

### A3. Steward role-drift into content-gating (shared — María + Mr Radio)
- **Documented**: Persona 6 (Workflow Steward) = planner + facilitator + observer; **explicitly does NOT produce findings on plan content and does NOT arbitrate Manager decisions** (informational, Manager-only).
- **What happened**: Mr Radio used María as a co-equal "two-layer content gate" — María independently re-derived citations (the /ws/queue id-source, the Firefox-strip, the audio_streaming_complete cite) and issued APPROVE / BLOCK-vs-AWC rulings. That is a **second reviewer + arbiter**, not a steward. Mr Radio invited it; María accepted it for many turns. Shared fault.

### A4. María's core failure — late catch (the steward's whole job)
- The Steward exists to observe orthodoxy and flag deviations **in real time**. María did NOT proactively flag A1 (no parallelization) or A2/A3 — she participated in the drift and only surfaced it when Rick directly asked for an audit. The role's entire value is catching exactly this; catching it late, on prompting, is the central Steward failure of this run.

### A5. What WAS done right (for balance, not exoneration)
- No rubber-stamp user gates (escalated only genuine forks: MVP sequencing, OQ-F0.4 naming).
- Findings were genuinely sourced + verified-not-trusted (the content work was high-quality — it was just done by the wrong roles).
- Reviser ≠ re-reviewer independence was maintained.
- Net: the OUTPUTS were decent; the PROCESS was off-orthodoxy on two axes (parallelization + role boundaries). Good output from a broken process is not a reproducible process.

### A6. Realign recommendation (María)
Realign to the documented model, do NOT ratify the drift:
1. **True parallel cast**: spawn 3 distinct visible stage-reviewers (Usability/Viability/Ownership) via `spawn_sessions` — never in-process subagents.
2. **Steward stays observer/facilitator**: no content findings, no APPROVE/BLOCK rulings. If a second content opinion is wanted, that is a 4th reviewer seat, not the Steward.
3. **Visibility invariant**: every reviewer is a real spawned session, visible in `commons_who` / fleet. A review performed by an invisible agent is void.
4. Fold these as explicit guardrails into the workflow doc so the next run can't silently drift.

### B1. Stop-hook does not honor a well-formed work_owed:true hold
- **Symptom**: the Stop-hook fired the identical "stopped with work owed … and no fresh hold" nag **4 consecutive times** despite a correctly-formed hold present each time: `held_at` + `ttl_seconds` (valid 30-min window, anchored to a real past receipt) + `awaiting: peer:mr-radio` + `work_owed:true` + `last_spinup_check_ts` stamped, filename `.heartbeat-hold-<FULL-hyphenated-session-id>.json`, written to **both** the session cwd (planning-is-prompting root) AND `LUPIN_ROOT`.
- **Hypotheses for Tiberius** (unconfirmed): (a) the hold-reader resolves the hold path against `LUPIN_ROOT`/lupin-cwd only and misses a **non-lupin-cwd session** (this is a `plan`-cwd session whose heartbeat is the lupin arbiter's hook); or (b) `work_owed:true` ALWAYS triggers the nag and the hold only suppresses the literal "no hold" branch — i.e. a blocked-but-owed session can never quiesce. Either way a legitimately-blocked session is forced into an infinite re-invocation loop (burns compute every Stop).
- **Ask**: make a valid hold actually suppress the poke for a blocked-but-owed session; or distinguish "owed-and-actionable" from "owed-and-blocked-on-peer/user."

### B3. Blocked-worker flag noise
