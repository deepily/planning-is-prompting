## 1. Problem

A worker assigned task **X** routinely discovers that X is really **X1 + X2 + X3**. Today's write scope is **managers-first** (F4 / #13 D1): workers do **not** self-create tracking items. That leaves two bad options:

- **Force a DM round-trip** — the worker DMs the manager to mint each subtask. High friction; the manager becomes a bottleneck for the worker's own decomposition.
- **Let workers mint freely** — breaks managers-first and re-opens the deadlock/ownership ambiguity #13 exists to prevent.

Rick's ask (2026-06-16): give workers the capability to **add subtasks RELATED to the parent — not new tasks outside it.**

---

## 2. The model — parent-scoped child creation (a constrained middle tier)

This is the **middle tier** between managers-first (no worker writes) and full widening (workers self-track anything):

> A worker MAY create a **child** task **iff** (a) it sets a `parent_id`, AND (b) the worker **owns the parent**. A worker may **NOT** create a top-level (parent-less) task — that stays managers-first.

Invariants (all consistent with #13 "accountability never transfers down"):

- **Children inherit the parent's `accountable_manager`.** Accountability rolls **UP** to the manager via the parent — the worker decomposes the *doing*, never assumes the *accountability*.
- **The manager's accountable item is the PARENT.** Verification rolls up: the manager verifies the parent done ⇔ all children done + verified. The manager never has to track each child individually unless he wants to.
- **Workers expand *within* an assignment, never beyond it.** A child must be a genuine decomposition of its parent (X1/X2/X3 of X), not unrelated new work — enforced structurally by the required `parent_id` + ownership check, and socially by the "related, not new" rule.

```
Manager assigns X to Worker W        (managers-first — manager mints X, accountable_manager = M)
        │
        └── W realizes X = X1 + X2 + X3
                 └── W self-creates X1, X2, X3 as CHILDREN of X
                       • parent_id = X
                       • owner = W,  accountable_manager = M (inherited)
                 └── M's accountable item stays the PARENT X;
                     M verifies X done ⇔ all of X1..X3 done+verified
```

---

## 3. Mechanism (build sketch — lupin/store side)

- **Store field**: `parent_id` (nullable) on a task item. Top-level items have `parent_id = null`.
- **Write-gate rule**: a worker (non-manager) POST is accepted **iff** `parent_id` is set AND the requester owns the parent item. A worker POST with `parent_id = null` is rejected (managers-first preserved).
- **Inheritance**: on child create, `accountable_manager` is copied from the parent (not settable by the worker).
- **Roll-up query**: `task_query(parent_id=X)` returns the children; the manager's verification of X checks all children are `verified`.
- **One-level cap** (D2): reject a child POST whose `parent_id` points at an item that *itself* has a non-null `parent_id` (no grandchildren, for now).
- **Fan-out FYI** (D1): on a successful child create, fire a `dm_send` to the parent's `accountable_manager` (directed, inline, no gate) — "worker W fanned X into X1/X2/X3."
- **Child→manager escalation** (D3): a child's `awaiting:manager` targets the inherited `accountable_manager` directly (same path as a top-level item's await; no worker hop).
- **Parent-close guard** (D4): server-side, reject a parent `→done`/`→dropped` transition while any child is non-terminal.

---

## 4. Resolved (2026-06-16 guided walkthrough — Rick, all per recommendation)

1. **Manager visibility on fan-out → FYI, no gate, via `dm_send`.** On fan-out, a lightweight **directed `dm_send`** to the parent's manager fires — informational, NOT an approval gate; the worker decomposes freely. (Rick's rider: use `dm_send`, the cheap inline directed channel ~18× lighter than a commons post — not a broadcast `notify`.)
2. **Nesting depth → ONE LEVEL only to start.** A worker fans a parent into children; a child may NOT itself be fanned out (no grandchildren). Bounds complexity (no recursive roll-up); widen depth later only if real cases demand it.
3. **Child escalation → DIRECT to the parent's inherited manager.** A child's `awaiting:manager` rolls straight to the parent's `accountable_manager` (one chase target for the whole subtree). The worker still coordinates the child's *non*-manager dependencies; routing a block-on-the-manager through the worker would recreate the #13 wait-chain.
4. **Parent-close → REQUIRES all children terminal (server-side).** A parent cannot go `done`/`dropped` until every child is terminal. Verification roll-up is sound (parent done ⇔ all children done+verified); no orphaned open children; a decomposition can't hide unfinished work.
