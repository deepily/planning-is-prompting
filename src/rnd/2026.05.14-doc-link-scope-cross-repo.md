## 1. Problem Statement

Document viewer links sent from Claude to Ricardo via `notify(... abstract=[Open: x](/app/docs?...&scope=Y))` are routed by a `scope` query parameter that maps to a registered repo on the Lupin server. Today the global Claude guidance (`~/.claude/CLAUDE.md` § `DOCUMENT VIEWER LINKS`) only documents two scope values: `docs` and `io` — both Lupin-internal built-ins.

Lupin's INI registers seven external scopes (`lupin`, `cosa-voice`, `planning-is-prompting`, `lupin-mobile`, `lookml`, `par-pacific`, `claude-plans`) that the guidance is silent about. When a Claude session is working in `planning-is-prompting` or `cosa-voice` and Ricardo asks for a doc link, Claude has no formal guidance on which scope to use — it either guesses, sends a broken link, or wrongly defaults to `scope=docs` (which routes to Lupin files, not the current repo).

The doc-link feature is high-value and used cross-session; the per-repo correction is mechanical and small. This design closes the gap on both the guidance side (so Claude *knows*) and the runtime side (so Claude can *verify*).

## 4. Detailed Design

### 4.1 Guidance update (`~/.claude/CLAUDE.md` § DOCUMENT VIEWER LINKS)

Replace the current "Scope routing" subsection with:

```markdown
**Scope routing**:

The scope query parameter identifies which registered repo the path is relative to. Built-in scopes route to Lupin only; external scopes route to other repos.

- **Built-in scopes (Lupin only)**:
  - `scope=io` — agent artifacts under Lupin's `io/` (research reports, podcast scripts, presentations, audio)
  - `scope=docs` — whitelisted Lupin project docs: root `*.md` (`CLAUDE.md`, `history.md`, `TODO.md`, `README.md`, `bug-fix-queue.md`) + prefixes `src/docs/`, `src/rnd/`, `src/workflow/`

- **External scopes** — name matches the registered repo name in Lupin's `external repos` INI key. Current registry: `lupin`, `cosa-voice`, `planning-is-prompting`, `lupin-mobile`, `lookml`, `par-pacific`, `claude-plans`.

**Choosing the right scope**:

1. If linking to a Lupin file → use `docs` or `io` (built-ins).
2. If linking to a file in a different repo → use that repo's external scope name.
3. To programmatically discover the correct scope for the current session's project: inspect the `doc_scope` field returned by `get_session_info()` (cosa-voice MCP). Shape: `{scope, base_url, allowed_prefixes, source}`. Already in the session-init payload, so no extra round trip.
4. As a fallback, consult the current project's local `.claude/CLAUDE.md` § `Doc Viewer Scope`.
```

### 4.3 Extend `get_session_info()` with a `doc_scope` dict (cosa-voice)

Rather than introducing a separate tool, extend the existing `get_session_info()` return shape with a `doc_scope` field — mirroring the `voice_persona` pattern already in place. The voice_persona precedent (a per-session, server-computed, immutable-ish property) is exactly the shape `doc_scope` wants.

**New return shape**:

```python
{
    "project": "plan",
    "session_id": "4926c582",
    "sender_id": "...",
    "server_url": "http://localhost:7999",
    "version": "0.3.0",
    "speakerphone_on": False,
    "tts_interaction_mode": "chorus",
    "claude_code": {...},
    "voice_persona": {...},

    # NEW:
    "doc_scope": {
        "scope": "planning-is-prompting",   # registered scope name; None if unknown
        "base_url": "http://localhost:7999/app/docs",
        "allowed_prefixes": ["src/", "workflow/", "docs/"],
        "source": "lupin-ini",              # "lupin-ini" | "builtin" | "unknown"
    }
}
```

**Resolution order** (server-side, computed once per session-info call, served from cache after the first hit):

1. Load Lupin INI registry at MCP startup; cache for process lifetime.
2. If session's `project` resolves to a registered external scope name → return that scope's config (`source="lupin-ini"`).
3. If session's `project` is `"lupin"` → return the `docs` built-in (`source="builtin"`).
4. Otherwise → return `{"scope": None, "source": "unknown"}`.

**Why this shape over a standalone tool**:

- Pattern-match: `voice_persona` already follows this exact pattern; `doc_scope` ships in the same envelope.
- Zero discoverability burden: Claude inspecting `get_session_info()` at session start naturally sees `doc_scope` — no need to remember a second tool exists.
- No new tool registration; no extra ToolSearch schema fetch.
- One round trip total (Claude already calls `get_session_info()` at startup per the MCP startup protocol).

**Ownership split** (clarified after Rick's 2026-05-14 call):

The dictionary that maps scope name → config already exists on the Lupin side, built at FastAPI startup in `src/cosa/rest/routers/_scope_registry.py::build_scope_registry()`. The right architecture is therefore:

- **Lupin owns the registry data AND the API contract.** Lupin exposes the registry via a new HTTP endpoint (recommended: `GET /api/docs/scopes`) or a public Python accessor. The Lupin-side work is tracked in `lupin/TODO.md` under "🔗 NEW — `doc_scope` registry exposure for cosa-voice consumption".
- **cosa-voice consumes** the exposed registry; it does NOT re-parse the Lupin INI itself. That avoids duplicating reserved-name handling, missing-path skip logic, and prefix-trim semantics — all of which already live in `_scope_registry.py`.

**Implementation in cosa-voice** (consumer half — gated on Lupin-side work):

- Query Lupin's `GET /api/docs/scopes` endpoint at MCP startup; cache the response for process lifetime.
- Inject `doc_scope` into the existing `get_session_info()` response builder using the cached registry.
- The `project` field is the primary lookup key. If cosa-voice's friendly project name doesn't match a registered scope directly (e.g., `"plan"` ↔ `"planning-is-prompting"`), maintain a small alias table in cosa-voice config: `project_alias_to_scope`.
- v1 does NOT support mid-process registry refresh; new external repos in Lupin INI require an MCP restart to pick up.

**Graceful degradation** (during the rollout gap):

Until both Lupin and cosa-voice work lands, `get_session_info()` returns `doc_scope: {"scope": None, "source": "unknown"}`. Claude falls back to:
1. Local repo's `.claude/CLAUDE.md` § `Doc Viewer Scope` (Phase 3 pins).
2. Global guidance in `~/.claude/CLAUDE.md` § `DOCUMENT VIEWER LINKS` (Phase 2 guidance).
3. Asking the user.

This ordering is documented in the §4.1 guidance block.

**Edge cases**:

- INI not readable (`LUPIN_ROOT` unset or file missing) → empty registry; all sessions get `doc_scope.source="unknown"`. Logged warning at startup.
- Project unknown (session outside any registered repo) → `{"scope": None, "source": "unknown"}`. Claude falls back to local CLAUDE.md scope pin or asks the user.
- Project ambiguous (should not occur given scope-name uniqueness) → first match wins; warning logged.

**Tests** (cosa-voice):

- Unit: registry build from a synthetic INI; alias-to-scope mapping; unknown-project fallback; `doc_scope` field present and correctly shaped in `get_session_info()` output.
- Integration: spin up MCP, call `get_session_info()` from a known-project session, assert `doc_scope.scope` matches expectation.
