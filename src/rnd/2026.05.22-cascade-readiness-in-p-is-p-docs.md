María leads and drives. Tiberius authors. Mr Radio (Run 5 Manager, back online) is the cascade-side authority.

## 2. The problem — the inputs mismatch

The cascaded-review pipeline has a **Step 0 "Cascade Preparation"** phase that exists *precisely because* a raw planning/design document is not born cascade-ready — it has to be reshaped (decomposed into independent sections, Recon checklist assembled, etc.) before Step 1 can fire. PG-5's Cause 1 is one face of this: the cascade's pipelining model assumes an input shape that a generic plan does not supply.

The Planning-is-Prompting workflow produces plans through its own lens — classify → pattern → work-breakdown → TodoWrite — with **no awareness of the cascade's input expectations**. So a p-is-p-produced plan handed to a cascade always incurs a separate Step-0 reshaping pass, and a plan that *can't* be cleanly reshaped (single-section, non-independent) fails the cascade's `prototype_scope ≥ 2` / independence-criterion gate outright.

**The fix Rick is asking for**: push cascade-readiness *upstream* into the p-is-p planning workflow, so plans are born in a shape the cascade can consume — shrinking (ideally eliminating) the Step-0 reshaping gap.

## 3. Goal & deliverable

A **"Cascade-Readiness"** guidance surface added to the p-is-p workflow docs: how to structure a plan, while planning it, so its outputs match what the cascaded-review process expects as input. Final placement (which of the three docs, and how much in each) is decided in Phase 1 discovery — §5 is the working proposal.

## 4. The cascade input contract — RATIFIED by Mr Radio (2026-05-22)

Mr Radio (Run 5 Manager / cascade-side authority, session `679e8f04`) ratified §4: "substantially correct; all config-key citations verified; four corrections, none structural." His crucial reframe: **§4 describes the contract for Step 0's OUTPUT, not the raw plan's input.** The cascade's Step 0 ("Cascade Preparation") exists precisely to *transform* a raw plan into this shape — the p-is-p guidance can *shrink* Step 0 but cannot *delete* it. So the contract splits by **who can satisfy each property**:

### 4a. Planner-pre-satisfiable — PUSH these into p-is-p (the cascade-readiness guidance)

All are work-breakdown properties a planner can produce directly:

- **≥ 2 sections** — CONFIRMED hard gate (the only true gate; `prototype_scope < 2` refused unless user-overridden — `plan-review-cascaded.md:11`/`:102`, `-common.md:113`).
- **Section independence** — CONFIRMED hard gate, load-bearing (Run 5 Step-0 light-review criterion 1; `section_sizing_heuristic = independence_criterion`, `-defaults.md:152`). **THE property the p-is-p guidance must drill**: a section reviewable without loading sibling context.
- **Explicit, minimal, ACYCLIC cross-section dependencies** — CONFIRMED + strengthened: Mr Radio added **acyclic** (Run 5 Step-0 light-review criterion 5 = "valid DAG topological order, no cycles" — a cyclic dependency set has no valid section ordering → no pipeline). Document provider AND consumer directions.
- **Comparable section scope** — CONFIRMED but RECLASSIFIED → **soft preference, not a gate**. The cascade does not refuse uneven sections; balanced scope only improves *pipelined* throughput (slowest section = bottleneck). Keep as a throughput nice-to-have.

### 4b. Intrinsically Step-0 — do NOT push upstream (stays cascade-side)

A planner produces a *sliceable* breakdown; converting it into these is Manager Step-0 work:

- Pre-cascade **Recon-checklist assembly** — the checklist is *assembled by* Step 0; the plan need only surface its project context legibly. Run-5 lesson: mandates cited in Recon must have **scope-applicability verified** (Run 5 cited the Lupin-wide 100%-coverage mandate without checking that `HeartbeatPokerJob` lives in CoSA, mandate-excluded).
- The Step-0 **light-review gate** and effective-config resolution.
- The **formal slicing manifest** + **Q-decision matrix** — Mr Radio's correction: these are produced in **both** review and authoring modes (Run 5, a pure-review cascade, produced a 5-section slicing manifest AND a Q-decision matrix — earlier draft wrongly tagged them authoring-only). Only **per-slice design docs** are genuinely authoring-only. Producing the *formal* manifest is Step-0 Manager work — the planner supplies the sliceable breakdown it is built from.

**Bottom line for §5**: the p-is-p guidance teaches a planner to produce §4a — ≥2 independent, acyclically-dependent, comparably-scoped sections. It does not (and cannot) absorb §4b. Mr Radio's verdict: "your §5 'primary home = p-is-p-01 work-breakdown step' is exactly right."

## Appendix A — Phase 2 Drafting Brief (for the author)

**Status**: Phase 2 design — drafted by María 2026-05-22, after Mr Radio's §4 ratification. This is the brief the author works from in Phase 3.

**Design constraint** (the one hard line, from §4): the p-is-p guidance teaches a planner to produce ONLY the §4a planner-pre-satisfiable properties — ≥2 sections, section independence, explicit acyclic cross-section dependencies, comparable scope. It does NOT absorb §4b (Recon-checklist assembly, the formal slicing manifest, Step-0 gates). The guidance *shrinks* cascade Step 0; it does not replace it.

### A.1 — `p-is-p-01-planning-the-work.md` (PRIMARY)

**Insert**: a new subsection — `### Cascade-Readiness (When the Plan Will Be Cascade-Reviewed)` — inside **Phase 3: Work Breakdown**, placed after "Task Breakdown Principles" and before "Manual Breakdown Methodology".

**Content outline**:
- *When this applies* — the plan is destined for `/plan-review-cascaded` (≥2-section plans where user attention is the binding constraint). If the plan gets only serial `/plan-review`, or no review, skip this.
- *Why* — 1–2 sentences: a cascade consumes a plan already decomposed into independently-reviewable sections; if the plan is not section-shaped, the cascade's Step 0 must reshape it (or it fails the ≥2-section gate). Structuring it cascade-ready at planning time shrinks that gap.
- *The four properties to build in* (the §4a list):
  1. **≥ 2 sections** — decompose the breakdown into ≥2 review sections. Hard gate: a 1-section plan cannot cascade.
  2. **Section independence** — each section reviewable without loading sibling context. The load-bearing property — give a concrete test/heuristic the planner can self-apply.
  3. **Explicit, minimal, acyclic cross-section dependencies** — document provider + consumer direction; the dependency graph must be a valid DAG (a cycle → no section ordering → no pipeline).
  4. **Comparable section scope** — soft preference (pipeline throughput; slowest section = bottleneck). Not a gate.
- *Boundary note* — the planner produces a *sliceable* breakdown; the cascade's Step 0 still assembles the Recon checklist and the formal slicing manifest. Do not attempt those here.
- *Cross-references* — `/plan-review-cascaded`; cascade Step 0.

**Also**: add one cross-reference line in "Integration with Other Workflows" pointing at `/plan-review-cascaded`.

### A.2 — `p-is-p-02-documenting-the-implementation.md` (REAL)

**Insert**: a new subsection — `## Structuring for Cascaded Review` — immediately after the "Doc Conventions for Plan-Review Compatibility" section (after its "Cross-Reference" sub-block, before "Creating Your Documentation Structure").

**Content outline**:
