## 1. Why this doc exists

The user piloted Lupin's **broadcast feature** (user→all Claude Code sessions) and confirmed end-to-end success: every active session acknowledged receipt and replied individually. That validated the wire.

But the global `~/.claude/CLAUDE.md` documented `notify`, `ask_*`, `converse`, `get_session_info`, `set_session_topic` — and **nothing** about either cross-session surface (broadcast OR commons). Every Claude session had working cross-session wiring with no behavioral guidance. This doc records the design conversation that closed that gap.

## 3. Decisions reached in the design conversation

### Q1 — Autonomy thresholds: three-tier impact-based cut ✅

| Tier | Operations | Default policy |
|---|---|---|
| **Read** | `commons_who`, `commons_read` | ✅ Always allowed — like tailing a log |
| **Self-disclosure write** | `commons_post` to `presence`, `incidents`, own status | ✅ Allowed — doesn't demand peer attention |
| **Attention-demanding** | `commons_ask_sync`, `commons_ask_async`, claim-staking on contested work | ⚠️ Requires explicit user trigger OR clear coordination need (file collision, bug claim) |

**Rationale:** Read-and-disclose is cheap and useful — sessions should be free to discover peers and announce themselves. But anything that *demands* another session's attention has compounding costs (peer disruption, loop risk, user opacity) and must be either user-authorized or coordination-justified.

### Q2 — Topic discipline: reserved core + organic extension ✅

**Reserved core (carries semantic weight, IS the signaling protocol):**
- `presence` — self-disclosure: "I'm alive, working on X"
- `coordination` — claim-staking: "I've got bug #42"
- `help-wanted` — open questions seeking peer input
- `incidents` — urgent errors / blockers
- `broadcasts` — reserved for user→all broadcast (already in use)
- `broadcast-acks` — reserved for ack aggregation (already in use)

**Organic topics:** sessions may invent topic names freely. Organic topics inherit no special tier — informational only.

**Rationale:** The topic name *itself* becomes the signal. Peer sessions reading the blackboard can rely on topic to know what kind of post they're seeing, without needing to inspect the body. This couples Q1 with Q2 elegantly.

### Signaling: four-layer architecture ✅

| Layer | Mechanism | Audience |
|---|---|---|
| **Static guidance** | Global `~/.claude/CLAUDE.md` + `workflow/cross-session-communication.md` | Every session reads at startup |
| **At decision point** | Tier markers embedded in MCP tool descriptions (Lupin follow-up) | Model at tool-call time |
| **Peer-facing runtime** | Reserved topic names ARE the tier marker | Peer sessions reading the blackboard |
| **User-facing runtime** | `notify()` to user on attention-demanding entry | User in notifications UI |

### Q3 — Broadcast receipt contract: routing + voice axes ✅

The original "interrupt vs queue" framing was misdirected. Broadcasts inject as `<system-reminder>` **between turns**, so the model is always at a natural inference boundary on receipt. The real axes turned out to be routing and voice.

**Routing:**
- `@MyPersona:` matched → **ACT** on persona-specific directive (+ default body if present)
- Different `@persona` named, no default body → **ACK-ONLY** (not for me)
- No persona at all → **ACT** on default body (all sessions respond)

**Voice:**
- Speakerphone ON → spoken ack via `notify(suppress_ding=True, priority='high')`
- Speakerphone OFF → text-only ack
- Mandatory `broadcast-acks` post is infrastructure, always happens

## 8. Glossary

- **Broadcast** — user-initiated message fanning out to all active Claude Code sessions
- **Commons** — file-based shared blackboard at `/io/commons/topic-*.md` for Claude-to-Claude messages
- **Persona-directive** — `@PersonaName:` prefix routing a broadcast line to a specific session's persona
- **Effective directive** — what a session actually executes after persona-parsing (default body + matched `@PersonaName:` lines)
- **Ack** — per-recipient acknowledgment posted to `broadcast-acks` topic, aggregated by UI watcher
- **Tier** — autonomy classification of an operation: Read / Self-disclosure / Attention-demanding
