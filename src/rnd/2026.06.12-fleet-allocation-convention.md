## 1. Shape: events + reconciliation (not raw counts)

A shared **`fleet-allocation`** commons topic. Each post is an **event**, not a count assertion — raw counters drift the first time a post is missed, and a missed reap-post otherwise becomes a permanent ghost slot.

**Event schema** (designed store-migratable — see §3):

```json
{
  "action"             : "spawn | reap | adopt | handoff",
  "session_id"         : "<8-hex of the affected worker>",
  "persona"            : "<allocated persona, VERIFIED post-spawn>",
  "manager"            : "<posting manager persona + session>",
  "repo"               : "<repo/lane>",
  "worker_count_after" : 3,
  "fleet_estimate"     : 7
}
```

- `worker_count_after` = the posting manager's own roster after the action (self-report).
- `fleet_estimate` = optional; the manager's read of the fleet total at post time (cheap visibility, never authoritative).
- Persona field carries the **verified** allocation (post-spawn `list_spawned_sessions` check — the spawn-ack rule), never the requested preference.

**Reconciliation** (coordination point = María): periodically — and ALWAYS before any cap-boundary call — reconcile the event-derived total against ground truth (`list_spawned_sessions` + `commons_who(retention_hours=1)` + bridge liveness). Events give visibility; reconciliation gives trust. Discrepancy = a flag-once note on `fleet-allocation`, not a poke storm.

## 2. Cap semantics — ✅ RULED (Rick, 2026-06-12, relayed by Tiberius qid `b5ce8683`)

**ALL sessions count inside the 8** — managers/stewards AND workers (≈5 worker slots with 3 standing seats up). Rick's emphasis, verbatim spirit per the relay: *the tight budget is the point — it forces cross-repo manager collaboration and vigilant spin-down of unused workers.* Matched the default proposal (the Max-plan freezes hit *total* concurrent sessions — the quota does not care about role).

Related existing bound: the per-manager soft cap of 8 + pool-exhaustion alarm (manager-autonomy D3, 2026-06-10) stays; the fleet-wide 8 is now the BINDING boundary in practice (a single manager can no longer reach their per-manager 8 without blowing the fleet cap).

## 3. Convergence guard: bridge to the task-store, not a rival ledger

Machine-readable fleet state is task-store territory by Phase 2. The event schema above is deliberately shaped ≈ a future `task_events` row (`actor`/`transition`/receipt-able). When the store's fleet view lands, the topic convention RETIRES into store writes with the same fields — no parallel-truth regression (§5 of the task-store design).

## 4. Reap vigilance (restating existing doctrine, new visibility hook)

Reap threshold stays as ratified (idle + no-owed-work + no-declared-hold ⇒ reapable, memento/pointer seeded). The NEW obligation is only the **reap event post** — the visibility hook that keeps the fleet total honest. Prompt reaping is now also a quota-protection act, not just hygiene.
