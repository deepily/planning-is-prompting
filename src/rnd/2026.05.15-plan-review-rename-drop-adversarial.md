# Plan-Review Rename: Drop "Adversarial" from Pass-2 Nomenclature

**Status**: 🟡 PROPOSED — awaiting Mr. Rick's approval
**Author**: Rachel 🕊️ (cosa-voice session `02f528ee`)
**Date**: 2026-05-15
**Trigger**: Mr. Rick (voice, this session) reported a session-level mis-characterization of Pass 2 as a security/threat-model review (path traversal, manifest tampering, Unicode bypass). Investigation confirmed the canonical doctrine is **correct** — Pass 2 has always been about ownership-language and executor-tagging — but the word *adversarial* is ambiguous enough that sessions confabulate it into OWASP-style threat-model semantics. Mr. Rick's call: **eliminate the word entirely**, replace with something that cannot be misread.

---

## 1. Problem

The canonical Pass 2 hunts test-ownership gaps:

- Missing `EXECUTOR: AI` / `EXECUTOR: HUMAN <reason>` tags
- Bare verbs ("verify," "confirm," "observe") without a subject
- `EXECUTOR: HUMAN` lines lacking same-line justification
- "Manual E2E" / "manual test" residuals
- Steps the AI couldn't actually execute (requires GPU, UI click, privileged access)
- Sign-off punted to user without AI-produced evidence

This is **TEST OWNERSHIP MANDATE** enforcement — "the user is the designer and user of the software, NOT the tester."

It is **NOT**:
- A security review
- A threat-model exercise
- An OWASP attack-surface audit
- A vulnerability hunt

But the word *adversarial* drags sessions toward the wrong interpretation. Today's drift evidence: a session described Pass 2 verbatim as "*what attacks does this enable, path traversal, manifest tampering, Unicode bypass*" — every one of those is an OWASP category, none of them is an executor-tagging concern. The drift is reproducible: any session reviewing a milestone that touches file paths, input validation, or serialization will pattern-match "adversarial" to its English-default attacker semantics.

**The fix is to rename the pass so the doctrine cannot be misread.**

---

## 2. Diagnosis: How the word *adversarial* fails

Three reinforcing failure modes:

1. **English-default semantics**: In software contexts, "adversarial" first-resolves to attacker mindset. OWASP, red-team, threat-model — all use *adversarial* this way. The canonical doc tries to redirect via the subtitle "(Ownership Language)" and explicit prompt text, but the *word itself* fights the doctrine.
2. **"Hostile outsider" metaphor in the prompt body**: §8's prompt instructs the AI to read "as a hostile outsider whose goal is to find every place where..." The image evokes an attacker even though the sentence redirects to ownership-language. Skim-reading locks onto the metaphor and misses the actual hunt list.
3. **Milestone-context priming**: Sessions running plan-review on file-IO, input-handling, or serialization work have OWASP categories pre-activated. *Adversarial* then resolves to attacker semantics with zero friction.

The grep audit confirms the drift is **NOT** in the doctrine itself (zero hits for `path traversal`, `manifest tampering`, `Unicode bypass`, `attack`, `threat model`, `security review`, `exploit`, or `hostile actor` across `workflow/`, `.claude/`, `src/rnd/`, `history.md`, `CLAUDE.md`, `README.md`, `~/.claude/CLAUDE.md`, or `global/CLAUDE.md`). It lives only in spoken/in-context outputs — i.e., it's a renaming problem, not a doctrine problem.

## 3. Proposed Nomenclature

**Option set considered**:

| Replacement | Pros | Cons |
|---|---|---|
| "Aggressive" | Mr. Rick suggested; preserves intensity | Still carries combative connotation |
| "Thorough" | Mr. Rick suggested; neutral | Bland — loses the suspicious reading stance |
| "Forensic" | Implies cold methodical scrutiny; matches evidence-based hunt | Three syllables; slightly clinical |
| "Skeptical" | Captures the hostile-reader stance accurately | "Skeptical review" is uncommon phrasing |
| "Ownership-Language" (no adjective) | Names the subject directly; impossible to misread | Loses the *stance* descriptor |
| **"Ownership Audit"** (proposed) | Names the subject + carries auditor connotation (evidence-based, suspicious by job description) | Slightly formal |

**Rachel's recommendation**: drop the adjective entirely from the section title; describe the reading stance in the body text using **"forensic"** as the primary stance descriptor. This gives:

- **Section title**: `Pass 2: Ownership-Language Audit`
- **Doc subtitle**: `Plan Review (Fitness + Ownership Gate)` (currently `Plan Review (Fitness + Adversarial Gate)`)
- **Body stance descriptor**: "Read the docs as a forensic auditor whose goal is to find every place where..." (replaces "hostile outsider")
- **Cross-references**: "Pass 2 (Ownership Audit)" replaces "Pass 2 (Adversarial)" everywhere

**Alternative if Mr. Rick prefers an adjective in the title**: `Pass 2: Aggressive Ownership-Language Audit` or `Pass 2: Forensic Ownership-Language Audit`. Both preserve the intensity signal Mr. Rick wanted while killing the security-semantics drift.

1. **Lost intensity signal**: dropping "adversarial" without replacement may make Pass 2 sound less critical. **Mitigation**: keep "Audit" in the title (carries auditor weight) + add the disambiguation banner + describe stance as "forensic" in body.
