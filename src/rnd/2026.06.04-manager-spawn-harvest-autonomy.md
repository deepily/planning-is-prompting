## 1. The problem

Manager-role sessions (cascade Managers, fleet Managers like Tiberius) lack a **standing, bounded authorization** to spawn and harvest worker sessions on an as-needed basis. The cosa-voice *mechanism* already exists — `spawn_sessions()` / `dismiss_sessions()` / `list_spawned_sessions()` — but the **authorization doctrine around it is missing**, so managers default to a per-instance user gate ("may I spawn a replacement?"). That default introduces latency exactly when bounded autonomous action would have been correct.

This is the **same failure family as FM-19** (completion-discipline: *difficulty ≠ defer*) and the Heartbeat-Hook's whole reason for being — an agent with owed work idling because nothing authorized it to proceed. Here the "owed work" is *fleet management* and the missing authorization is *standing spawn/harvest permission*.

## 2. The principle (proposed)

> **Managers hold explicit, bounded, standing authority to spawn and harvest workers as-needed — autonomous within the bounds, gated only at the bounds.**

Model it on the **three-tier commons autonomy** ladder (Read / Self-disclosure / Attention-demanding) and the **blast-radius-scaled authorization** rule (authority scales with reversibility). The default flips from *"ask before any spawn/reap"* to *"act within the envelope; escalate only when an action crosses a named boundary."*

## 3. Design dimensions (to think through)

### 3.0 Ratified envelope (2026-06-04 — Tiberius's Lupin-side encoding of Rick's grant) ✅

This is the **concrete boundary as ratified**, captured by the manager (Tiberius) as a feedback memory immediately after the grant. It supersedes the more-conservative guess in §3.2 (respawn-onto-own-substrate is **standing**, not gated — see the refinement note). §3.1–§3.5 below are the design rationale; this is the operative envelope:

| Tier | Actions | Rule |
|---|---|---|
| **STANDING** (no per-instance ask) | spawn / respawn **any** persona — *including onto its own substrate*; `persona_preference` honored | Seed continuity via a memento **OR** a doc / dm-history pointer when no memento exists. |
| **STILL GATED** (Rick's per-instance word) | commit / push (unchanged); any destructive / irreversible op | Blast-radius rule — peer relay can't authorize. |
| **MANAGER HYGIENE** (required, but *not* a gate) | reap idle workers cleanly **with a memento** (Krishna "no-zombies" lesson); `notify()` Rick **after** spawn for visibility | Never block on pre-approval — visibility is post-hoc, not a gate. |

**The nuance to codify (Tiberius's catch):** the standing grant covers the **SPAWN**; ordinary **blast-radius care still applies to the EDIT** (coordinate shared files, don't ship two competing hooks). Spawn freely; edit carefully. *(This very session is the worked example — the Rachel respawn was standing-authorizable, but the `stop.py` Branch-C edit still gets the 3-way seam review.)*

**Refinement vs. §3.2:** the originating-incident hesitation (memento-less respawn onto a peer's own file) is resolved toward **"do it, just provide a continuity pointer"** — a doc / dm-history pointer suffices when no memento exists. So identity-continuous respawn is **standing with a continuity seed**, not user-gated. §3.2's "likely stays gated" guess is overridden by the ratified envelope.

### 3.1 What a manager may do WITHOUT asking (standing authority)
- **Spawn a fresh worker** for owed/queued work, up to a per-manager concurrency cap. (Reversible — a fresh worker is cheap to reap.)
- **Reap an idle / unproductive / completed worker** promptly (the harvest half — ties to the harvest-discipline MANDATE + FM-3 `dismiss_sessions` bug + FM-16 cross-lineage).
- **Re-spawn a departed worker to continue queued work** *when a seed memento exists* (continuity is preserved; see 3.3).

### 3.2 What still needs the user (the bounds — escalate here)
- **Exceeding the concurrency cap** (fleet-wide poke/throttle budget; Max-plan window).
- **Spawning into a different project / cwd** than the manager's own lane.
- **Irreversible / shared-infra actions** (e.g. a `:8000` bounce) — Rick's *direct* word, per blast-radius rule; a peer relay is not enough.
- **Re-spawning a specific persona onto their OWN in-flight shared substrate WITHOUT their memento** — the exact hesitation in the originating incident. This is higher-risk (a memento-less twin editing a peer's own file) and likely stays gated OR is conditioned on a memento precondition rather than freely standing.

### 3.3 The memento precondition (continuity vs. risk)
"Spawn fresh for new work" is low-risk and standing-authorized. "Rehydrate persona X onto their prior context" requires a **seed memento** to exist (cross-ref the `plan-memento` skill + the cascade Manager-rehydration SA-1 item). The workflow should make this split explicit: **fresh spawn = free; identity-continuous respawn = memento-gated**.

### 3.4 Guardrails (bound by construction — no fleet-storm vector)
- Per-manager **concurrency cap** (cross-ref Wave-3 prereq B fleet concurrency-cap).
- **Persona-pool exhaustion alarm** — `extra-N` personas signal the pool is exhausted (already the harvest-discipline tripwire); standing spawn authority must respect it.
- **Cost / Max-window awareness** — each child shares the rolling OAuth window; off-peak scheduling note already in the spawn tool's own guidance.
- **Visibility** — every autonomous spawn/reap a manager makes should be *announced* (commons + a user-visible `notify` at milestone) so standing authority never becomes invisible authority.

### 3.5 The harvest side is half the workflow
Authorization to **reap** is as important as authorization to **spawn**. The originating incident was a spawn-hesitation, but the dual failure — letting unproductive workers linger (pool exhaustion, `extra-N` creep) — is the harvest-discipline mandate already on the TODO. This workflow is the **authorization counterpart** to that mechanism: managers may, and *should*, reap promptly without asking.
