# Empirical Proof Design — "Owed Work Comes FROM the Store, Not the Transcript"

**Date**: 2026-06-17 · **Author**: María 🌸 · **Status**: SPEC (build-to) · **Feeds**: lupin task `e50ab966` (Tiberius lane) · **Sibling**: card E2E-hardening `0a0dc74c`
**Origin**: Fresh-eyes review Finding 1 (`src/rnd/2026.06.17-store-only-fresh-eyes-review-findings.md`); proof principles drilled with Rick 2026-06-17.

> **Purpose**: give the worker a written specification — not just a DM brief — for the one test that turns the store-only cutover's central claim from *asserted* into *proven*.

---

## 1. The claim under test

> After the cutover (`heartbeat.owed_source_from_store=True`), a session's owed-work count is read **from the unified store** (`:7999 /api/tasks?count_only=true` → SQL `COUNT(*)`) and **not** from transcript replay.

## 2. Why current evidence is insufficient (the logic gap)

Every existing test mocks the seam (`_owed_count_from_store`), fakes `urlopen`, or skips `count_only` against real Postgres; the cutover's "parity 4/4" was a **manual** drain run. More fundamentally:

> **A test where the two sources AGREE proves nothing.** If store and transcript report the same count, a passing assertion is equally consistent with "read the store" and "read the transcript." Agreement cannot discriminate between the two hypotheses. **Parity is the one condition under which the claim is unprovable.**

## 3. The governing principle

> **A claim is proven only if the test could have failed.** The test must construct a world in which a *wrong* implementation produces a *visibly wrong* answer, then show the real implementation does not.

This demands (a) a **divergence** (store ≠ transcript) so the source is observable, and (b) a **negative control** (flip the suspected cause, see the answer change) so the result is falsifiable.

---

## 4. Resolved design decisions (the three open sub-questions)

### Q1 — One fixture or two? → **ONE shared harness fixture, parametrized cases.**

The whole point is that store and transcript are **two independent inputs to the same oracle**. A single fixture must control **both axes independently**:

- a real Postgres-backed store + live `:7999` (reuse the `test_task_store_integration.py` real-DB fixture);
- a helper to seed owed rows via the **real** `POST /api/tasks` with the mirror's exact `owner_persona` stamping (no direct DB inserts — exercise the real write path);
- a **deterministic transcript fixture** (a fixed file with a known count of owed native items) the Stop hook reads.

Each case parametrizes the triple **(store_count, transcript_count, flag)**. Two separate fixtures would re-introduce the very isolation that caused the original false-pass — rejected.

### Q2 — How to make the flag-OFF negative control deterministic? → **Same divergence, flag as the SOLE variable, fixed inputs.**

Run the *identical* setup twice, changing **only** the flag:

| Run | store_count | transcript_count | flag | Oracle MUST report | Proves |
|---|---|---|---|---|---|
| ON | N (>0) | 0 | `True` | **N** | reads the store |
| OFF | N (>0) | 0 | `False` | **0** | does NOT read the store (reads transcript) |

The pair is the proof: flipping *only* the flag flips the source. Determinism is guaranteed by:
- a **fixed transcript fixture file** (known owed count, not a live session's mutating transcript);
- a **fixed store seed** (known N, seeded with explicit timestamps via the POST — no wall-clock/`Date.now` nondeterminism);
- the **flag as the only independent variable** between runs;
- pointing the store client at a **real local server** with its normal bounded timeout (no network flake); assert on the exact integer, not a range.

### Q3 — Non-lupin run: real plan-session or simulated stamp? → **Simulated stamp in CI + one documented real-session smoke.**

Post-cutover the mirror is retired, so the historical non-lupin **write**-gate bug is moot for new writes. The live non-lupin risk is now in the **read** path — whether the oracle counts rows for a non-lupin session's resolved `owner_persona` correctly.

- **CI (deterministic, required)**: seed rows via the real POST with a **non-lupin** `project` value (e.g. `plan`) under a known `owner_persona`, then invoke `_owed_count_from_store` / `query_owed` with that session's identity params and assert the count is correct. This exercises the real store + real query path with non-lupin data — the part that can break — with zero session-spawn flakiness.
- **Manual smoke (documented, not in fast CI)**: one real non-lupin (plan) session that `task_create`s owed work and runs its Stop hook — the only thing that can catch a bug in how a *real* session **derives** its `owner_persona` at call time (which the simulated stamp pre-supplies). Record as a periodic check, not a gate.

## 5. The full test matrix

| # | store | transcript | flag | Oracle MUST report | What it proves |
|---|---|---|---|---|---|
| 1 | N>0 | 0 | ON | N | store is the source |
| 2 | 0 | M>0 | ON | 0 | transcript is dead (store wins even when empty) |
| 3 | N>0 | 0 | OFF | 0 | negative control — flag governs the source |
| 4 | 150 | 0 | ON | 150 | **saturation** — true `COUNT(*)`, not a page-length capped at 100 |
| 5 | N (non-lupin project) | 0 | ON | N | non-lupin read path counts correctly |
| 6 | store **unreachable** | 0 | ON | 0 + `heartbeat_store_unreachable` log, **no spurious poke** | fail-safe holds |

Cases 1+2 are the divergence core; 3 is the negative control; 4 is the boundary the `count_only` guard exists for; 5 is the non-lupin read; 6 is the documented fail-safe (architecture §C).

## 6. Receipts (auditable, not inferred)

Each case asserts the returned integer **and** logs the actual SQL executed (or the HTTP request line incl. `count_only=true` + filter params), so the proof is citable. The test run id is the receipt that finally closes DoD `33b2b0be`.

## 7. Acceptance criteria

- [ ] One shared real-Postgres fixture; both store and transcript independently controllable.
- [ ] Cases 1–6 implemented, each asserting an exact count.
- [ ] Negative control (case 3) present and passing — the suite **can** fail.
- [ ] Saturation case (4) seeds >100 rows and asserts the true count.
- [ ] Non-lupin read (case 5) via simulated stamp; real-session smoke documented.
- [ ] Receipts logged (SQL/HTTP params) per case.
- [ ] Standard loop: worktree → 100% L/B/F on changed surface → fresh-critical review (María, reproduce-not-trust) → merge `--no-ff` HELD → push is Rick's word.
