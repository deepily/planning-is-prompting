## 1. Problem statement

A recurring failure mode in the fleet: a **manager and a worker each wait on the other**, and neither makes progress.

**Founding incident (2026-06-16, broadcast `4f266f16`)**: Mr Radio (manager) reported "awaiting Krishna" while Krishna (worker) reported "awaiting Mr Radio" — and Krishna's heartbeat poke evaluated to "nothing owed," so nothing woke either of them. The work simply stalled, invisibly, until Rick noticed.

The deadlock has two halves:

1. **Symmetric waiting.** Both parties treated themselves as the *waiter*. When everyone waits, no one drives.
2. **Silent "nothing owed."** A blocked worker that goes quiet reads to the liveness oracle as *idle-and-fine* (QUIET), not *blocked-and-stalled* (STALL). The signal that should have screamed was silent.

This design kills both halves.

---

## 2. The model — accountability never transfers down (asymmetric chase)

**Core principle (ruled direction, 2026-06-16):** **accountability never transfers down the hierarchy.** When a manager assigns work, the manager does **not** hand off the *tracking* of that work — only the *doing* of it. The manager holds an **OPEN accountable tracking item** for every assignment until **the manager personally verifies it done**.

Consequences:

- The manager's own agenda always contains "verify X (assigned to <worker>)" as an open item → the manager stays **poked to CHASE**, never lapses into passive waiting.
- The chase is **asymmetric**: the manager pursues the worker; the worker does **not** carry the burden of pursuing the manager. This breaks the symmetric-waiting half of the deadlock by construction — there is exactly one designated driver per assignment, and it is the manager.
- A worker that finds itself genuinely blocked **on the manager** does not go silent. It declares an explicit, machine-readable status: **`awaiting:manager:<name>`** — a STALL signal, not a QUIET one. This breaks the silent-"nothing-owed" half.

```
Manager assigns task T to Worker W
   │
   ├── Manager mints an OPEN accountable item:  "verify T (owner: W)"   ← stays open until MANAGER verifies
   │
   └── Worker W does T
          │
          ├── done → W reports → Manager VERIFIES → Manager closes the accountable item
          │
          └── blocked on Manager → W sets status `awaiting:manager:<name>`  ← STALL, surfaced (not silent)
```

## 3. Ratified decisions (2026-06-16 guided walkthrough)

### D1 (#13a) — Write scope → **KEEP MANAGERS-FIRST**

The manager **creates and assigns** each work item and **holds** the open accountable tracking item until the manager verifies done. Workers do **not** self-create tracking items.

- **Why**: consistent with the already-ratified unified-task-store **F4 ruling** (2026-06-12: managers-first writes, everyone reads) and with "accountability never transfers down." Smallest blast radius.
- **Widening** to worker self-tracking remains the **deferred, double-anchored prove-then-widen follow-up** (already filed in TODO Pending Decisions), revisited after the store's Phase 1–2 land + a calibration window.
- **Worker-found work**: a worker that discovers new work surfaces it to the manager (via DM), and the manager mints the tracked item. The worker never self-mints a tracked item (day-one scope).

### D2 (#13b) — Worker-await behavior → **PASSIVE + ESCALATE IF AGED**

When a worker declares `awaiting:manager:<name>`:

- **Normally passive** — the manager-tick surfaces awaiting-me items (asymmetric chase preserved). The worker does **not** poke the manager on every block.
- **Escalate if aged** — only if the await goes unanswered past a **threshold** does it fire **ONE** escalation poke to the manager.
- **Why**: passive is correct *until the manager goes dark*. A single aged-escalation poke is the **dead/stalled-manager backstop** — it closes the exact hole #13 exists to kill (a manager whose tick has died would otherwise never surface the awaiting item), at minimal noise (one poke, not a storm).

### Already-settled (#13c) — Manager-tick cadence → **ADAPTIVE SELF-PACED** (not re-asked)

The manager-tick *cadence* was already ruled by the **2026-06-15 "Always-On Manager-Tick Loop" decision** (D1 → adaptive self-paced; the loop serves the manager-figure to stay on top of own + assigned work). The accountability design **consumes** that loop as its chase engine rather than defining a new cadence.

## 4. Mechanism (how it runs)

### 4.1 The manager's agenda = the accountable items

The manager-tick loop's agenda is `task_query( accountable_manager=self )` against the unified task-store. Every assignment the manager made appears as an open accountable item carrying:

| Field | Meaning |
|---|---|
| `owner` | the worker doing the work |
| `accountable_manager` | the manager who must verify (stays accountable; never transfers) |
| `status` | `assigned` / `in_progress` / `awaiting:manager:<name>` / `done-claimed` / `verified` |
| `next_chase_ts` | when the manager should next chase this item (Tiberius's "no pending-X graves" rule) |

The manager-tick wakes (adaptive cadence), reads its accountable items, and **chases** anything past its `next_chase_ts` that isn't `verified`.

### 4.2 Worker-await as a first-class STALL status

`awaiting:manager:<name>` is a status the oracle treats as **NOT-owed-by-the-worker** but **owed-by-the-manager** — i.e. a STALL on the manager's side, never a QUIET on the worker's. It carries an `awaited_since_ts`.

### 4.3 Aged-escalation backstop (D2)

A lightweight watcher (or the worker's own next heartbeat) compares `now − awaited_since_ts` against the threshold `T_escalate`. On the first crossing, it fires **one** escalation poke to the manager's session and marks the item `escalated` (so it does not re-fire). If the manager is alive, the manager-tick would already have surfaced it; the poke matters only when the manager's tick has stalled/died.

- **RULED (2026-06-16, Rick)**: `T_escalate` = **adaptive — 2× the manager's current adaptive tick interval**. It fires only when the manager has plausibly *missed* a tick, so it never duplicates a healthy tick.
- **Config rider (Rick)**: the multiplier is **NOT hardcoded** — it is a `configuration_manager` INI key (fleet/lupin-side `lupin-app.ini`), **default `2`**, tunable over time. The watcher reads the current tick interval from the manager-tick loop and multiplies by the configured factor.

### 4.4 Verification closes the loop

"Done" is a worker **claim** (`done-claimed`), not a closure. The accountable item closes only when the **manager verifies** (→ `verified`). This is the "accountability never transfers down" invariant made operational: the manager cannot discharge the item by the worker's say-so alone.

**Verification rigor — RULED (2026-06-16, Rick): TIERED.**
- **Fleet-auditable or cross-persona work** → verification MUST cite a **primary artifact** (receipt / test result / commons entry / job-id), consistent with the standing no-confabulated-results rule.
- **Trivial in-lane items** → a plain **manager attestation** suffices (no artifact ceremony).
- The boundary is blast-radius: rigor scales with auditability, exactly as we already gate auditable claims elsewhere.

---

### 4.5 Escalation hygiene — clear on ack-or-self-park, never over-fire

*Empirical note (Mr Radio 🦉, 2026-06-16): the arbiter fired "blocking María — post status or unblock" at the manager **2–3×** while the worker was actually validly parked + fine. The lesson below is the fix.*

**Every escalation in this mechanism must be idempotent and self-clearing — not a repeating alarm.**

- A worker's `awaiting:manager` flag — and any arbiter escalation derived from it — **CLEARS** on EITHER (a) **manager-ack** (the manager has seen the await and owns the chase), OR (b) **the worker's self-park** (the worker has declared a valid heartbeat-hold / posted status). Once cleared, it does **not** re-fire.
- The aged-escalation poke (§4.3) marks the item `escalated` after its single fire for exactly this reason — **one poke, not a stream.**
- A **validly-parked worker** (declared hold + documented `work_owed`) is **NOT** a blocking condition: the arbiter MUST read the worker's declared hold before escalating "blocking" to the manager. Over-firing at a manager about a worker who is fine is noise that erodes the signal — the same FM-18-family failure the heartbeat work has been fighting.

**Build implication**: the watcher/arbiter checks `(manager-ack OR worker-hold)` before each would-be escalation; escalation state is **one-shot-then-cleared**. This is the careful part of the build alongside the `T_escalate` INI key.

## 6. Mechanism questions — RULED (2026-06-16 guided walkthrough, all per recommendation)

1. **`T_escalate` value** → **adaptive, 2× the manager-tick interval**, with the multiplier exposed as a `configuration_manager` INI key (default `2`, tunable; never hardcoded — Rick's config rider). Folded into §4.3.
2. **Verification rigor** → **tiered** — cited artifact for fleet-auditable / cross-persona work; manager attestation for trivial in-lane items. Folded into §4.4.
3. **Widening invariant** → **invariant holds** — when worker self-tracking is later widened, the manager STILL holds a distinct accountable item per assignment until he verifies. The worker's self-minted *doing-item* and the manager's *accountable-item* remain separate records; widening adds the former without removing the latter. This is what makes the deadlock-fix survive the widening — see §3 D1's deferred-widening note. The duplication is the feature.
