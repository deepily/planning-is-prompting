## 0. Ratified Design v1 (2026-06-03 — Rick walkthrough)

**Naming.** **Heartbeat Hook** = THIS — a per-instance, *local* Claude Code `Stop`-hook that self-pokes on quiescence-with-work-owed. **Heartbeat Poker** = the EXISTING agentic daemon (`cascade_heartbeat_scheduler.py` / `HeartbeatPokerJob`) that polls + DMs. **Two separate, complementary implementations** (Hook = generic per-instance; Poker = managed-fleet liveness + cross-session/`owner_id` AFK keep-alive).

**Ratified decisions:**
1. **Two separate, complementary impls** (not a merge).
2. **Hook = LOCAL self-poke** — reads its OWN state and decides; no central-arbiter dependency; deliberately does NOT lean on the `:7999` plane (the FM-7 fragile plane this design routes around). The cross-fleet *arbiter* role (dependency graph, `owner_id` AFK keep-alive, fleet throttle) belongs to the **Poker** — deferred to a later conversation. Optional future: the Hook may *emit* a quiescence event the Poker consumes, but its poke never *depends* on it.
3. **Discriminator = DECLARED-HOLD-ONLY.** The paused worker must **defend its quiescence** (Rick's framing): declared + fresh hold → don't poke · work owed + no fresh hold → poke · nothing owed → don't poke. Work-owed signals (strongest first): declared hold-reason · TODO `in_progress`/unstarted owned items · Pending-Decisions not blocked on user · open `expect_reply` DM.
4. **Trigger = `Stop` hook** *(verified via claude-code-guide).* Fires at the idle boundary (not `Notification`/`SubagentStop`). Hook input carries no owed-work flag → the hook inspects state itself (the hold file + TODO/transcript).
5. **Poke = native `Stop` `{"decision":"block","reason":"…"}` re-prompt — NO tmux.** The block path re-injects `reason` as context and forces the instance to continue; cleaner + integrated + reinforces LOCAL (the instance's own hook re-prompts itself). tmux/`cc_notification_listener` stays the *Poker's* cross-session delivery path.
6. **Safety = MANDATORY per-session poke-cap** (default 3). Built-in loop guard is undocumented (`stop_hook_active` unconfirmed) → treat as our responsibility. Block only while `poke_count < cap`; at cap → `{"continue": true}` + `notify()` "max nudges reached, awaiting user." Pure-JSON-only stdout; ~5s timeout (not the 600s default); silent-failure-safe.
7. **Artifact = per-session `.heartbeat-hold-<session_id>.json`** (gitignored; runtime-state family w/ `.claude-session.md`, `.claude-memento.md`). **Per-session = multi-writer safe** (Rick's catch — many instances write these; one shared file would collide). Each instance writes/reads only its own (hook derives the filename from the `session_id` in its input); a future Poker globs `.heartbeat-hold-*.json` for the fleet view. **Format (JSON, Rick-approved):**
   ```json
   {
     "session_id": "af1465b9",
     "persona": "María 🌸",
     "held_at": "2026-06-03T16:45:00Z",
     "ttl_seconds": 900,
     "work_owed": true,
     "reason": "Holding on Tiberius's gate-zero confirmation",
     "awaiting": "peer:Tiberius"
   }
   ```
   Fields: `session_id` (= filename suffix) · `persona` · `held_at`+`ttl_seconds` (freshness; expired ⇒ undeclared ⇒ pokeable) · `work_owed` (false ⇒ done ⇒ never poke) · `reason` · `awaiting` (`user:<name>` / `peer:<persona>` / `commons:<topic>` / `none`).

**`Stop`-hook decision logic (local):**
1. Read `.heartbeat-hold-$session_id.json`.
2. Present + fresh (`now − held_at < ttl_seconds`) + `reason` non-empty → **honor hold** → `{"continue": true}`.
3. Else determine work-owed (file `work_owed`, else TODO/Pending scan). None owed → `{"continue": true}` (genuinely done).
4. Work owed + `poke_count < cap` → `{"decision": "block", "reason": "Stopped with work owed and no fresh hold: <specifics>. Resume, or declare a hold."}`; increment counter.
5. `poke_count >= cap` → `{"continue": true}` + `notify()` "max auto-nudges reached, awaiting user."

**Remaining knobs (non-blocking; settle at build / with Rachel):** exact TTL value · opt-out / genuine-end-of-session mechanism · the shared `Stop`/`PostToolUse` hook substrate with Rachel's token-rate instrumentation (TODO line 11) — build the substrate once, fan out to (a) harvest-rate and (b) the poke.

**Build (Lupin-side):** hook script + `settings.json` wiring (`.claude/settings.json` so each project opts in). **First** verify CC's loop-guard behavior empirically (does returning `block` twice loop, or is `stop_hook_active` set?). Bound by construction before any live install — no new storm vector (cf. Sam TTS-storm postmortem).

### 0.1 Spike findings (2026-06-04 — empirically confirmed, build session)

Implementer **Tiffany 💍** (Lupin session, Tiberius-managed) ran the §0-#6 loop-guard spike against Lupin's *existing* production `stop.py`. Confirmed outcomes — these are now binding for the build:

1. **Loop-guard CONFIRMED both ways** (claude-code-guide docs **+** production `stop.py:658,687-691`, which already depend on it): Claude Code **does** set `stop_hook_active=true` on the `Stop`-hook re-fire. → **§0 #6 settled**: the per-session poke-cap is a **BACKSTOP**, not the sole guard. CC's native guard breaks the loop; our cap is the policy bound on top.

2. **Not greenfield — `stop.py` IS the shared substrate.** The heartbeat is an **additive extension** of the existing production `stop.py`, *not* a new hook. Its decision flow maps to: Branch A speakerphone-skip (early-exit) → loop-guard early-return → **Branch B** voice-driven block (the ONLY existing block path, already capped by `MAX_STOP_BLOCKS=3`) → **Branch C** no-voice quiescence (idle-waiter / anything-else).

3. **Clean seam = Branch C** (no-voice quiescence *is* the heartbeat boundary). Heartbeat slots in additively, **downstream of the loop guard**, returning **before** the idle-waiter, with its **own separate per-session heartbeat counter** (distinct budget from `MAX_STOP_BLOCKS` — do NOT merge the two caps). **Voice always wins** — the heartbeat never pre-empts a voice-driven block.

4. **Binding correction — poke rides `reason`, NEVER `systemMessage`.** claude-code-guide wrongly claimed a `Stop` hook's `reason` is "shown to user, not fed to Claude"; **production contradicts** — `reason` IS acted on by Claude (the re-prompt depends on it), and CC **silently ignores `systemMessage`** (per `hook_common.py:549`). The poke text MUST be emitted as `reason`.

5. **Cross-lane gate stands**: because the same `stop.py` will carry Rachel's token-rate instrumentation (TODO line 11), the Branch-C insertion is a **3-way seam-factoring** (Tiffany / María / Rachel) and `stop.py` stays **untouched** until Rachel reviews. Independent, no-collision work proceeds in parallel: the standalone `.heartbeat-hold-<session_id>.json` artifact module (built/tested to the §0 #7 schema, zero `stop.py` touch).

6. **Clarification — `expect_reply` DM directionality (resolves a §0 #3 / §4 ambiguity).** The work-owed signal "open `expect_reply` DM" means an **INBOUND** unanswered question the instance *owes a reply to* ⇒ **work owed ⇒ poke**. An **OUTBOUND** `expect_reply` DM where the instance is *awaiting a peer* is the opposite — a **legitimate hold** ⇒ it belongs in the hold artifact's `awaiting` field (`peer:<persona>`), NOT the work-owed oracle. (§4's phrasing "an open expect_reply DM this instance is genuinely *waiting on*" reads as the outbound/hold case; §0 #3's "open expect_reply DM" as a *work-owed* signal only makes sense as the inbound-owe case — this split reconciles them.) The oracle's input is named `unanswered_inbound_questions` accordingly. Surfaced by Tiffany during the work-owed-oracle build; ruling by María.

**Decision:** **land v1 as-is and put it into use immediately** while continued development proceeds. v1 = the local self-poke / hold-management core (honor fresh declared holds · poke stale/reasonless declared holds · `oracle_verdict=None`; no undeclared-lazy-stop catch yet).

**Continued development REOPENED — two extension tracks:**
- **(a) FM-19 undeclared-lazy-stop catch (the "v2" oracle wiring)** — gated on §C.3 Q4 (authoritative per-session work-owed source + `owned_by_me` semantics). María owns.
- **(b) Arbiter consumption of poke-responses — the Hook→Poker bridge.** This **promotes §0 decision #2's "optional future"** (*"the Hook may emit a quiescence event the Poker consumes, but its poke never depends on it"*) from deferred to **ACTIVE development**. The **arbiter** (the agentic **Heartbeat Poker** — the cross-fleet daemon, NOT the local Hook) consumes the local Hook's **poke-events + poke-outcomes** (did the instance resume? declare a hold? hit the cap?) to build the cross-fleet liveness / dependency view. This is a concrete first piece of the deferred **Heartbeat-Poker arbiter deep-dive** (P0 pending decision) — it gives that design real input data.

**Design steer (María, design author) — EMIT NOW, CONSUME LATER:** v1 lands as-is and is NOT gated on the arbiter. But the **first extension step** should be **forward-compatible poke-event emission** — v1's poke path writes a consumable poke-event/response record (a per-session event artifact in the `.heartbeat-*` runtime family, or a commons event) **even though nothing consumes it yet**. Then the arbiter is added as a *pure consumer* with **zero Hook retrofit**. Design the emission seam for the future consumer up front; the local poke still never *depends* on the arbiter (§0 #2 invariant preserved — the Hook is autonomous; the arbiter is an observer of its exhaust).

**Invariant to hold:** local self-poke remains independent and `:7999`-free. The arbiter consuming poke-responses is an *additive observer*, never a dependency in the poke path.

**Emission-record schema (canonical — María design-author + Tiffany impl, manager-blessed 2026-06-04).** The forward-compatible poke-event record v1 emits:

- **Artifact:** `~/.claude/heartbeat-events/<session_id>.jsonl` — **JSON Lines, append-only, per-session** (multi-writer-safe — each session writes only its own; arbiter globs `~/.claude/heartbeat-events/*.jsonl`), gitignored.
  - **Location = FLEET-WIDE**, default `~/.claude/heartbeat-events/` (`base_dir` injectable for tests). **Deliberate, documented divergence** from the hold artifact (which lives in the *local project root* because its consumer is local): the events log's consumer is the **cross-fleet arbiter**, so a single fleet-wide dir gives it ONE glob across all sessions/projects, durable across reboots, no project-root clutter. ⇒ this is the one place the `cu.get_project_root()` default intentionally does NOT apply. (If the agentic Poker already globs a known fleet dir — e.g. the cosa-voice bridge-file dir — align there; Rachel owns that substrate.)
- **`schema_version`-pinned** (= `1`) so v2 can evolve safely.
- **Emit policy:** v1 emits `{poke, honored, cap_reached}`. **v2 ADDS `idle`** (Rick's §6.2 HYBRID ruling, 2026-06-04; idle-GATE ruled by María 2026-06-04). The genuine-idle beacon fires when **`not_owed`** — i.e. NO `Task*` has latest status in `{in_progress, pending}` (covers BOTH "tracked work, all completed" AND "no tracked tasks") — **edge-triggered / de-duped** (fire on the *transition* into idle, NOT every idle stop). Still SKIP plain per-turn `not_owed` between owed-work (the edge-trigger handles that). The arbiter reads this beacon as `declared-available` (vs inference-derived `quiet (inferred)`).
  - **GATE RULING (María, overriding the `+ empty-total-Task*-set` STRICT steer):** idle = **`not_owed` alone**, NOT `not_owed AND empty-total-set`. Reason: the empty-total-set narrowing would **exclude the all-completed case** — a session that tracked tasks and finished them all is the *strongest* "I'm done, reassign me" signal, and empty-total-set fires only for sessions that *never tracked work* (the LEAST authoritative idle — untracked work can't be ruled out). The narrowing inverts the trust gradient. The "no false declared-available" intent is served instead by **(a) the arbiter trust-label** (`declared-available` is manager-*weighed*, never auto-reassigned) **+ (b) supersession** (a transient between-tasks idle is corrected the instant the next `poke`/`honored`/`cap_reached` lands). Matches Tiffany's built gate ("empty *owed*-`Task*` set" = `not_owed`) — **no rework.** *(v2.1 option if between-tasks idles prove noisy in production: a settle-delay or "≥1 resolved task" qualifier. Not now.)*
  - Fields:
  - `schema_version` (= 1; adding the `idle` enum value is shape-compatible) · `session_id` · `persona` (from hook input/session — known even on an undeclared-stop poke with no hold) · `ts` (ISO-8601 UTC)
  - `outcome` — `poke` | `honored` | `cap_reached` | `idle`  *(`idle`: v2; `reason` omitted, `work_owed=false`, `awaiting=null`)*
  - `poke_count` — per-session heartbeat poke count *after* this event · `cap` — configured per-session poke-cap
  - `work_owed` — bool, **`null` in v1** (`oracle_verdict=None`) → real bool at v2 with **zero schema change**
  - `awaiting` — from the hold artifact if present, else `null`
  - `reason` — poke reason text, **only when `outcome==poke`** (the human "why" for arbiter→user escalation), else omitted
- **Module:** `emit_outcome( session_id, persona, outcome, poke_count, cap, work_owed=None, awaiting=None, reason=None, ts=None, base_dir=None ) -> bool` + `read_events( session_id, base_dir=None )` for the consumer.
- **Write contract:** fire-and-forget, wrapped `try/except`, **NEVER raises into the poke path** (emission failure ⇒ `False`, poke still proceeds). Single-line JSONL append; never-raise read; 100% cov (mirrors the hold module).
- **Arbiter (v2 consumer)** tails recent lines across sessions → fleet liveness (who poked when), dependency view (`awaiting`), stuck-detection (repeated `cap_reached` with `work_owed=true`).
- **Deferred knob (v2, not v1-blocking):** JSONL rotation / line-cap (rotate daily or cap at N) so the log can't grow unbounded.

## 1. Motivation

A recurring, empirically-observed failure: a Claude Code instance reaches a **quiescent / stopped** state with **in-scope work still owed**, and simply *sits* — because nothing pokes it. This session produced two live specimens within ~90 minutes:
1. The **Gate-Zero setup idle** — the steward sat waiting through Gate-Zero with no one to nudge it forward.
2. The **mid-incident hold** — parked non-blocked owned work ("waiting for Tiberius's ping") instead of doing it; Rick had to poke via broadcast.

Both are instances of the **FM-19 boundary** (completion-discipline / *difficulty ≠ defer*): the work was owed and non-blocked, but the instance idled. Rick's framing, verbatim and only half-joking: *"do you literally need a heartbeat poker to give you a tap on the shoulder?"* — the answer, uncomfortably, was yes.

**The rich irony driving this doc:** the mechanism proposed here is *exactly* what would have auto-caught both idles. We are designing the thing the steward needed an hour before it was proposed.

## 2. Core concept

Use **Claude Code stock hooks** — specifically the **`Stop` hook** (fires when an instance finishes responding and goes quiescent) — as a *naturally-occurring* heartbeat trigger, rather than relying on an **external polling daemon**.

When an instance goes quiescent, its own `Stop` hook fires. That hook evaluates *"does this instance actually have work left?"* and, if so, injects a poke back into the session via **`tmux send-keys`** — e.g. *"Hey, don't you have something to work on? Are you sure you're finished? Get back to work."*

The poke is **self-originating** (the instance's own lifecycle event triggers it), so there is **no external daemon polling `commons_who`**, no topic-name plumbing, no liveness-sweep timer to maintain. The quiescence event *is* the signal.

## 3. Why this is attractive vs. the current poker

| Dimension | Current external poker (`cascade_heartbeat_scheduler.py` / `HeartbeatPokerJob`) | Stop-hook self-poke (proposed) |
|---|---|---|
| Trigger | Daemon polls on a timer (`commons_who` + DM) | Native `Stop` lifecycle event — fires exactly when an instance goes idle |
| Plumbing | Commons DM topic per recipient (PG-6: topic-name slugify bug; 422s on spaces) | `tmux send-keys` directly into the session (infra already proven by `cc_notification_listener`, which injects messages this way) |
| Coverage | Only sessions the daemon knows about / was launched for | **Generic** — any Claude Code instance with the hook installed self-pokes |
| Failure mode | Daemon dies → no pokes; orphan-listener / reap hazards | No long-lived daemon to die; poke is event-driven |
| The hard part | "Is the worker stalled?" (liveness) | **"Is the instance *legitimately done/holding* vs *lazily stopped*?"** (see §4) |

The existing **tmux-injection path already exists** — `cc_notification_listener` injects messages into sessions via `tmux send-keys` (observed live this session: "Injected message via tmux 'cc-tmux-session-…'"). So the *delivery* mechanism is proven; the novelty is the *trigger* (Stop hook) and the *decision* (poke-or-not).

## 4. The hard problem — the hold-vs-lazy discriminator

**This is the crux, and the reason it needs real design, not just a hook script.**

A naive Stop-hook poker that pokes **every** quiescent instance re-creates the exact noise problem cosa-voice already fights (cf. the TTS-storm incident, `src/rnd/2026.06.02-sam-tts-storm-incident-postmortem.md`). It must distinguish:

- **Lazily stopped, work owed → POKE.** In-scope mandated work left undone; instance idled rather than continuing (the FM-19 case).
- **Legitimately holding → DO NOT POKE.** Waiting on a *real* cross-lane dependency (e.g. the steward correctly holding on Tiberius's Gate-Zero gate), or genuinely complete, or blocked on a true user decision.

Poking a legitimately-holding instance is not just noise — it pressures it to *manufacture* activity, which is the inverse failure (busy-work / premature action). The discriminator is the whole ballgame.

**Candidate signals for "work owed":**
- An open TODO / task list with `in_progress` or unstarted owned items.
- A `## Pending Decisions` queue with items not blocked on the user.
- A self-declared **hold-reason** the instance writes before stopping ("HOLDING on: Tiberius's Gate-Zero gate") that the hook can read and honor — i.e. **legitimate holds must be *declared* to be respected.** Undeclared stop = pokeable.
- Cross-referencing commons: is there an open `expect_reply` DM this instance is genuinely waiting on?

**Design principle (proposed):** *a hold is only honored if it is declared.* This flips the burden correctly — an instance that wants to not be poked must state why it's holding (a durable, inspectable hold-reason). Silence with work owed = poke. This also dovetails with the steward discipline of always stating *what* one is holding on.

## 5. Open design questions (for the post-grind investigation)

1. **Which hook?** `Stop` (end of a response turn) vs `Notification` (idle-waiting-for-input) vs `SubagentStop`. The "quiescent" state Rick means is probably the `Stop`/idle boundary — needs verification against Claude Code's actual hook semantics.
2. **Self-poke vs central arbiter.** Does each instance's own hook decide-and-poke, or does the hook *report* quiescence to a central arbiter (the `:7999` backend) that holds the work-ledger and decides? Central arbiter = better cross-session view (knows who's waiting on whom) but reintroduces a coordination dependency.
3. **The "are you really done?" oracle.** How authoritative can the work-owed check be? TODO parsing is heuristic. Is there a stronger signal?
4. **Escalation / loop discipline.** If poked and the instance *still* stops with work owed, escalate how? Cap the poke count? (Avoid an instance ↔ hook poke-loop — a self-inflicted storm.)
5. **Opt-out / quiet hours.** Per-instance disable; respect a genuine end-of-session.
6. **Relationship to existing poker.** Does this *replace* `cascade_heartbeat_scheduler.py` / `HeartbeatPokerJob`, or *complement* it (Stop-hook for the generic per-instance case; external daemon for managed-fleet liveness sweeps)? Likely complement.
7. **Shared infra with Rachel's TODO-line-11 work.** Option (b) there is *also* a Stop/PostToolUse hook parsing session JSONL (for token-rate harvest). **Both want a Stop/PostToolUse hook reading session state → cosa-voice.** Build the hook substrate *once*, fan out to (a) harvest-rate instrumentation and (b) the poker. Coordinate with Rachel so we don't ship two competing hooks.
