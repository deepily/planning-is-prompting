## 1. Executive Summary

The Phase D cascaded plan-review prototype ran end-to-end against the synthesized 2-section toy plan (`src/rnd/2026.05.18-toy-input-plan-email-verification.md`). **The cascade's core value proposition was confirmed**: 12 findings surfaced across 2 reviewers in ~5 min of actual work; Tiberius (manager) absorbed 9 autonomously and escalated only 3 cross-section foundational findings to Mr. Rick. The "save the user's attention" hypothesis is validated.

**However, total wall-clock was ~55 min** of which an estimated **~30 min was dead air time** caused by three reproducible bugs/limitations. The reviewers' actual review work took 1.5-2.5 min per stage; the rest was the cascade waiting for itself.

A rerun from zero with the lessons learned applied should compress the cycle to ~15-25 min — closer to the underlying reviewer compute time.

## 3. What Didn't (Dead Air Time)

**Two complementary dead-air accountings** (per Tiberius's Q1 input):

| Metric | Owner | Run 1 value | Definition |
|--------|-------|-------------|------------|
| **User-facing dead air** | María (Workflow Steward) | **~30 min** | Time Mr. Rick's attention was consumed by stall-detection |
| **Pipeline-wide dead air** | Tiberius (manager) | **~80 min** | Time between worker post-complete and manager detection-complete |

Both are useful. The user-facing metric is the value-proposition KPI (we promised to save user attention). The pipeline-wide metric is the worker-blocking KPI (sets the upper bound on wall-clock compression). **Specifically**: Rachel's Section A Stage 1 posted at 15:03:53; Tiberius detected it at 15:40:43 — that's a 37-min detection-delay. Arnold's Section A Stage 2 and Rachel's Section B Stage 1 also incurred ~40+ min detection-delays each.

**Post-escalation cycles ran clean** (no dead-air contribution): F1 re-litigation cycle = ~3 min; cluster-resolution cycle = ~4 min (incl. brief DM-truncation hiccup recovered fast). The dead-air problem is detection-stale, not execution-slow.

**Attribution to three compound failure modes**:

| Symptom | Root cause | Approx dead time contribution |
|---|---|---|
| Manager couldn't see reviewer posts even after reviewers posted on time | Read-side body-display truncation in `commons_read` API | ~15 min (×2 across sections) |
| Manager couldn't notice silence to detect it himself | Turn-based-CC limitation (no autonomous ticks) | ~10 min |
| Initial cascade invocation needed manual wake-up | Manager dormancy + push-mode wake unreliability | ~5 min |

**Each dead-air episode required Mr. Rick to notice and either wake a session or ask María to investigate.** That's the inverse of the value proposition — the user's attention was paid not on the work itself, but on detecting that the cascade had silently stalled.

## 4. Three Failure Modes (documented)

### 4.1 Read-side body-display truncation (NEW, reproducible)

**Symptom**: `commons_read` API body field returns only the header/first-paragraph for entries above some length threshold (~250-500 chars suspected). Disk content is intact; only the API response truncates.

**Impact on this run**: caused Tiberius to miss BOTH Arnold's Section A Stage 2 review AND Rachel's Section B Stage 1 review even though both were posted on time. Each missed detection added ~10-15 min of dead time before María's status-ping woke him to disk-read.

**Distinct from** Sub-bug B (write-side truncation), which was a pre-existing Lupin bug from 2026-05-17. NOT reproduced this run — the writes were all intact.

**Fix path**: Lupin-side investigation + fix. Operational workaround (see §5): manager MUST disk-read every section topic on every wake event.

**Status**: ✅ **FIXED 2026-05-18 post-Run-1** via Rio's investigation; verification probe by Tiberius at 18:37 UTC confirmed `commons_read` now returns full ~2200-char body with all sentinel markers present and embedded YAML structure intact. Disk-read remains the defense-in-depth fallback (the manager's universal-step-zero discipline still applies), but the API workaround is no longer mandatory for Run 2.

**Knock-on impact on telemetry collection** (Tiberius's preemptive heads-up, 2026-05-18 16:47): the §10 telemetry scrape plan called for `commons_read(since=..., limit=200+)` — for long review posts, these would return header-only. **Updated telemetry collection method**: disk-read the commons topic files directly at `<lupin>/io/commons/cascaded-prototype-*.md` (and `dm-*.md` for DM threads). Same workaround as manager-side Lesson 1 below.

### 4.2 Manager turn-based-CC limitation (LOAD-BEARING)

**Symptom**: Claude Code sessions are turn-based, not autonomously-ticking. The manager cannot fire heartbeat pings without being itself triggered by an inbound event. So even if §6.4's heartbeat-ping cadence is configured to "every 5 min", nothing in the runtime actually fires it on a schedule.

**Impact on this run**: when reviewers finished their work and went idle, the manager had no event to react to. He couldn't detect "Rachel has been silent for X minutes" because nothing was poking him to look. The §6.4 heartbeat protocol assumption is fundamentally incompatible with CC's runtime model.

**This is the load-bearing v2 finding.** Several mitigation paths are viable (see §6); the design doc §6.4 must be rewritten in light of this constraint.

**Status**: design-doc level. Not a bug to fix — an architectural assumption to revise.

### 4.3 Push-mode wake unreliability + manager dormancy

**Symptom**: cosa-voice push-mode via tmux send-keys does not reliably wake fully-idle CC sessions. DMs may dispatch with `push_mode_active: true, dm_dispatched: true` and still not actually wake the recipient. We hit this on the initial cascade invocation (Tiberius dormant after first acks landed) AND on reviewer-to-manager turnaround (Rachel/Arnold posts not waking Tiberius).

**Impact on this run**: required manual user intervention (Mr. Rick gently waking Tiberius in tmux) to revive the cascade. Once revived, the queued DMs landed and work resumed.

**Status**: file as Lupin-side bug-fix-queue investigation. Push reliability is the load-bearing assumption beneath the entire DM-based cascade architecture.

## 5. Operational Lessons (immediate v2 workflow guidance tightening)

**LESSON 1 — Manager disk-read discipline** (Tiberius committed to this live during the run):

> Manager MUST disk-read every section topic on EVERY wake event, not just when he suspects a post landed.

Rationale: playbook §5 assumes manager observes topic posts in real-time, which is incompatible with BOTH read-side body-display truncation AND turn-based-CC. Disk-read sidesteps both bugs.

**LESSON 2 — Manager-side wake-coupling on dispatch**:

When the manager DMs a worker triggering work, the manager should also queue a "self-check at T+X min" via a side channel. Currently, when dispatching to Rachel, Tiberius had no way to schedule "and check back in 5 min to see if she posted". The wake-coupling needs an external mechanism.

**LESSON 3 — Briefing topic must be a direct-DM channel for the manager**:

María's orientation briefing went to a new commons topic Tiberius wasn't tailing. The GO signal had to be hand-DM'd. v2: either the manager auto-subscribes to the briefing topic at workflow launch, or the briefing is sent as a direct DM to the manager (with workers receiving topic-pointer copies).

**LESSON 4 — Manager-as-phantom recovery workflow guidance**:

Original design §3.7 covered worker-phantom under `park_and_escalate`. Manager-phantom is a stronger case — the whole pipeline stalls. v2 needs explicit "if manager unreachable for N min, user wakes manager" workflow guidance.

**LESSON 5 — Ack-format ambiguity**:

Briefing said "acknowledge his DM" without naming WHOSE DM. Three workers pre-acked María's brief; one (Mr. Radio, correctly) waited for Tiberius's role-assignment DM. v2 briefing language: "Wait for Manager's role-assignment DM specifically (not the Workflow Steward's briefing); acknowledge his DM with 'ready, [role]'."

**LESSON 6 — Universal step zero** (Tiberius observation):

Manager's first action on every wake event is `disk-read every active topic`. Broader than Lesson 1 — applies to ALL topics (section topics + DM topics + briefing topics), not just suspected-post-landed cases. Encode as pre-flight checklist in the manager system prompt.

**LESSON 7 — Pre-emptive worker probes have second-order value** (Tiberius observation):

When the manager probes other workers while suspecting one is phantom, the probes confirm liveness of the others (Rio in 75s, Arnold in 115s in Run 1) AND surface that dormancy is SELECTIVE — refines the hypothesis space. v2 workflow guidance: when one worker is suspect, preemptive-probe the rest in parallel.

**LESSON 8 — Multi-finding clusters → single combined escalation** (Tiberius observation):

Manager's Trigger 1+2 combined escalation gave Mr. Rick ONE decision-point instead of three per-finding escalations. Saved cognitive cycles + UX-friendly (`ask_multiple_choice` with 3 options vs 3 sequential yes-no questions). v2 workflow guidance: cluster cross-finding escalations whenever the findings share a root cause or resolution will touch the same section.

**LESSON 9 — Manager-classification posts as worker-facing audit trail** (Tiberius observation):

`kind: "manager_classification"` posts to section topics give workers visibility into how their findings were classified without the manager having to DM each one. v2 workflow guidance: manager posts classification to section topic at every stage-close, not just on escalations.

**LESSON 10 — Workarounds-mid-run become workflow guidance for the rest of the run** (Tiberius observation):

When body-display truncation forced Tiberius into "post on topic + brief DM pointer" pattern at 16:40, he applied it for ALL subsequent communications. v2 workflow guidance: emergent operational patterns adopted mid-run apply universally for the run's remainder; don't treat them as one-time hacks.

**LESSON 11 (meta) — Self-audit discipline** (Tiberius observation):

Manager failed disk-read discipline TWICE in Run 1 (Rachel false-stall; then Arnold + Rachel-B detection delay). Both times the failure was forgetting that disk-read should be step ZERO of every wake, not step N when something suspicious is sensed. v2: add a "self-audit checklist" to the manager system prompt — "Did you disk-read all active topics this turn? If no, do it before responding."

## 6. Recommendations for Rerun

**Mr. Rick's stated goal**: "clear everybody's context and rerun the experiment from zero so that we pull all of that latency and dead-air-time out of the process."

**Three feasibility tiers for eliminating dead air time**:

### 6.A — Cheap (apply immediately for rerun)

1. **Lesson 1 in effect**: Tiberius disk-reads every section topic on every wake event
2. **Lesson 5 in effect**: tighten briefing language on ack target
3. **Lesson 3 in effect**: GO signal sent as direct DM, not topic post
4. **All 5 sessions fresh** with clear context — eliminate any carry-over state
5. **María observes more aggressively** (per Tiberius's Q3 input): **medium-aggressive, every 5 min, always disk-read first, ping only on confirmed stall patterns**. Fixed cadence > judgment-driven (workers + manager learn to expect it). Complements §6.B's external scheduler — scheduler pokes the manager mechanically; María watches for actual stall patterns the scheduler can't detect (two-layer reliability)

Expected impact: cuts ~15-20 min of read-truncation dead time. Cascade should drop from 55 min to ~30-35 min.

### 6.B — Medium (1-2 hour engineering)

6. **External heartbeat scheduler** (Tiberius's recommended shape, per Q2): a separate process (cron, scheduled GitHub Action, systemd timer, or small daemon) that pokes the manager via `commons_send_to(recipient="tiberius", body="heartbeat", expect_reply=False)`. Cadence: **2-3 min during active cascade, 5+ min idle**. Scope: **manager-only** — workers are pull-driven by manager DMs, so worker-pokes create noise without value. State machine: scheduler tracks cascade-active vs cascade-complete (driven by manager's `kind: cascade_complete` post). Failure handling: if no manager response to 3 consecutive pokes, escalate to user via `notify()`. This is the v2 fix for the turn-based-CC limitation — the manager doesn't need to autonomously tick because the external scheduler does it for him.
7. **User-side dead-man's-switch**: if the manager hasn't pushed a status notify in N min, Mr. Rick gets a heads-up `notify()` saying "cascade may have stalled — consider checking commons_who". Implemented as the failure-handling branch of the same external scheduler in #6.

Expected additional impact: cuts another ~5-10 min. Cascade should drop to ~20-25 min.

### 6.C — Heavy (requires Lupin bug fixes)

8. **Fix read-side body-display truncation** in the `commons_read` API — eliminates Lesson 1's need for disk-read fallback.
9. **Fix push-mode wake reliability** for dormant CC sessions — eliminates the dormancy-revival manual intervention.

Expected additional impact: cuts the residual dead-air down to near-zero. Cascade approaches reviewer compute time (~10-15 min for a 2-section plan).

**Recommended rerun stance**: ship 6.A immediately (no engineering), file 6.B as a follow-on R&D plan, treat 6.C as a Lupin bug-fix-queue item already in progress per Mr. Rick's assignment.

6. **María observes aggressively** — periodic `commons_who` + disk-read of cascade-* topics to catch any new dead-air patterns before they balloon
7. **Tiberius applies Lesson 1** — disk-read every section topic on every wake event
8. **Telemetry**: same 5 metrics as Run 1 (intervention count, message count, wall-clock, per-stage breakdown, finding-by-severity); add three new metrics:
   - **dead-air ratio** = (wall-clock − sum of stage-active times) / wall-clock
   - **detection-delay** per stage = (manager-detected-ts − worker-post-ts), reported per stage (Tiberius Q1 — load-bearing manager-side failure mode in turn-based-CC)
   - **Run-1-vs-Run-2 deltas** on user-intervention count + pipeline-wide dead-air + detection-delay distribution
   
   **Severity-tag metadata schema additions** (Tiberius Q4):
   
   | Field | Type | Purpose |
   |-------|------|---------|
   | `cross_section` | bool | Orthogonal to severity — captures cross-section dimension independently |
   | `closure_action` | enum | `"ignore" \| "documented" \| "revised" \| "escalated" \| "voted"` |
   | `parent_finding` | string | For downstream-stage sharpenings of upstream findings |
   | `rounds_used` | int | Re-litigation rounds consumed (0 if none) |
   | `votes_called` | int | Votes invoked for this finding (>0 means contentious) |
   
   **Stamping discipline**: manager stamps authoritative `manager_classification` at stage-close; reviewers stamp `severity_proposed`. Both useful for proposed-vs-final comparison.
   
   **Collect via disk-read** of `<lupin>/io/commons/cascaded-prototype-*.md` + `dm-*.md` files directly — NOT via `commons_read` API, which truncates long bodies (per Tiberius's 2026-05-18 16:47 heads-up; cross-references §4.1 above).

**Acceptance criterion for Run 2**: dead-air ratio < 50% (Run 1's was ~55%). If under 30%, ship Lessons 1-5 as v2 workflow guidance and consider 6.B unnecessary.
