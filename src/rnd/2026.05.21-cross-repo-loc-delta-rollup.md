## 1. Motivation

Rick has been tracking per-repo git LoC deltas (additions, deletions, file types, daily breakdown) using `cosa.repo.run_git_loc_delta` since 2026-05-16. The tool writes a stable per-branch CSV (`{repo}/io/git-loc-delta/{repo}-{branch-slug}-loc-delta.csv`) that grows day-by-day as a persistent artifact of the branch's progress.

**The new need**: a **global cross-repo daily snapshot** consolidating LoC deltas across every repo Rick worked in on a given day. Single global headline + per-repo breakdown. Use cases:

- **End-of-day progress tracking**: "How much did I do today across all repos?" — single artifact instead of mental-summing N per-repo §6 summaries
- **Ad-hoc mid-day curiosity**: "What's my cross-repo activity so far today?" — invocable anytime
- **Long-run progress visibility**: comparing today's global delta against the rolling window
- **External reporting**: feeding executive briefings (per Session 93's `executive-briefings/` precedent) with quantitative cross-repo activity numbers

**Why not just re-read N per-repo summaries**: scales poorly past 3 repos; loses the cross-repo aggregate view; redundant per-repo formatting overhead.

## 3. Design Decisions (Ratified)

### 3.1 Origin: PIP workflow + cosa code (Option A)

**Locked**: workflow originates in PIP; code originates in cosa. Same hub-spoke split as the existing per-repo §6.

| Concern | Owner | Where |
|---|---|---|
| Workflow shape (when to invoke, how to surface) | PIP | `workflow/loc-delta-global.md` + `.claude/commands/plan-loc-delta-global.md` |
| Aggregator CLI implementation | cosa | `cosa.repo.run_git_loc_delta_global` (Rachel-owned) |
| Repo-list source-of-truth | Lupin | `lupin-app.ini § external repos` |

**Rejected alternatives** (full pros/cons in the original `ask_multiple_choice` abstract from this session):
- **Cosa entirely** — breaks hub-spoke pattern; cosa starts owning workflow-shape decisions
- **Lupin entirely (endpoint + cron)** — couples day-end summary to infrastructure uptime; workflow-as-service vs workflow-as-doc

### 3.2 Firing + surfacing pattern: Ad-hoc slash + testing-server cron (Option A)

**Locked**: two complementary surfaces.

- **Phase 1 (ship now)**: `/plan-loc-delta-global` slash command for ad-hoc invocation. Manual; calls cosa CLI; renders via `notify()` in the invoking session.
- **Phase 2 (ship later)**: testing-server scheduled job (cron-adjacent cadence, e.g. 23:00 local). Pushes the global headline as a notification regardless of session activity.

**Rejected alternatives**:
- **Appended to every session-end** — redundant when N sessions/day in N repos; couples to per-repo §6 reliability
- **On-demand slash command only** — manual-only undercuts automation value
- **Cron daily push only** — loses on-demand flexibility

### 3.3 Schema v2 (Rachel-ratified, Rick-greenlit "EXPAND FULL")

**Per-row CSV schema** (extends current v1 with two new columns):

```csv
date,repo,file_type,added,deleted,files_touched,commits
```

**Sidecar JSON** (alongside CSV at `{repo}-{branch-slug}-loc-delta.meta.json`):

```json
{
  "csv_schema_version": 2,
  "repo": "cosa",
  "branch": "wip-v0.1.7-...",
  "rev_range": "merge-base..HEAD",
  "since": "2026-05-15",
  "until": "2026-05-21",
  "generated_at": "2026-05-21T20:30:00+00:00"
}
```

**CLI flag**: `--repo-name NAME` added to `cosa.repo.run_git_loc_delta`. Defaults to `os.path.basename(target_repo_root)`. Lets a sub-repo like `cosa` inside `lupin` self-identify as `cosa` rather than its parent path.

**Rejected schema additions** (Rachel's principled push-backs accepted):
- Per-row `commit_sha_range` — denormalized; same SHA repeated per row. Replaced by sidecar JSON's `rev_range` field (one-per-file).
- Sidecar `csv_row_count` / `total_added` / `total_deleted` — versioned-derived-state is a maintenance trap; stale sidecar = silent inconsistency. Aggregator computes from CSV on-the-fly (sub-microsecond per file in pandas).

**Backward-compatibility**: aggregator reads sidecar if present; falls back to filename-derived repo identity if absent. Both schema generations work side-by-side during rolling-overwrite transition.

### 3.4 Plot module library-shape (Rachel-ratified)

**Signature**:

```python
def plot_summary(
    daily,           # dict[date_str -> {added, deleted, files_touched, commits, by_file_type, by_repo (NEW)}]
    summary,         # dict from agg.summary()
    output_path,
    group_by   = "file_type",   # NEW: "file_type" (default) or "repo"
    title_meta = None,           # NEW: dict for title enrichment
    debug      = False,
)
```

**Critical**: takes pre-aggregated dict input (not CSV path). Means the global aggregator builds a `daily_aggregated` dict with `by_repo` parallel to `by_file_type`, then calls `plot_summary(..., group_by="repo")`. Same matplotlib code, different group key.

**Title enrichment** via `title_meta` dict allows polymorphic titles:
- Per-branch case: `title_meta={"branch": "wip-v0.1.7-...", "rev_range": "..."}`
- Global case: `title_meta={"scope": "global", "repos": ["lupin", "cosa", "plan"], "rev_window": "2026-05-14..2026-05-21"}`

Missing `title_meta` falls back to stem-based title (belt-and-suspenders).

## 4. Phase 1 Specification

### 4.1 Cosa-side: `cosa.repo.run_git_loc_delta_global` (Rachel owns)

**Module location**: `<cosa>/repo/run_git_loc_delta_global.py` (sister CLI to existing `run_git_loc_delta.py`).

**Inputs**:
- `--repos REPO_PATH [REPO_PATH ...]` — list of repo paths to aggregate. Required.
- `--output FORMAT` — `console` / `json` / `csv` / `markdown`. Default `console`.
- `--save-output PATH` — optional file output.
- `--since YYYY-MM-DD` / `--until YYYY-MM-DD` — date range filter (passes through to per-repo reads or filters the aggregated DataFrame).
- `--plot` / `--plot-output PATH` — optional matplotlib output via the new `plot_summary(..., group_by="repo")` call.
- `--verbose` / `--debug` — standard.

**Algorithm**:
1. For each repo path in `--repos`:
   - Resolve CSV path: `{repo_path}/io/git-loc-delta/{repo}-{branch-slug}-loc-delta.csv` (derive `{repo}` from sidecar's `repo` field if present, else basename of repo path; derive `{branch-slug}` from current HEAD)
   - Load CSV via pandas
   - Load sidecar JSON if present; check `csv_schema_version` for compat handling
   - If schema v1 (no `repo` column): inject `repo` column with the filename-derived identity
3. `pd.concat` all per-repo DataFrames
4. Group by `date` → aggregate `added`, `deleted`, `files_touched`, `commits`
5. Group by `(date, repo)` → per-repo daily breakdown
6. Group by `(date, file_type)` → per-file-type aggregate (across repos)
7. Emit per `--output` format. For `--plot`: build `daily_aggregated` dict with `by_repo`, call `plot_summary(..., group_by="repo")`.

**Output JSON shape** (for programmatic consumption):

```json
{
  "since": "2026-05-14",
  "until": null,
  "repos": ["lupin", "cosa", "planning-is-prompting"],
  "summary": {
    "total_added": 1247,
    "total_deleted": 312,
    "total_files": 53,
    "total_commits": 21,
    "total_days": 5,
    "net": 935
  },
  "days": [
    {
      "date": "2026-05-21",
      "added": 458,
      "deleted": 27,
      "files_touched": 12,
      "commits": 5,
      "by_repo": [
        {"repo": "planning-is-prompting", "added": 260, "deleted": 14, "files_touched": 6, "commits": 2},
        {"repo": "cosa", "added": 198, "deleted": 13, "files_touched": 6, "commits": 3}
      ],
      "by_file_type": [
        {"file_type": "markdown", "added": 380, "deleted": 14, "files_touched": 8, "commits": 4},
        {"file_type": "python", "added": 78, "deleted": 13, "files_touched": 4, "commits": 1}
      ]
    }
  ]
}
```

## 7. Open Questions

### 7.1 Repo discovery — INI parse or hardcoded list?

The PIP-side workflow needs to know which repos to aggregate. Two options:

- **Parse `lupin-app.ini § external repos`** at invocation time. Source-of-truth. Adds Lupin INI as a runtime dependency for the workflow.
- **Hardcoded `LOC_DELTA_REPOS` list** in the workflow doc / slash command. Simpler; drifts when new scopes registered.
- **Hybrid**: default to INI parse with `--repos` override flag for ad-hoc subsets.

**Lean**: hybrid. Discuss in walk-through.

### 7.2 Missing-CSV handling — silent skip vs error vs report-as-stale?

If `lookml/io/git-loc-delta/lookml-main-loc-delta.csv` doesn't exist (no session-end ran in that repo on this branch yet):
- **Silent skip** — least surprising; risks invisible data loss
- **Error / abort** — most defensive; blocks the rollup
- **Report as stale** — include the repo name in output with `(no data)` notation

**Lean**: report as stale. Discuss.

### 7.3 Branch-aware aggregation — all branches per repo or current-branch only?

Each repo has potentially N branches each with its own per-branch CSV. Should the rollup:
- **Aggregate all branches per repo** — gives full activity view; risk of double-counting if a commit appears in multiple branches
- **Current branch per repo only** — matches the "what am I actively working on" mental model; misses sub-branch experiments
- **Filter to most-recently-touched branch per repo** — heuristic

**Lean**: current branch per repo, with `--all-branches` flag to opt into the full view. Discuss.

### 7.4 Single-day vs multi-day window default?

The slash command's default `--since` / `--until` window:
- **Today only** — matches the "what did I do today" framing
- **Branch lifetime** (current branch per repo since merge-base) — matches the per-repo §6 default
- **Last 7 days** — rolling weekly view

**Lean**: today only (matches the cross-repo daily snapshot framing). `--since`/`--until` flags for windows. Discuss.

### 7.5 Author filtering — needed?

The per-repo tool has `--author EMAIL`. Useful for the global rollup if Rick wants to filter to his own commits across repos vs include collaborators?

**Lean**: pass-through to per-repo invocations; default no-filter. Discuss.
