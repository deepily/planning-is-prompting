## Goal

Add a user-email-verification step to the registration flow so that user-supplied email addresses are confirmed as reachable before account activation. A verification email is sent immediately after registration; the user clicks a one-time link to mark their email as verified; unverified accounts cannot perform sensitive actions.

**Scope**:
- Two sections, independent enough to be reviewed in isolation
- Synchronous web flow only (no mobile push, no SMS fallback)
- Best-effort spam-resilience (rate-limit, no captcha for v1)

**Out of scope** (intentionally — keep the toy plan small):
- Account-recovery flow using verified email (separate concern)
- Re-verification on email-change (deferred)
- Multi-region email delivery (single region for v1)

## Section A: Verification Token — Data Model and Storage

**Depends on**: existing `users` table (no schema migration touches it)

**Decision**: Verification tokens are short-lived, single-use, opaque random strings stored server-side. The user never sees the token in cleartext after the verification email is sent.

### Data shape

A new `email_verification_tokens` table:

| Column | Type | Notes |
|--------|------|-------|
| `token_hash` | `bytea` (PK) | SHA-256 of the random token; raw token never stored |
| `user_id` | `bigint` (FK → users.id) | Cascading delete with user |
| `issued_at` | `timestamptz` | Defaults to `now()` |
| `expires_at` | `timestamptz` | Defaults to `issued_at + INTERVAL '24 hours'` |
| `consumed_at` | `timestamptz` | NULL until the link is clicked successfully |

**Index**: `(user_id, expires_at)` for the "do they have any unexpired token?" query path.

### Token generation

- Generate 32 bytes from a cryptographically secure RNG → URL-safe base64 (~43 chars)
- Hash with SHA-256 before storing; the cleartext token leaves the server exactly once (in the verification email URL)
- On verification request: hash the user-supplied token, look it up, atomically mark `consumed_at = now()` if not already consumed AND not expired

**Open sub-question 1**: do we store the token table in the primary application Postgres, or in a separate cache layer (Redis) given the short TTL? TBD — defer to ops review. Primary Postgres is the safer default for v1 (durability over latency).

### Acceptance criteria

- `EXECUTOR: AI` — pytest suite at `src/tests/auth/test_verification_token.py` passes 100%; covers token generation, hash storage, atomic consumption, expiry rejection, and replay prevention
- `EXECUTOR: AI` — schema migration applied cleanly against a fresh test database; rolled-back migration restores prior state byte-equivalent
- The verification flow works correctly end-to-end across the supported browsers

### Risks / known unknowns

- Token-table growth: with 24h TTL and a daily cleanup job, the table stays bounded. Cleanup job is in scope for Section A.
- `EXECUTOR: AI` — scheduled cleanup job at `src/jobs/expire_verification_tokens.py` runs every 6 hours via the existing job scheduler; deletes rows where `expires_at < now() - INTERVAL '1 hour'` (small safety margin)

## Section B: Verification Email and UX

**Depends on**: Section A's `email_verification_tokens` table; the application's existing email-sending infrastructure (SMTP gateway already deployed for password-reset emails)

**Decision**: On successful registration, immediately queue a verification email containing a one-time link of shape `https://<host>/verify-email/<token>`. The link lands on a server-rendered page that consumes the token and either confirms or shows a friendly error.

### Email composition

- Subject: `Please verify your email address`
- Body: short HTML + plain-text alternative; contains the verification link, an expiry mention (24 hours), and a "didn't request this?" footer
- From: the application's standard transactional `From:` address

The email is composed inline in `src/auth/verification.py` for v1 simplicity.

### Verification page (`GET /verify-email/<token>`)

- Look up token by SHA-256 hash; atomically consume if valid
- On success: set the user's `email_verified_at` timestamp; redirect to `/dashboard?verified=1` which shows a one-time toast
- On expired or already-consumed token: render a friendly error page with a "resend verification email" button
- On unknown token: render a generic "this link is invalid" page (don't leak whether the token ever existed)

### Resend flow

A `POST /resend-verification-email` endpoint, rate-limited to 1 request per user per 5 minutes. Issues a new token (the old one becomes orphaned but will expire naturally).

### Acceptance criteria

- `EXECUTOR: AI` — pytest at `src/tests/auth/test_verification_flow.py` covers the success path, expired-token path, replay path, and resend path; pass = all 4 scenarios green
- `EXECUTOR: AI` — Playwright test at `src/tests/e2e/test_verification_flow.py` covers the full registration → email-receipt-simulation → click-link → dashboard flow against the headless Chrome runner; pass = green
- Manual E2E pass against the real SMTP gateway to confirm the email actually delivers in production-like conditions

### UX details

- Toast on dashboard after successful verification: "Your email is verified. Welcome!"
- Registration confirmation page (before verification): "We sent a verification link to {email}. The link expires in 24 hours."
- Header banner shown on every page while `email_verified_at IS NULL`: "Please verify your email to unlock all features." with inline resend button
