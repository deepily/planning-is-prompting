## Key learning (Steward, owned)

**A well-reasoned correctness criterion can be scoped too narrowly — enumerate ALL call sites before ruling on exception handling.**

- The Steward's criterion was **`except OSError`, not bare `except Exception`** (graceful on expected FS faults, loud on unexpected logic bugs — the no-silent-fallback mandate). Sound — **for the hook call site**, which is PostToolUse (observational, non-blocking; an uncaught exception can't break the tool).
- But `touch_bridge_mtime()` has **two** call sites. The **MCP middleware** stamps **before `call_next`** — so a propagating exception there **short-circuits the real cosa-voice tool → tool-fatal, fleet-wide.** That makes **total-no-throw structurally required at the primitive**. Narrowing to `except OSError` wouldn't "fail loud" — it would force a bare `except Exception: pass` at the middleware boundary with *zero* observability. Narrowing makes masking **worse**, not better.
- **The trap (Krishna caught it):** Clayton's *offered* fallback — "bare primitive → `except OSError`" — is **incorrect in isolation**: it leaves the middleware tool-fatal on a non-`OSError` logic bug. The only mandate-pure narrowing would be BOTH `primitive → OSError` AND a middleware guard carrying its own observability rider ("Design Z'") — a **real change, not the 3-liner it looked like**. So the **broad `except Exception` was UPHELD** (observable swallow-point in the primitive); there was **no narrow-the-catch re-loop** — my interim "~3-line re-loop" projection was itself superseded by the ruling. The only adopted change was nit **N2** (a 1-line coupling-contract comment).
- **Why the mandate isn't violated by the catch-all:** Clayton's total-catch is **observable** (counter + one-shot stderr), not silent — so it isn't a "silent fallback hiding errors." And the Steward's own **C4 honest-age design** neutralizes the false-idle worry: a swallowed stamp ages the session to an honest "bridge 45m," **never a false-LIVE boolean**. The degraded answer is *accurate*, so there's no WRONG answer for the mandate to forbid.
- **The catch:** the Steward reasoned correctly from one call site; the **independent refute-first reviewer found the second.** That asymmetry is precisely what an adversarial seat exists to catch — and it caught it over two more-senior leans.

**Process corollary (for the workflow):** when a fork is about exception/error handling, the criterion-setter should explicitly enumerate every call site of the function in question before ruling. Add to the reviewer's anchor-prep: "is this criterion scoped to ALL call sites, or just the one in front of me?"
