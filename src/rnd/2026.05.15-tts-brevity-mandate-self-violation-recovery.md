## 1. Problem

The TTS Brevity Mandate, ratified in Session 81 (2026-05-04) and tightened in Session 82 (2026-05-05), explicitly requires:

1. **Headline-not-inventory** — numbers, file lists, percentages, paths, terms-in-conflict live in `abstract`, never in the spoken `message`.
2. **Speak the verdict, not the rationale** — skip confidence statements, process meta, and deferred-work explanations.
3. **Tiered length cap** — routine close-outs ≈ 60 words / 20 s; substantive turns ≈ 80–120 words / 30 s.
4. **Two-channel asymmetry** — `abstract` carries rich detail (UI card), `message` carries the spoken takeaway. The channels are **complementary, not duplicates**.

Rio's offending `notify()` violated rules 1, 2, and the spirit of 4:
- Spoke "phone mode" / "QUIET mode" / "per-session DND" terminology aloud — that's inventory.
- Spoke the May-14 reframe date and the recommended-fix instruction — that's rationale and process meta.
- Recited both abstract content AND offered the abstract — duplicated channels.

Equally important: the `abstract` carried the full table but **contained zero working doc-viewer links**, despite the DOCUMENT VIEWER LINKS mandate in `~/.claude/CLAUDE.md` (which says: "When the user asks to view a project file… respond with a `notify()` whose abstract contains a markdown link to the document viewer").

## 2. Root cause analysis

The violation pattern is **drafting the spoken message and abstract in parallel**, then trimming the abstract for richness instead of trimming the message for brevity. The author (me) treated the spoken line as an executive summary of the abstract, not as the verdict-only headline. Three contributing factors:

| # | Factor | Mitigation |
|---|---|---|
| 1 | The per-turn rider warns about brevity but is one paragraph long, easy to skim past after several turns. | Re-anchor: re-read the rider's brevity rules **before** composing each closing `notify()` until muscle memory holds. |
| 2 | Substantive-turn ceiling of ≈120 words leaves room for inventory creep when the substance is genuinely interesting. | Cap honest length: write the message, count words; if any noun or term appears that also appears in the abstract table, strike it from the message. |
| 3 | DOCUMENT VIEWER LINKS mandate covers user-asks-to-view-a-file but isn't applied reflexively when the abstract **references** files (audit findings, R&D reports). | Extend the reflex: any time `abstract` names a file or line range, include a working `/app/docs?path=…&scope=…` link to it. |

5. **Doc-Link Literacy at Startup (MANDATORY)** — strengthen the same Phase A / Phase B startup surfaces (`~/.claude/CLAUDE.md`, `global/CLAUDE.md`, `workflow/claude-config-global.md`, `workflow/session-start.md`) with an explicit orientation note that every Claude Code session MUST internalize at startup. Companion to hook 4 (persona-first) — both are "before-you-respond" PRECONDITIONS, not "nice-to-have" knowledge:
   - **Triggering observation (Mr. Rick, 2026-05-15 voice rebuke #2)**: *"We also want to add to their startup that they understand the notion of dock-links and the importance of only pushing dock-links into the abstract section."*
   - **What a doc-link is** (orientation text to be added at the head of the DOCUMENT VIEWER LINKS section): A doc-link is a markdown anchor of the shape `[Open: <filename>](/app/docs?path=<repo-relative-path>&scope=<scope-name>)`. It opens the named file in Mr. Rick's document viewer in a new tab. The `scope` query parameter is resolved per the four-step priority order documented in DOCUMENT VIEWER LINKS § "Choosing the right scope." Doc-links are how sessions hand Mr. Rick a clickable reference to a file without ever speaking the path aloud or dumping the file contents into chat.
   - **Where the scope and path variables come from** (Mr. Rick voice message #3, 2026-05-15): the `get_session_info()` MCP call that hook 4 already mandates ALSO returns a `doc_scope` envelope alongside the `voice_persona` envelope — shape `{scope, base_url, allowed_prefixes, source}`. That envelope IS the canonical startup-context lookup for building doc-links from this session. No second round trip permitted; no manual INI parsing tolerated; no guesswork about which scope name applies to "this repo." Session-start orientation MUST state explicitly: *"After `get_session_info()` returns, you MUST know (a) your persona, AND (b) your doc-link scope envelope — both arrive in the same call. You MUST use `doc_scope.scope` as the `scope=` query param. You MUST validate every `path=` against `doc_scope.allowed_prefixes` before emitting the anchor. Anchors emitted without prefix validation are violations."* Hook 4 (persona resolution) and hook 5 (doc-link construction) are COUPLED into a single startup ritual: one `get_session_info()` call resolves identity AND link-building primitives — and BOTH outcomes are MANDATORY before the first user-facing text.
   - **The hard rule**: Doc-links live **only in the `abstract` parameter of `notify()`** (and the message bodies of `commons_post()`, where chat-channel rendering accepts markdown). They MUST NOT appear in:
     - the spoken `message` parameter of `notify()` (URLs are profoundly TTS-hostile — they verbalize as gibberish character-by-character),
     - the spoken `message` parameter of `converse()`, `ask_yes_no()`, `ask_multiple_choice()`, `ask_open_ended_batch()` (same TTS hostility),
     - terminal-channel responses where the user is in speakerphone mode (the chat is for the screen, but in speakerphone the screen is unattended — the spoken contract is what reaches the user).
   - **Why "abstract only"**: The two-channel asymmetry is structural — `message` is converted to speech, `abstract` is rendered to the UI card as markdown. Speech can't render anchors, so a URL in `message` becomes audible noise. The card CAN render anchors, so a URL in `abstract` becomes a tappable link. Putting links anywhere else either wastes them (terminal scrollback Mr. Rick isn't reading) or actively degrades the speech channel.
   - **Reflex pairing with hook 3 (BIDIRECTIONAL MANDATE)**: Hook 3 states "abstract that *references* a file MUST *link* to that file." Hook 5 closes the symmetric gap: "doc-links MUST appear in the abstract — and NOWHERE ELSE." Together these form a bidirectional, non-negotiable rule — links flow INTO the abstract from any file reference; links flow OUT of the abstract NOWHERE. No exceptions. No "but the spoken channel is short so just this once" rationalization. No "the terminal scrollback is fine" rationalization.
   - **Verification at session-start (canonical workflow)**: `workflow/session-start.md` Step 1 to also gain — *"BEFORE any user-facing text, you have read the DOCUMENT VIEWER LINKS section of `~/.claude/CLAUDE.md` (or its inline summary in the per-turn rider) and can articulate (a) what a `/app/docs?path=…&scope=…` link is, and (b) the rule that such links belong only in the abstract parameter, never in spoken channels."*
   - **Anti-patterns to memorialize** (new sub-section in `workflow/cosa-voice-integration.md` alongside hook 1's self-audit checklist):
     - ❌ Speaking a URL in any form (full URL, shortened URL, path fragment with extension)
     - ❌ Reading "open the link in the abstract" or "see the linked document" in the spoken channel — that's still inventory leakage about the link's existence; the abstract carries its own affordance
     - ❌ Including a markdown anchor like `[Open: foo.md](...)` inside the spoken `message` — TTS will read "open colon foo dot m d" plus the parenthesized URL phonetically
     - ✅ Spoken: *"Plan serialized — link in the card."* (verdict; abstract carries the actual `[Open: ...]` anchor)
     - ✅ Spoken: nothing at all about the link; let the abstract's anchor speak for itself
