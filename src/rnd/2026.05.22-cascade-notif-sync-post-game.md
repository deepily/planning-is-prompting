## §2 — Actionable Conclusions (Roadblock-Removal)

The conclusions below target the roadblocks Rick named explicitly ("remove roadblocks and unnecessary gating and waste of time like we did today"). Each names: (a) what happened, (b) why it matters, (c) actionable conclusion + workflow-doc redline target, (d) priority for tomorrow.

### 2.1 — Step-3 user-gate friction *(highest priority)*

**What happened**: The cascade parked at Step 3 for ~4 hours waiting on the user's explicit decomposition approval. The decomposition (Tiffany's §0.1 — four sections A/B/C/D) had already been validated by Sam's Step-0 light-review (all 6 criteria PASS; zero blocking gaps). The user gate was a rubber-stamp on an already-validated decision.

**Why it matters**: This is the run's clearest waste-of-time exhibit. Mandatory rubber-stamp user gates trade real cascade latency for near-zero decision value. Mr. Radio's premature-go overstep (which he correctly stood down) is part of the same exhibit — the gate's brittleness invited the workaround.

**Actionable conclusion**: Step 3 becomes **cast-ratified-conditional**. The Manager may ratify an *uncontested* decomposition concurred by the cast (Author + reviewers + Observer) once the Step-0 light-review has passed. User-blocking fires only when the decomposition is *genuinely contested* — cast disagreement, novel decomposition shape, or a deliberate user-config opt-in for high-stakes plans.

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Step 3; add a new config knob `step_3_gate: cast_ratified | user_blocking` defaulting to `cast_ratified`.

**Companion doctrine** (the brittleness-invited-workaround anti-pattern, Mr. Radio's addition): codify explicitly that **peer-relayed user intent is NEVER authorization**. Mr. Radio's premature-go was triggered partly by treating a peer-relay paraphrase as if it were a user go; the classifier-deny was the safety net. A too-brittle gate invites sessions to look for workarounds, and sessions then risk fabricating authorization. Cast-ratified-conditional addresses the brittleness; the peer-relay doctrine addresses the workaround instinct.

**Priority**: **P0** — the headliner.

### 2.2 — Inter-stage user-gating

**What happened**: Rick's mid-afternoon expansion of (2.1) — *"I should not be a gate at any point between the start and finish of the three review phases."* The cascade design already routes inter-stage progression through the Manager (only the 7 escalation triggers pull the user in), but the principle deserves explicit codification so the doctrine is robust against future drift.

**Why it matters**: A mandatory user gate at every stage boundary would multiply the waste from (2.1) by ~12× (3 stages × 4 sections). Codifying the principle prevents drift.

**Actionable conclusion**: Affirm in workflow doctrine that **no mandatory user gate fires between Stages 1, 2, and 3** for any section. The Manager arbitrates inter-stage progression. The user is escalated to only on the 7 documented escalation triggers (foundational findings, cross-section conflicts, hard contradictions with prior user decisions, etc.).

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Stage progression; add an explicit "no inter-stage user gate" statement alongside the existing escalation-trigger list.

**Priority**: **P0** — same family as 2.1; codify together.

### 2.3 — PG-1 cold-casting fork *(carry forward from Run 5)*

**What happened**: Rachel 🕊️ correctly recused herself from the Stage-1 seat citing `reviewer_context_scope = narrow` + PG-1 cold-cast doctrine — she was the rehydrated Run-5 Manager carrying heavy prior-cascade context. The Manager (Mr. Radio) re-sourced Tiberius 🌑 for Stage 1 on a "cold-enough via orthogonal context" judgment — Tiberius had been doing unrelated implementation work, so his context was orthogonal to a notification-bloc plan review. The call worked cleanly (Stage 1 produced quality findings; Sam endorsed at Stage 3), but it represents a doctrine fork that is currently implicit.

**Why it matters**: There are now two operative interpretations of "cold":
- **Strict cold-cast**: full `/clear` → genuinely fresh persona, no surviving conversation context.
- **Orthogonal-context-acceptable**: a reviewer carrying context unrelated to the review subject is "cold-enough" without a `/clear`.

Both have defensible cases. Strict cold-cast is the safer default; orthogonal-context-acceptable is the pragmatic Manager-judgment call when no fresh reviewer is available. The fork is currently implicit — risk of drift.

**Actionable conclusion**: Pin the fork explicitly in workflow doctrine. Strict `/clear` is the default; the Manager may invoke an "orthogonal-context-acceptable" judgment subject to **explicit context-distance criteria** (Mr. Radio's refinement to keep this from drifting into a Manager-arbitrary loophole):

- **(a) Orthogonal repository** — reviewer's prior context is in a different repo than the review subject.
- **(b) Orthogonal feature surface** — reviewer's prior context touches a different feature surface (different bloc / different UI / different service).
- **(c) No overlapping recent commits** — reviewer's prior context did not touch the files under review.
- **(d) Narrow-scope reviews ONLY** — orthogonal-context-acceptable is allowed only for `reviewer_context_scope = narrow` reviews; broad-scope reviews still require `/clear`.

Mr. Radio's Tiberius call met all four. Explicit criteria let future Managers reproduce the judgment.

Subject to the existing Reassignment Bias-Risk Guardrail (rubric-differential discipline / cooling-off / cross-check).

**Sub-direction worth exploring** (Mr. Radio's suggestion): a **cold-reviewer pool** — pre-`/clear`'d sessions standing by for cascade work — and/or a **self-clear mechanism** so the user is not the bottleneck on a `/clear`.

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Stage 1 cast preparation + `workflow/plan-review-cascaded-common.md` §Reviewer Reassignment Latitude.

**Priority**: **P2** — affects every future cascade's cast preparation; lower-priority than the gating-reduction items (2.1 + 2.2) per Mr. Radio's order-of-attack suggestion.

### 2.4 — Participant-role doctrine: accept-and-flag-precondition

**What happened**: Rick flagged that Rachel's recusal *framing* leaned "unavailable / source someone else" rather than "I accept; my precondition is a `/clear`." Rachel's *substance* was doctrinally correct (PG-1), but the framing read as a refusal, which is what triggered Rick's escalation ("she cannot decline a job; they may suggest that I clear their memory, but they cannot decline").

**Why it matters**: A cast participant assigned a role should never *decline* — they accept and may flag a precondition (e.g. needing a `/clear` to take the seat cold). The Manager then handles the precondition. Small framing-doctrine point but it directly affected Rick's perception of cascade health today.

**Actionable conclusion**: Codify in workflow doctrine: **participants accept assigned roles**; they may flag preconditions for doing the role well; they do not *decline*. The Manager handles preconditions (re-cast, schedule a `/clear`, etc.).

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Cast roles + a brief reviewer-onboarding note in `workflow/plan-review-cascaded-personas.md`.

**Priority**: **P1** — low implementation cost, directly addresses Rick's explicit framing concern.

### 2.5 — Author-section-posting overhead

**What happened**: Each section required a full Stage-0 `author_draft` re-post to its section topic (`cascade-notif-sync-section-a` etc.), even though the input plan was already serialized and well-structured. The reposts were faithful but added work for the Author — Tiffany did 4 full draft posts + 8 revisions across the cascade.

**Why it matters**: Rick flagged this earlier in the day as worth examining. **Data point** (Mr. Radio's measurement): Tiffany's Stage-0 reposts took ~7 minutes × 4 sections ≈ **~28 minutes of re-formatting per cascade**. For plans with clean phase boundaries, lighter section-pointers could save **~25 minutes per cascade**. When the input plan is already well-structured with explicit per-section scoping (Tiffany's §0.1 mapped each section to a plan §Phase), is the full `author_draft` re-post needed, or is an in-place plan-slice pointer (commit + path + §Phase X + scope sentence) sufficient?

**Actionable conclusion**: Pilot a **plan-slice-pointer** mode in a future cascade. Author posts a section header with a pointer (`plan path + §Phase X + scope sentence + commit/SHA`) instead of a full draft re-post. Reviewers read the plan slice. Revisions are diffs against the plan. If the pilot works, make it the default mode for cascades whose input plan is already cascade-shaped.

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Stage 0 — add an opt-in `author_input_mode: full_repost | plan_slice_pointer` config knob.

**Priority**: **P1** — ~25-minute-per-cascade savings on the pilot success path; not blocking.

### 2.6 — PG-5 parallel-vs-serial *(carry forward from Run 5)*

**What happened**: Run 5 (May 21) ran fully serial — pass-by-pass, one reviewer at a time. PG-5 (root-cause doc `src/rnd/2026.05.22-run-5-serial-vs-parallel-root-cause.md`) identified three compounding causes and flagged true parallelism as an unbuilt re-architecture. **This cascade DID achieve genuine section-pipelining**: Tiberius pipelined A→B→C→D Stage 1 while Section A advanced to Stage 2 (Krishna) and then Stage 3 (Sam). **Actionable conclusion**: Acknowledge the section-pipelining win in workflow doctrine. Keep the PG-5 intra-section-parallelism re-architecture as a separate carry-forward item; revisit when the cascade workflow is otherwise stable.

**Workflow redline target**: Cross-reference the PG-5 root-cause doc; add a brief note in `workflow/plan-review-cascaded.md` §Pipeline about the section-pipelining achievement.

**Priority**: **P3** — observation, not an immediate-action item. The current cascade behaviour is workable for typical 4-section plans.

### 3.1 — Cluster-family doctrine *(standout positive)*

Krishna 🦚 surfaced four findings (F-Krishna-A1 / B3 / D1 / D2) sharing a single root pattern (wire contracts asserted but only fabricated-tested). Mr. Radio classified them as ONE cross-section cluster-family with a **single shared fix doctrine**: ground wire contracts against the authoritative server source (`valid_types` whitelist + emit sites) or a *mandated* live capture; never a hand-authored fixture standing in for the real contract. Tiffany applied the doctrine **uniformly 4-for-4** (AC-A5 / AC-B7 / AC-D4 / AC-D6). Section C was correctly excluded (no raw-wire exposure). Sam endorsed at Stage 3.

**Why it matters**: This is exemplary cross-section finding handling. A naïve cascade would route each of the four findings independently (4× the work for the Author, 4× the chance of inconsistent application). The cluster-family approach is faster AND produces a more consistent result.

**Action**: Name the doctrine explicitly: **"cross-section finding cluster recognition"** (Mr. Radio's naming proposal). Recognition criteria: (a) **≥3 finding-instances** sharing the same root pattern (this cascade had 4 — A1/B3/D1/D2); (b) shared fix doctrine; (c) NOT a cross-section *conflict* — each section's application is independent. **Mechanism**: **the reviewer recommends cluster-family treatment; the Manager ratifies.** Krishna's Stage-2 recommendation was the trigger for Mr. Radio's classification here — this is the prescribed sequence.

**Workflow redline target**: `workflow/plan-review-cascaded-common.md` §Manager classification — add a "cluster-family" tier alongside the existing severity tiers.

### 3.2 — Cascade-learning-loop forward-direction

Tiffany applied **proactive forward-sweeps** — she folded F-Tib-A1 (canonical test-file home) into Section B's draft at Stage 1, and F-Krishna-A2 (observation-mechanism via `@visibleForTesting` getter) into Section B at Stage 2, before either finding was independently surfaced for B. Both downstream stages (Krishna at Stage 2; Sam at Stage 3) endorsed the proactive sweeps explicitly.

**Why it matters**: This is the cascade-learning-loop working as designed. Each section that the Author drafts after the first benefits from earlier sections' findings, even before reviewers reach them. The result: subsequent sections need fewer revision turns and have fewer finding instances.

**Action**: Name "Persona-2.A point 14 (doctrine-sweep on revision-mechanism change)" as a positive Author doctrine to apply by default. Add to the Author's pre-cascade Recon checklist. **Companion downstream doctrine** (Mr. Radio's addition): when an Author has had a chance to forward-sweep a confirmed finding, **downstream reviewers should explicitly confirm (or flag) the sweep** in their own reviews — confirmations are part of the cascade-learning-loop, not just corrections. Both Krishna (Stage 2) and Sam (Stage 3) endorsed Tiffany's sweeps explicitly here; codify the bidirectional pattern.

**Workflow redline target**: `workflow/plan-review-cascaded.md` §Author-side discipline.

- **Krishna's F-Krishna-A1 flag**: explicitly raised "severity-placement to the Manager" with reasoning, AND explicitly recommended "no user escalation" ("escalating a citation-grounding gap would spend the human attention the cascade exists to protect").
- **Mr. Radio's down-classification of F-Tib-B1** (same pattern applied to F-Tib-D1): Tiberius proposed Layer 3 / medium on both; Mr. Radio reclassified Layer none / inconsistency on each with explicit reasoning that the finding challenged no frozen Q-decision — systematic discipline, not a one-off.

Both moments produced findings that were genuinely substantive but author-fixable — exactly the kind of finding that should NOT escalate to the user. The discipline matches Rick's repeated guidance that the cascade should "protect the human attention it exists to protect."

**Action**: Continue. Existing doctrine wording is good; no redline needed. Worth a callout in the workflow-overview section as exemplary practice.
